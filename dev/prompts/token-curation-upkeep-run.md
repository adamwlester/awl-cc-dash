## Context

It pulls ahead, by explicit operator decision on 2026-07-05, the non-minting portion of the token-hygiene audit, plus two operator asks: the badge/chip size standard and a component upkeep audit. What stays deferred to sprint 2, deliberately: minting NEW color tokens, any semantic alias layer, and any palette rework.

The operator's intent, recorded at scoping: every color belongs to the token system so the operator can retune values in `tokens.css` without ever touching code. When you find a hardcoded color, map it to an existing token that is close in value AND semantically fits the token's documented role. Be smart about deliberate divergence: if a surface was plainly recolored away from a token precisely so it would NOT signal that token's meaning, keep the hardcode and flag it. When in doubt, flag rather than convert.

The operator has explicitly authorized more initiative than prior runs: make the judgment calls yourself, document each with evidence (before/after screenshots for anything visible), and reserve flags for genuine ambiguity or apparent deliberate divergence. Good work that is wrong in a documented, reversible way beats a report full of punted decisions. The stage-0 checkpoint below is where the visible and destructive calls get ratified in one rapid-fire batch; use it instead of scattering questions through the run.

To be unambiguous about color: this run never redesigns the palette and never picks new colors. It rewires how existing colors are expressed (hardcode to token), and it may CORRECT a hardcode that plainly reads as a drifted or sloppy copy of an existing token by snapping it to that token (operator decision 2026-07-05: a small rendered shift back toward the intended token value is a fix, not a design change; every snap is proposed at the stage-0 checkpoint and carries before/after evidence). Colors with no fitting token stay hardcoded and get flagged; that flag list is input to the operator's separate, light palette revisit, nothing more.
