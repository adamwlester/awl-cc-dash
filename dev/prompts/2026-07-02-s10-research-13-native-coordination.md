## System context
awl-cc-dash is a single-window Electron desktop app (Windows, one operator) that talks to a FastAPI "sidecar" service on `127.0.0.1:7690` over HTTP + Server-Sent Events. The sidecar is the brain: it owns all cross-agent coordination state. The sidecar drives Claude Code coding-agent sessions as DETACHED tmux sessions running inside WSL2 (Windows Subsystem for Linux 2, Ubuntu) — one real `claude` CLI process per agent — and reads each agent through two channels: `tmux capture-pane` (a ~1s poll of the terminal screen, classified into idle / generating / permission-prompt) and the agent's JSONL (JSON-Lines) transcript file on disk. The primary control path is this tmux / terminal-UI (TUI) bridge — **NOT** the Claude Agent SDK. (A second, limited-use in-process Agent-SDK driver exists only for non-interactive utility passes like a "Revise/Summarize" text rewrite; it never drives the interactive coding agents and is not the main path.) A third inbound channel exists: each WSL agent's Claude Code hooks POST back to the sidecar over HTTP (the sidecar resolves the WSL2 default-gateway IP, because `localhost` from inside WSL2 does not reach the Windows host).

Question-specific detail you need:

- **The dashboard's CUSTOM coordination spine** (all owned by the sidecar):
  - **INBOX** — a "needs-you" surface of typed cards, one per blocked agent (types: permission, error, warning, plan, decision — an open-ended set). Cards are raised from two sources: bridge screen-state (a permission menu on screen, or an error/stall), and the HTTP hook channel (a `plan` card from the agent's ExitPlanMode tool call; a `decision` card from its AskUserQuestion tool call).
  - **LINKS** — an agent-to-agent link joining two agents, carrying one relationship: DIRECT (a reply-to conversation — when the source agent finishes the turn that answers a linked peer, the sidecar routes that turn's output back to the peer) or SHARED (passive context awareness). Links carry an internal delivery-timing label (Now / Inject / Next / Queue / Hold — treat these as opaque routing options; they are not load-bearing for this question) and End-After caps (default 25 exchanges) to prevent runaway.
  - **SCRATCHPAD** — an always-current shared board every agent auto-reads, delivered as a bounded per-agent DELTA off a read-watermark (each agent sees only posts past its own bookmark). Stored as a markdown file in the project folder.
  - These ride a few sidecar primitives: an event envelope + addressing (every event carries a source and a `recipients[]` array), a per-agent prompt queue with delivery dispositions, the HTTP hook channel, and read-watermarks.
- **Maturity of the spine:** inbox, links, and scratchpad are built; the DIRECT-link reply-routing is live-verified end-to-end (a source agent's finishing turn routes back to its linked peer, capped by End-After). One caveat to carry: the plan/decision **hook** path (ExitPlanMode / AskUserQuestion → inbox cards) is wired but **spike-gated and unproven** under the bridge — unlike the inject-delivery hook path, which is proven live. So "the hook channel raises plan/decision cards" is intended-and-wired, not yet confirmed.
- **How the dashboard sees and drives agents:** it SEES via capture-pane (screen state, ~1s poll), the JSONL transcript on disk, and inbound HTTP hook callbacks. It DRIVES via tmux keystrokes (`send-keys`) and via hook responses. It does NOT use the Agent SDK for the main path.
- **Why "adopt" is subtle:** native Claude Code coordination primitives are, in general, TOOLS a Claude Code session calls from the INSIDE. This dashboard drives from the OUTSIDE via tmux. So each primitive must be judged on two axes: **OBSERVABILITY** — can the sidecar SEE the primitive's state/effects via capture-pane, the JSONL transcript, or a hook event? — and **DRIVABILITY** — can the sidecar INITIATE or ROUTE the primitive from outside, via `send-keys`, a hook response, or launch config?
- **Already handled today — SUBAGENTS:** a subagent is a sub-identity of its parent. Today the sidecar DERIVES the subagent roster (identity / status / usage) from the PARENT agent's own transcript, by pairing each subagent-spawn tool call (`Task` / `Agent`) with its matching tool-result. (Ingesting each subagent's OWN separate transcript file — the `<parent>/subagents/` directory Claude Code writes — is a deferred follow-on, not built.) A known limit: over tmux the bridge CANNOT distinguish a *pending* subagent from an *active* one.
- **Prior-art signal** (from earlier internal research — treat as an unverified second-hand pointer, not fact): a comparable open-source Claude-Code monitoring dashboard reportedly observes native run-state and subagent lifecycle purely via Claude Code HOOKS (PreToolUse / PostToolUse / SubagentStart / SubagentStop, etc.), never by reaching into the SDK. Verify independently before relying on it.
- **On the progress bar (to avoid a false lead):** the dashboard already shows a per-run progress bar, but it does NOT come from `TodoWrite`. It parses a system-prompt-mandated MARKDOWN checklist (`- [ ] ...` / `- [x] ...` lines) out of the agent's own assistant TEXT. Any `TodoWrite` tool output that lands in the transcript is currently bucketed as "other" and NOT consumed. So adopting `TodoWrite` as a real coordination/progress source would be a net-new integration, not a description of current behavior.

## The question
Which native Claude Code coordination concepts — **`Task`** (spawn a subagent), **`TodoWrite`** (a structured task/checklist tool), **`Workflow`** (a deterministic multi-agent orchestration script), **`SendMessage`** (message/continue a running agent by id), and **agent-teams / teammate-spawning** — should this dashboard ADOPT, versus keep as its OWN custom coordination spine (inbox of typed cards, agent-to-agent links, shared scratchpad)? The single decision axis that settles it: which of these primitives are actually REACHABLE and OBSERVABLE when driving Claude Code as a TUI over tmux (this dashboard's primary path), as opposed to only via the Agent SDK (which the dashboard does NOT use for the main path)?

## What is already known / assumed (do not re-derive)
- The custom spine works: inbox, links, and scratchpad are built and tested; DIRECT-link reply routing is live-verified end-to-end.
- Subagents are OBSERVABLE today (roster/status/usage derived from the parent's transcript); the only gap is that *pending-vs-active* for a subagent cannot be distinguished over tmux (an engine limit).
- The hook channel already delivers PostToolUse / Stop (inject delivery, proven live) and PreToolUse(ExitPlanMode | AskUserQuestion) (plan/decision cards — wired but spike-gated/unproven); a broader hook set (SubagentStart / SubagentStop / Notification / UserPromptSubmit) is available to register but is not yet wired.
- Two independently-spawned tmux agents are separate `claude` OS processes with NO shared in-process bus — any cross-agent routing goes THROUGH the sidecar today. There is no native inter-process message bus between two hand-spawned CLI sessions.
