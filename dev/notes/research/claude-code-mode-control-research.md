# Controlling Claude Code TUI sessions programmatically: permission mode, orchestration prior art, run-state, and context

Deep research for the **awl-cc-dash** dashboard (tmux + WSL2 driven Claude Code TUI sessions, no Agent SDK).

Scope notes. "TUI" means terminal user interface; "CLI" means command-line interface; "JSONL" means JSON Lines (the transcript format); "MCP" means Model Context Protocol; "IPC" means inter-process communication. Your running Claude Code build is **2.1.187** (from `DEVLOG.md` and the bridge driver comment). Findings are current to late June 2026. Sources are linked inline; every approach carries a confidence tag of **confirmed-working**, **plausible-untested**, or **dead-end**.

Repo grounding (read via the GitHub connector, HEAD `d98725f`). Your architecture is Windows -> `wsl -d Ubuntu -- bash -c '...'` -> `tmux` -> Claude Code TUI. `bridge/bridge.py` (`TmuxBridge`) is the low-level driver: `create()` spawns `tmux new-session -d -s NAME -c CWD 'claude --model X'` then `_clear_startup_gates()` clears the folder-trust and bypass-warning gates; `send()` uses `send-keys -l` plus a separate `Enter`; `read()`/`status()` use `capture-pane -p -J -S`; `_detect_state()` classifies idle/generating/permission_prompt from the screen (spinner glyph with ellipsis -> generating; numbered "1. Yes" menu -> permission_prompt; horizontal rule plus the prompt marker and not generating -> idle); `parse_permission_prompt()` anchors on the bottom menu. `bridge/transcript.py` finds the transcript by hashing the pane cwd and parses `message.usage`. The sidecar `BridgeDriver` (`sidecar/drivers/bridge.py`) wraps this, advertises `CAPABILITIES = {interrupt, context, permission, resume, set_model, set_effort}`, derives overall context from the latest assistant `usage` over a hardcoded 1,000,000 token window, and (the crux) leaves **`set_mode()` a deliberate no-op** with the note that permission mode is "Shift+Tab-cycle-only (relative), no absolute set." The bridge driver is implemented but flagged "not yet live-verified," and `DriverConfig.permission_mode` defaults to `acceptEdits`. Your design intent (`design/DESIGN.md`) is a five-segment, always-editable-while-running permission control: **Plan / Ask / Edit / Auto / Bypass**.

## Executive summary

1. **No clean absolute mode-set exists, and that is now confirmed at every layer.** There is no slash command (issues [#15407](https://github.com/anthropics/claude-code/issues/15407), [#15041](https://github.com/anthropics/claude-code/issues/15041), both closed as duplicate feature requests); `settings.json` `defaultMode` is read only at startup and does **not** hot-apply to a running session (the open request [#55507](https://github.com/anthropics/claude-code/issues/55507) asks for exactly that auto-reload); there is no env var, signal, or IPC channel that mutates a running TUI's mode; and the keybindings system exposes only a single `chat:cycleMode` action, with **no per-mode "set" action** ([keybindings reference](https://code.claude.com/docs/en/keybindings)). The Agent SDK can set mode dynamically, but you are not using it.

2. **Keystroke-cycling with verification is the best available method, and it is now far more reliable than it used to be because the cycle order is documented and deterministic.** The official [permission-modes doc](https://code.claude.com/docs/en/permission-modes) states the order: `default -> acceptEdits -> plan`, with optional modes slotting in **after `plan`, bypassPermissions first and auto last**. So the full possible cycle is `default -> acceptEdits -> plan -> [bypassPermissions] -> [auto] -> (wrap)`. tmux's keycode for Shift+Tab is `BTab` ([tmux modifier-keys wiki](https://github.com/tmux/tmux/wiki/Modifier-Keys)). With a known starting mode (readable from the transcript `permissionMode`, a hook payload's `permission_mode`, or the status bar) you can compute the exact number of `BTab` presses and verify the result. **Direct verdict: reliable mode-setting in your tmux-TUI setup is achievable, but only as read-current -> compute-presses -> send `BTab` xN -> verify-and-retry, with launch flags pre-arming the optional modes. There is no shortcut around the cycle.**

3. **Two preconditions make the cycle deterministic, and you must set them at launch.** (a) `bypassPermissions` is unreachable mid-session unless you launch with `--allow-dangerously-skip-permissions` (adds it to the cycle without activating) or one of the other enabling flags; you literally "cannot enter bypassPermissions from a session that was started without one of the enabling flags." (b) `auto` only appears when your account qualifies (v2.1.83+, specific plan/model/provider) and shows a **one-time opt-in prompt** on first cycle-to; pre-accept it once or detect-and-answer it. `dontAsk` never appears in the cycle (startup-only), and it is not one of your five UI segments anyway.

4. **The single highest-value find is that mature orchestrators do not set mode either; they observe it via hooks, and they expose a cleaner architecture you should adopt.** [bjornjee/agent-dashboard](https://github.com/bjornjee/agent-dashboard) is the closest analog to your repo (tmux-native control plane, JSONL state detection, phone PWA). It registers Claude Code **hooks** that fire on `PreToolUse`/`PostToolUse`/`PermissionRequest`/`Stop`/`Notification`/`SubagentStart`/`SubagentStop` and write `state`, `permission_mode`, and `current_tool` to a per-agent JSON file. It reads `permission_mode` from the hook payload and never sets it (it even auto-adjusts `/effort` on plan-mode transitions by sending keys, exactly your effort technique). This push-based hook stream is dramatically more reliable than screen-scraping for run-state, and every tool/turn hook payload carries `permission_mode` for free, giving you authoritative live mode-reading.

5. **Two concrete, low-effort upgrades to your repo fall out of the prior art.** (a) Replace your cwd-hash transcript finder with the canonical `~/.claude/sessions/{PID}.json` -> `sessionId` mapping that both [recon](https://github.com/gavraz/recon) and agent-dashboard use (join the pane PID to the session file; robust when multiple sessions share a cwd). (b) Your hardcoded 1,000,000 token context window is only correct for 1M-context Opus models; recon maps the window per model (200,000 for most, 1,000,000 for the 1M-context Opus). And for run-state, layer a **deterministic transcript check** (any `tool_use` id without a matching `tool_result` means "still working") over your spinner heuristic, as agent-dashboard does, to kill state flicker.

## Question 1 (top priority): setting permission mode on a running TUI session

### The mechanism, settled

The [permission-modes doc](https://code.claude.com/docs/en/permission-modes) is now explicit and authoritative. The facts that govern any automation:

- **Six modes exist**: `default`, `acceptEdits`, `plan`, `auto`, `dontAsk`, `bypassPermissions`. Your five UI segments map cleanly: Plan = `plan`, Ask = `default`, Edit = `acceptEdits`, Auto = `auto`, Bypass = `bypassPermissions` (the VS Code label table in the same doc confirms this mapping). `dontAsk` is the sixth, startup-only, not in your UI.
- **The Shift+Tab cycle is deterministic.** Base order is `default -> acceptEdits -> plan`. Optional modes slot in after `plan`: bypassPermissions first, then auto. So if both are enabled the full cycle is `default -> acceptEdits -> plan -> bypassPermissions -> auto -> (wrap to default)`.
- **Reaching bypassPermissions requires a launch flag.** It "appears after you start with `--permission-mode bypassPermissions`, `--dangerously-skip-permissions`, or `--allow-dangerously-skip-permissions`"; the `--allow-` variant adds it to the cycle without activating. "You cannot enter `bypassPermissions` from a session that was started without one of the enabling flags; restart with one to enable it."
- **Reaching auto requires eligibility plus a one-time opt-in.** Auto needs v2.1.83+ and a qualifying plan/model/provider; first cycle-to shows an opt-in prompt (accept, or "No, don't ask again" removes it from the cycle).
- **Current mode is readable three ways**: the transcript records `permissionMode` (you already know this; the `system/init` line carries it, see issue [#13468](https://github.com/anthropics/claude-code/issues/13468)); a hook payload carries `permission_mode` on every tool/turn event; and the status bar shows mode text (for example `⏵⏵ accept edits on` for acceptEdits per the doc).

### Ranked approaches

**1. Read-compute-send-`BTab`-verify cycling, with launch flags pre-arming the cycle (confirmed mechanism; integration plausible-untested).** Launch every session with `--allow-dangerously-skip-permissions` so bypass is in the cycle without activating; if the account qualifies and you pre-accept the auto opt-in once, auto is in the cycle too. Then to set a target mode: read the current mode, compute the forward distance in the known cycle, send `tmux send-keys -t NAME BTab` that many times (one call per press, small sleeps between), and re-read the mode to confirm; retry on mismatch. Every primitive here is confirmed: the cycle order ([doc](https://code.claude.com/docs/en/permission-modes)), `BTab` as the Shift+Tab keycode ([tmux wiki](https://github.com/tmux/tmux/wiki/Modifier-Keys)), and mode-reading. The integrated loop is what you have not yet wired (your `set_mode()` is a no-op). This is the only method that gives absolute set, and it is exactly the keystroke-cycling-with-verification you hypothesized.

**2. Rebind `chat:cycleMode` to an unambiguous, easily-sendable key via `~/.claude/keybindings.json` (action confirmed + hot-reload confirmed; rebind tactic plausible-untested).** The [keybindings reference](https://code.claude.com/docs/en/keybindings.md) confirms `chat:cycleMode` (default Shift+Tab) is the only permission-mode action, that changes to `keybindings.json` are "automatically detected and applied without restarting," and that you can bind any key. Rebinding the cycle to a function key or a Ctrl-chord that nothing collides with removes any risk that `BTab`'s escape sequence (`CSI Z`) is mis-encoded on the path through tmux, and removes platform variance (the doc footnote notes that on Windows without VT mode the cycle defaults to `Meta+M`, not Shift+Tab; you avoid that by running in WSL/tmux, but rebinding makes it explicit). You still cycle, you just cycle with a key you fully control. There is also `confirm:cycleMode` (Shift+Tab in the Confirmation context) for cycling while a permission dialog is open.

**3. `/plan` prefix to enter plan mode by typed command (command confirmed; persistence untested).** The [permission-modes doc](https://code.claude.com/docs/en/permission-modes) and others confirm you can "enter plan mode by pressing Shift+Tab or prefixing a single prompt with `/plan`." This is a typed mechanism (like your working `/model` and `/effort`), so it is attractive for the Plan segment specifically. The open question is persistence: the doc phrases it as "a single prompt," so it may apply only to that turn and revert afterward. Worth a direct test; if it persists, it is a clean typed path into plan mode without cycling.

**4. PreToolUse / PermissionRequest hook reading a dashboard-written control file, to drive the autonomy axis without touching mode (deny path confirmed-working; allow path known-buggy in interactive mode).** A hook is a shell command that reads JSON on stdin and returns a decision; because it re-runs on every matching tool call, it can read an external per-session control file your dashboard writes, giving you instant runtime control of permission behavior without restart or settings reload. The decision schema ([hooks reference](https://code.claude.com/docs/en/hooks)): PreToolUse returns `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"allow"|"deny"|"ask","permissionDecisionReason":"..."}}` (plus a v2.1.89 `defer` for headless, issue [#41791](https://github.com/anthropics/claude-code/issues/41791)); PermissionRequest returns `{"hookSpecificOutput":{"hookEventName":"PermissionRequest","decision":{"behavior":"allow"|"deny"|"ask",...}}}`. The **deny path is rock-solid** and fires even in bypass mode (bypass skips interactive prompts, not hooks), so a dashboard-controlled blocklist is reliable. The **allow path is the catch**: issue [#52822](https://github.com/anthropics/claude-code/issues/52822) (v2.1.119, Linux) reports `permissionDecision: "allow"` does not suppress the native prompt in interactive mode (the hook runs and parses fine, but the prompt still shows), and issue [#55255](https://github.com/anthropics/claude-code/issues/55255) (v2.1.126) reports that allowlist, mode toggles, and `skipAutoPermissionPrompt` all failed to suppress Edit prompts specifically (only the launch flag worked). So a hook cannot be relied on to silence prompts in your interactive TUI; treat it as a deterministic guardrail (deny) plus a state reporter, not a substitute for acceptEdits/bypass. Also note hooks "can tighten but cannot loosen": allow only skips the prompt for calls that were already going to be allowed; it never overrides a deny/ask rule.

**5. Restart the session with the target mode flag (confirmed-working; lossy).** `--permission-mode <mode>` sets any mode at startup including `dontAsk` and `bypassPermissions`. This is the reliable way to reach `dontAsk` (unreachable by cycling) or to guarantee bypass. The cost is the obvious one the feature requests complain about: a restart loses in-memory conversation state. With `--resume`/`-r SESSION_ID` you can relaunch the same session id, but mid-turn state and any unsaved context are still disrupted, so reserve this for between-task mode changes, not mid-run.

**6. `settings.json` `defaultMode` hot-apply (dead-end).** Confirmed not to hot-apply. Issue [#55507](https://github.com/anthropics/claude-code/issues/55507) requests "auto-reload settings on file change for the active session" as a missing feature; the [permissions doc](https://code.claude.com/docs/en/permissions) describes `defaultMode` purely as a startup default. Editing it changes only the next session.

**7. Slash command to set mode (dead-end).** No such command. [#15407](https://github.com/anthropics/claude-code/issues/15407) and [#15041](https://github.com/anthropics/claude-code/issues/15041) request `/yolo`, `/dangerously-skip-permissions`, `/permissions bypass on`, etc.; both closed as duplicates with nothing shipped. (`/permissions` opens the allow/ask/deny rules UI; it does not set the mode.)

**8. Signal, env var, or IPC to a running TUI (dead-end).** No documented signal changes mode (SIGINT interrupts, SIGTSTP suspends). Env vars (`CLAUDE_CODE_ENABLE_AUTO_MODE`, `CLAUDE_CODE_EFFORT_LEVEL`, `--permission-mode`) are read at startup only. The `~/.claude/sessions/{PID}.json` files are read-only state, not a control channel. The only programmatic control surface for an interactive TUI is keystrokes; the only "API" is the Agent SDK, which you have ruled out.

### Gotchas to design around

- **Idle-gating is mandatory.** agent-dashboard explicitly does not dispatch keys when the agent is in a "stop state" because `tmux send-keys` would land inside a textarea the user is composing in (plan-review box, AskUserQuestion answer, idle prompt). Gate every `BTab` (and every typed command) on a verified safe state, not mid-prompt.
- **Cycle membership shifts.** If you do not pass the bypass enable flag, the cycle is only `default -> acceptEdits -> plan`, so your press-count math must key off which optional modes are actually enabled for that session (decide this at launch and record it per session).
- **The auto opt-in prompt interrupts the cycle.** First cycle-to-auto shows a menu; either pre-accept once or detect-and-answer it before continuing the count.
- **Suppression has regressed repeatedly; verify on 2.1.187.** Per #52822 and #55255, in some recent builds cycling to acceptEdits/auto/bypass did not actually suppress prompts (Edit especially). Confirm on your exact version that landing on a mode changes behavior, not just the indicator. Your existing prompt-answering path (Enter = yes, Escape = no on the parsed menu) is the universal fallback that works regardless of these resolver bugs.

## Question 2 (highest-value): prior art, Claude-Code-as-a-service and multi-session orchestrators

I cloned and read the four most relevant projects at source level. None of them programmatically sets permission mode; the mature ones observe it via hooks and focus their engineering on reliable state, input, and recovery. Ranked by relevance to your repo:

**1. bjornjee/agent-dashboard (confirmed-working; closest analog, adopt its hook architecture).** [github.com/bjornjee/agent-dashboard](https://github.com/bjornjee/agent-dashboard) (HEAD `3403273`). A Go TUI (Bubble Tea) plus a phone PWA, a "control plane on top of `tmux capture-pane`," reading Claude Code JSONL transcripts. How it handles your concerns:
- **Mode**: read-only. A Claude Code plugin registers hooks (`adapters/claude-code/hooks/hooks.json`); the async `agent-state-fast.js` hook on `PreToolUse`/`PostToolUse`/`PermissionRequest` (matcher `*`) writes `state`, `permission_mode`, `current_tool` to `~/.agent-dashboard/agents/{session_id}.json`. `permission_mode` comes straight from the hook payload. It never sets mode; "plan mode is signaled, not gated."
- **Run-state**: a granular model (`permission`/`plan`/`question`/`error`/`running`/`idle_prompt`/`done`/`pr`/`merged`). Tool-specific hook signals beat the generic event (`ExitPlanMode` -> `plan`; `AskUserQuestion` -> `question`; `PermissionRequest` -> `permission`). On `Stop`, a heuristic (`detect.js`: question-regex on the last assistant message plus the `❯` U+276F glyph in the last 5 pane lines) is overlaid with a deterministic JSONL check (`pending-tools.js`: any `tool_use` id lacking a matching `tool_result` means still-working) to kill flicker.
- **Input injection**: `tmux send-keys -t target text Enter` for prompts and replies; effort changes via `send-keys '/effort <level>' Enter`.
- **Permission prompts**: answered by sending keys (for plan, Escape selects Deny which opens a feedback box, then the reply text is sent as the denial reason).
- **Session recovery / discovery**: walks the PID tree reading `~/.claude/sessions/{PID}.json` for the `sessionId` (`agent-state-reporter.js::findSessionId`).
- **Context**: per-agent token breakdown persisted to SQLite, plus live rate-limit bars auto-discovered from Claude OAuth credentials.
- **Versus your repo**: same control-plane philosophy and JSONL parsing, but it is push-based via hooks rather than poll-based via screen scraping, its state model is richer, and it uses the canonical PID->session file instead of a cwd hash. The hook event stream is the single biggest idea to port.

**2. gavraz/recon (confirmed-working; mirrors your detection, plus the PID->session mapping).** [github.com/gavraz/recon](https://github.com/gavraz/recon) (Rust, HEAD `ca5a0e7`). A tmux-native dashboard.
- **Run-state**: `pane_status()` is almost identical to your `_detect_state()`: spinner char plus `…` (U+2026) -> Working; `❯ N.` selection or "Esc to cancel" on the last line -> Input; else Idle. Independent corroboration that your approach is sound.
- **Session discovery**: `build_live_session_map()` joins `~/.claude/sessions/{PID}.json` (`sessionId`, started_at) to tmux panes by PID, then matches `~/.claude/projects/*/*.jsonl` by filename stem. Cleaner than cwd hashing.
- **Context**: tokens summed from JSONL `usage`; the window is **model-dependent** via `model::context_window()` (200,000 default, 1,000,000 for the 1M-context Opus). Directly relevant to fixing your hardcoded 1M.
- **Gotcha it documents**: `/clear` writes a new JSONL without updating the session-to-process mapping, so data goes stale until restart. It does not set mode.

**3. primeline-ai/claude-tmux-orchestration (confirmed-working; the canonical input-injection patterns).** [github.com/primeline-ai/claude-tmux-orchestration](https://github.com/primeline-ai/claude-tmux-orchestration). A worker-pool starter kit. The patterns it codifies match and extend yours:
- `send-keys -l` (literal) plus a separate `Enter` (combining them races the input buffer) -- exactly what you do.
- **`paste-buffer` for multiline**: `send-keys` breaks on newlines, so `load-buffer` the text into a tmux buffer and `paste-buffer` it in cleanly. This is an upgrade for your `send()` when prompts contain newlines.
- A `.ready` file handshake so commands only go out when the pane is actually idle (prompt-collision avoidance, the same idle-gating principle).
- It cites [absmartly/Tmux-Orchestrator](https://github.com/absmartly/Tmux-Orchestrator) for the send-keys verify-retry pattern and [Dicklesworthstone/ntm](https://github.com/Dicklesworthstone/claude_code_agent_farm) for idle detection (12-line capture window, spinner override, ANSI strip, "Double-Enter Protocol"). Does not set mode.

**4. Dicklesworthstone/claude_code_agent_farm (confirmed-working; scale, auto-restart, context-threshold).** [github.com/Dicklesworthstone/claude_code_agent_farm](https://github.com/Dicklesworthstone/claude_code_agent_farm). Runs 20+ agents in tmux with a real-time monitor, heartbeat tracking, pane-title status, auto-restart with adaptive idle timeout, and a `--context-threshold N` to auto-clear context when it gets high. Useful reference for lifecycle auto-stop limits and context-driven restarts. Does not set mode.

**Others worth knowing (confirmed-working as projects; less aligned).** [adamwulf/ittybitty](https://github.com/adamwulf/ittybitty) (pure-bash, tmux-only, agents spawn agents, `ib nuke` kill-switch, max-agent cap, pre-approved tool list rather than bypass, git worktrees per agent); [twaldin/tmux-orchestrator](https://github.com/twaldin/tmux-orchestrator) (plugin; isolated tmux windows, git worktrees, two-way messaging); and Anthropic's own experimental [agent teams](https://code.claude.com/docs/en/agent-teams) (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`, split-pane mode uses tmux; note teammate permission requests bubble to the lead, and the doc advises pre-approving operations in permission settings to reduce friction). The original Tmux-Orchestrator family leans on prompting agents to self-coordinate and on broad permissions; treat its security posture with care.

**Bottom line for Q2**: the prior art has solved run-state, input injection, permission-prompt handling, session discovery, and context monitoring well, and you can lift those patterns directly. It has not solved absolute mode-setting, which corroborates the Q1 verdict: the cycle is the only lever, and the sophisticated tools simply observe mode rather than set it.

## Question 3: driving the TUI reliably (general)

Ranked techniques, each confirmed in real code (yours and the prior art):

**Run-state detection.**
1. **Hook-driven push state (confirmed-working; most reliable).** Register hooks that fire on `PreToolUse`, `PostToolUse`, `PermissionRequest`, `Stop`, `Notification`, `UserPromptSubmit`, `SubagentStart`, `SubagentStop`, `StopFailure` and have them write state (or POST it) per session. Because your sidecar already runs an HTTP server, the **HTTP hook type** is a natural fit: `{"type":"http","url":"http://localhost:PORT/hooks/...","timeout":N}` posts the event JSON to you ([hooks reference](https://code.claude.com/docs/en/hooks)). This replaces polling with a push stream and gives `permission_mode` on every tool/turn event for free. agent-dashboard's `hooks.json` is a working template.
2. **Spinner-plus-ellipsis screen heuristic (confirmed-working; what you and recon both do).** A sparkle/spinner glyph at line start with `…` means generating; the `❯`/numbered menu means a prompt; horizontal rule plus `❯` and not generating means idle. Keep this as a fallback and for sessions without hooks.
3. **Deterministic transcript overlay (confirmed-working; the robustness fix).** Tail the JSONL ~32 KB, collect `tool_use` ids from assistant entries minus `tool_result` ids from user entries; nonempty set means the agent is still working, which overrides screen flicker (agent-dashboard `pending-tools.js`). Add this to your `_detect_state()` to harden generating-vs-idle.

**Input injection.**
1. **`send-keys -l` then separate `Enter` (confirmed-working).** Literal mode plus a separate Enter avoids buffer races; you already do this, and the prior art agrees.
2. **`load-buffer` + `paste-buffer` for multiline (confirmed-working; upgrade).** `send-keys` mishandles newlines; buffer-paste injects multiline text cleanly (primeline-ai). Worth adding to `send()` for newline-containing prompts.
3. **Typed slash commands that support non-interactive use (confirmed-working).** `/model <name>` and `/effort <level>` set on Enter (you confirmed this; agent-dashboard relies on it too). `/fast` opens an interactive panel (no clean scrape), and there is no `/thinking` command; for those two, prefer the keybinding actions below.
4. **Keybinding actions for fast/thinking toggles (action confirmed; integration plausible-untested).** `chat:fastMode` (Meta+O) and `chat:thinkingToggle` (Meta+T) exist as keybinding actions even though no slash commands do ([keybindings reference](https://code.claude.com/docs/en/keybindings.md)). Rebind them to clean keys and send those to toggle. Both are toggles (no absolute on/off), so read current state first; thinking-on typically shows `thinking` blocks in the transcript.

**Permission-prompt handling.**
1. **Answer keys on the parsed menu (confirmed-working; your current approach, universal).** Enter selects option 1 (Yes); Escape denies. This works regardless of the suppression regressions in #52822/#55255, which is why it is the safe default. Option 2 (always-allow) is unverified; leave it off unless you test it.
2. **PreToolUse/PermissionRequest deny hook for guardrails (confirmed-working).** Exit 2 or `permissionDecision: "deny"` blocks a tool in any mode including bypass, and feeds the reason back to Claude. Pair a deny hook with a deny rule for full coverage on file tools. Do not rely on the allow path to silence prompts in interactive mode (see Q1 item 4).

**Session recovery after restart.**
1. **PID->session file mapping (confirmed-working; adopt).** Read `~/.claude/sessions/{PID}.json` for the `sessionId` (walk the pane's process tree if a shell wraps `claude`), then locate the transcript directly. More robust than cwd hashing when multiple sessions share a directory (recon, agent-dashboard).
2. **Persisted runtime record + resume-or-create (confirmed-working; you already do this).** Your `runtime_store.py` plus `resume()` rebinds to a live tmux session by name on sidecar restart; keep it, and consider also persisting the resolved `sessionId` so you do not re-discover it.
3. **`claude --resume SESSION_ID` to relaunch a dead session (confirmed-working; lossy).** Restores conversation history but not mid-turn state; fine for recovery, not for seamless mode changes.

**Known fragilities and mitigations (all confirmed):** keystrokes landing in a textarea when the agent is composing (gate on verified safe state); `/clear` orphaning the session-to-PID mapping (re-discover after a clear, or kill-and-recreate as recon advises); the `Windows-without-VT` cycle-key change to `Meta+M` (avoided in WSL/tmux; rebind to be explicit); and suppression regressions per version (verify behavior, keep the answer-keys fallback).

## Question 4: context monitoring

**Overall context usage is cleanly available; the per-category breakdown is not.**

**1. Statusline `context_window` object (confirmed-working; cleanest overall live signal).** The [statusline doc](https://code.claude.com/docs/en/statusline) shows the JSON piped to your statusline command each turn includes a `context_window` object: `total_input_tokens`, `total_output_tokens`, `context_window_size`, `used_percentage`, `remaining_percentage`, and `current_usage` (`input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`). Critically, `used_percentage` is computed input-only (`input_tokens + cache_creation_input_tokens + cache_read_input_tokens`, excluding output), which **matches both the `/context` percentage and your own `derive_context_usage()` formula**. As of v2.1.132 these reflect current context, not cumulative session totals (the cumulative bug was [#13783](https://github.com/anthropics/claude-code/issues/13783), fixed before your version). So a `statusLine` command that writes this JSON to a per-session file (or POSTs it to your sidecar) gives you accurate live overall context per agent with no `/context` call and no interruption. It updates per turn (when messages change), not continuously mid-run.

**2. Your existing transcript-derived overall percentage (confirmed-working; keep, with one fix).** Taking the latest assistant entry's `usage` (input + cache_read + cache_creation) over the window is the same number the statusline reports, so your math is validated. The one bug: the window is model-dependent. Map it per model (200,000 for most; 1,000,000 only for the 1M-context Opus variants) rather than hardcoding 1,000,000, or read `context_window_size` from the statusline payload directly. recon's `model::context_window()` is a working reference.

**3. Per-category breakdown (system prompt / system tools / MCP tools / memory / messages / free space) (dead-end for clean structured access).** This is what `/context` renders, and it is computed internally and **never exposed as structured data**. The statusline `current_usage` breaks tokens down by **type** (input/cache/output), not by **content category**, so it does not give the `/context` table. Multiple feature requests confirm the gap: [#12565](https://github.com/anthropics/claude-code/issues/12565) and [#11535](https://github.com/anthropics/claude-code/issues/11535) ask for token/context data in statusline or hook input, and note that summing JSONL token values comes out lower than `/context`. There is no statusline field, hook field, or transcript record that yields the per-category split.

**4. Live mid-run per-category reading (dead-end).** `/context` queues until the agent is idle and renders to the screen; it does not interrupt a run and does not emit structured per-category data. There is no workaround that produces a live mid-run per-category breakdown. If you need the categories, the only path is to run `/context` at an idle boundary and parse its screen output (the same fragile approach you are trying to avoid), accepting that it is a point-in-time snapshot, not a live feed. For live monitoring, the overall percentage (items 1-2) is the clean signal; reserve `/context` parsing for occasional idle-time category audits.

## Recommended path

**For mode-setting, the single best approach for your tmux-TUI setup is deterministic `BTab` cycling with verification, pre-armed at launch.** Concretely:

1. **Launch flags.** Start every session with `--allow-dangerously-skip-permissions` so `bypassPermissions` sits in the cycle without activating. If your account qualifies for `auto`, accept its one-time opt-in once so it is permanently in the cycle; record per session which optional modes are enabled so your press-count math is correct.
2. **Make the cycle key bulletproof.** Add a `~/.claude/keybindings.json` that rebinds `chat:cycleMode` (and `confirm:cycleMode`) to a single unambiguous key you can trivially send via `tmux send-keys` and that nothing else uses. Keybindings hot-reload, so this applies to running sessions immediately and removes any `BTab`-encoding or platform risk. (Validate the chosen key with `/doctor`, which flags multiplexer conflicts.)
3. **Set absolutely, verify always.** To reach a target mode: read the current mode (prefer the hook payload `permission_mode` if you adopt hooks, else the transcript `permissionMode`, else the status bar), compute forward distance in the known cycle, send the rebound cycle key that many times (gated on a verified safe/idle state, one press per call with small sleeps), then re-read and retry on mismatch. Handle the auto opt-in menu if it appears. For `dontAsk` (not in your UI) or a guaranteed bypass, use a startup `--permission-mode` flag instead of cycling.
4. **Wire `set_mode()` accordingly** (it is currently a no-op): implement the read-compute-send-verify loop above, advertise `set_mode` in `CAPABILITIES`, and surface plan as a special case (cycle to it, or test whether `/plan` persists as a cleaner typed entry).

**The adjacent finding that should change your plans: adopt a hook event stream.** Independently of mode-setting, register Claude Code hooks (ideally HTTP hooks posting to your sidecar) on `PreToolUse`/`PostToolUse`/`PermissionRequest`/`Stop`/`Notification`/`UserPromptSubmit`/`SubagentStart`/`SubagentStop`/`StopFailure`. This gives you a push-based, low-latency run-state stream that is more reliable than screen polling, hands you `permission_mode` for free on every tool/turn event (authoritative live mode-reading, which also makes your cycle-verification trivial), and lets a deny hook enforce dashboard-controlled guardrails in any mode. This is exactly how agent-dashboard, the closest analog to your project, is built, and it is the highest-leverage architectural change available. Keep your screen heuristics as a fallback for hookless sessions, and keep the answer-keys permission path as the universal prompt handler.

**Two quick repo fixes regardless of the above:** switch transcript discovery to the `~/.claude/sessions/{PID}.json` -> `sessionId` mapping, and make the context window model-dependent (or read `context_window_size` from the statusline payload) instead of hardcoding 1,000,000.

### Honest conclusion

No clean absolute mode-setting method exists for a running Claude Code TUI driven as a terminal process. There is no slash command, no `settings.json` hot-apply, no env var, signal, or IPC, and no per-mode keybinding action; the only in-session lever is the Shift+Tab cycle. Keystroke-cycling with read-and-verify (pre-armed with launch flags so the cycle is deterministic, and ideally with the cycle key rebound for reliability) is the best available method, and it is genuinely reliable when built that way. The mature orchestrators confirm this by observing mode via hooks rather than setting it. Your hypothesis stands.
