A single-window desktop app for running and coordinating many Claude Code agents at once. This document is the ground-truth reference for the dashboard's **UI/UX intent** — what each part is *for* and *why it behaves the way it does*.

This README avoids pixels/hexes/labels: the **values** all live in **[`tokens.css`](tokens.css)** (the single source of truth), and the [Design system](#design-system) section below holds the *rules* for using them and points to that file.

## Purpose & vision

Running several Claude Code sessions in tmux means constant tab-switching, copy-pasting prompts between terminals, and no real visibility into what each agent is doing or how they relate. The dashboard replaces that with **one window** where you can see every agent, its state and context, the messages flowing between agents, and compose/send prompts — **without ever touching the raw CLI.**

The thing that makes this more than "several terminals in a grid" is **context-sharing between agents** — links, a shared scratchpad, and agent-to-agent conversation. That is the defining feature; everything else exists to support it.

Guiding principles that shape the whole UI:

- **Bias toward on-screen, not floating.** Prefer giving a surface a pane or step-into view over floating it. Anchored, transient bits — menus, pickers, hover popovers, quick confirms — are fine; the thing to avoid is movable or stateful floating windows. When something wants that, make it a pane instead.
- **Compose-first.** Typing into a raw CLI is the thing this tool is meant to replace, so the prompt-composing surface is always present and is the primary action.
- **One identity per agent, everywhere.** Each agent gets a color + icon used consistently across the graph, feeds, log, history, and chips, so you can track an agent at a glance — see [Agent identity & naming](#agent-identity--naming).
- **Route control through the GUI, including approvals.** Anything you'd normally approve in the CLI — permission prompts especially — surfaces in the dashboard so you never drop back to a terminal to unblock an agent. (This is why approvals get a first-class home; see [the Inbox tab](#the-inbox-tab).)
- **Bridge is the backbone; the dashboard is the skin.** Agent lifecycle/send/read already exist below the UI; the dashboard is a control layer on top.

> **Out of scope (deliberately):** per-agent cost/spend tracking, a diff viewer, a terminal-UI (TUI) version, and keyboard-only (F-key) control — all considered and explicitly dropped. *(A raw live-CLI/terminal feed was previously dropped here too, on the grounds that the Team Feed's Messages tab covers what each agent is doing. That was reversed: a raw feed is now provided **on-demand** by the [Agent panel's Console tab](#agent-left-pane) — an expandable surface scoped to one agent, **not** an always-on pane. The Messages tab remains the curated, cross-agent **rendered** view; the Console is the raw, single-agent one.)*

## What it physically is

A desktop application (Electron + React) presenting one window. Each agent is a real Claude Code session running in tmux on WSL2, driven by a local Python (FastAPI) sidecar through the tmux bridge that already handles create/send/read/status; tmux also serves as the crash-recovery safety net. The tmux bridge is the primary path the dashboard is built around, and the design target throughout. The in-process Claude Agent SDK is not a whole-system fallback; it is reserved for specific non-interactive tasks that need no real terminal (such as the Revise / Summarize utility passes). Agents themselves always run on the bridge. That's the whole stack the UI sits on; this README stays focused on UX.

## Layout — the three-column model

A title bar on top, a status footer on the bottom, and three vertical columns between them. The **Agent** column (left) is the narrow one, full-height; the **middle** and **right** columns are the wide ones, each split top/bottom into two stacked panels — **Team Graph** over **Library** in the middle, **Team Feed** over **Prompt** on the right. Columns and their stacked sections are separated by draggable splitters. The whole frame is **full-bleed** — the header, the three-pane body, and the footer meet flush and run edge-to-edge, with no outer margin.

## Panels

Each panel below is the single source of truth for its own behavior. Cross-cutting ideas (identity, linking, the scratchpad, lifecycle) are described once in [How it works](#how-it-works--concepts-that-span-panels).

### Team Graph (middle, top)

The roster of live agents, one **card** per agent. Each card is a **fixed square** with a single-column layout, top → bottom: a thin **top meta strip** (the agent's **age stamp** — created datetime + auto-scaling "ago" duration, small + left-aligned — paired with the **status badge** in-flow at the right end); an **identity row** (icon + color, role, number + name on the left; the **model** right-justified and bottom-aligned to the name, with an **opus-only FAST lightning bolt** trailing it — bright when on, muted when off, hidden for non-opus); a full-bleed divider; three **settings chips** (mode · effort · think — icon heading + value, spread full width); **two labeled health-colored bars** — a **Ctx** bar (context usage %) and, beneath it, a **Turns** bar (live count against the agent's Max turns, e.g. *Turns 34/50*), each with a leading label and an inline value; an **activity band pinned to the bottom** (a textless **Run strip** + a two-track **Marquee** of the agent's current activity, each separated by full-bleed dividers); and a **subagent strip** of small **rectangular** clickable id-badges (each **filled with its run-state status colour with a bold white id** — no dot — shown collapsed to **one row** with a chevron; clicking expands an **in-flow accordion drawer** that grows the card downward to reveal the rest). The currently-selected card is filled (light teal) and marked as the one being viewed.

- **Selection drives the app.** Clicking a card focuses that agent in the Agent panel.
- **Status at a glance.** A single **status badge** (in-flow at the right end of the top meta strip) shows one of **four** run states — **active / idle / pending / error** — where *pending* (the warm attention tone) means "this agent is waiting on you" and *error* (the danger tone) means "this agent's run failed." It's deliberately **binary, not a count** — an agent can only be blocked on one thing at a time (see [Inbox](#the-inbox-tab)), so the *number* of agents awaiting you lives on the Team Feed's Inbox tab, not on each card. The badge is also a **shortcut button**: it jumps to **Inbox** when pending **or error** (both surface in the Inbox — a failed run as that agent's **Error** card), **Prompts → History** when active, or **Prompts → Compose** when idle.
- **Run + activity, pinned to the bottom.** The **Run strip** is a textless, single-flat-colour progress bar of *opposite polarity* to the load bars (no danger ramp for the live states) keyed off status — active green · pending warm · idle muted/empty · **error a solid danger bar** (a halted/failed run, static — no barber-pole, since it isn't working) — using a **barber-pole indeterminate** animation for "working, % unknown". When the agent **self-reports a checklist** of its major operations (the segmented run-strip model — docs/ARCHITECTURE.md §7.10), the strip instead renders **done ÷ total** as a **segmented bar** — equal segments split by navy separators (`--border`), where the **current segment carries the run's state**: *running* keeps the green **barber-pole shimmer**; a **blocking pause** (Permission / Plan / Decision / max-turns) recolours it **warm** (`--warning`, static — the run is waiting on you and the card's status badge reads *pending*); a **step-scoped error** recolours it **danger** (`--danger`, static). A non-blocking **Warning never pauses it** — the segment keeps shimmering and the badge stays *active*; a run **paused *between* steps** puts the warm colour on the **next** (upcoming) segment. A **structural / whole-run error** (the session died, a rate limit) skips the segments and falls back to the flat solid-danger `.run-error` **whole-bar**. **Retry** redoes the whole run from the last command, so the **segments reset** (done÷total recomputed) — never a step-level resume. The segmented bar keeps the **same height as every other run-strip** (no always-on label), so every card's bar seats identically in the band. The current **step name is hover-only** — carried on the bar's tooltip (*Step 4 of 5 · Run typecheck*) — so it never changes the card layout. The **barber-pole stays the floor** for any run that publishes no checklist, and a fabricated % is never shown (the denominator can grow mid-run, so the bar can honestly step back). The hover step name is **separate from the Marquee** below (the checklist's discrete step is the bar's tooltip; the Marquee is the continuous liveness stream). Beneath it, a neobrutalism two-track **Marquee** streams the agent's current activity as one scrolling line, pausing on hover and going static + muted when idle.
- **Health coloring.** Both the Turns bar and the Ctx bar use the same good→bad ramp the Timeline uses for context % (green → amber → red), so load reads at a glance without decoding the agent's identity color.
- **Subagents** appear as small **rectangular clickable** id-badges on the parent card — each **filled with its run-state status colour** (active / pending / idle / error, reusing the `--success` / `--warning` / `--muted` / `--danger` status tokens) with a **bold white id** (`A1`, `A2`, `B1`, … — **group + member**: the parent run lettered A·B·C… in occurrence order, then the spawn order within that run; an identifier, *not* a count, and **no `s` prefix** — full sender form `coder-01 › A2`), matching the larger status-badge treatment. The strip is an **in-flow accordion drawer** (like the Inbox cards): collapsed it shows only the **first row** of badges (vertically centred in the trigger) with a **chevron** action icon on the right; clicking the strip toggles it **open**, growing the card downward to reveal the remaining badges wrapped onto new rows (chevron right → down). The state **sticks** (a toggle, not a hover popover), and the chevron/drawer appear **only when the badges wrap past one row**. The section is **reserved at a fixed size even when empty** (0-subagent cards show a muted "no subagents" placeholder), so collapsed every card stays the same square height — a card only grows past its square while its drawer is open (its grid row grows; square row-mates stay square). **A badge click is isolated from the card's own select/drawer** (per-badge `stopPropagation`), then runs the **subagent-badge action** (the subagent model, docs/ARCHITECTURE.md §7.17 — which **resolved OQ-1**): it **(1)** focuses the **parent** agent in the Agent panel, **(2)** opens that agent's Details **[Subagents](#agent-left-pane)** accordion scrolled + highlighted to that subagent's row, and **(3)** scopes the Team Feed filter to that subagent on the **Messages** tab — clicking a helper selects its parent and scopes everything to the helper. (Its 4th **error** state — `sb-error` — mirrors the status badge.)
- **Per-card inbox — the footer's right half (`node-inbox`).** Beside the subagent strip (which keeps the **left**), a small **envelope + count** rectangle mirrors that agent's **open [Inbox](#the-inbox-tab) items** — the *parallel path* to the status badge (the badge is the fast jump to the **one blocking thing**; this envelope is the agent's **full open queue**, including non-blocking **Warnings**). It has **two states**: open items → a **teal** envelope + count, click to expand an **in-flow drawer** of **typed rows** that each **deep-link into the Inbox** (scroll + expand + select + highlight that agent's card *of that type*, mirroring the status badge's own jump); none → **dimmed + non-expandable**. The **count = open / unaddressed items** (*not* read) — so it can read **>1**, and can be **>0 while the status badge still reads *active*** (a non-blocking Warning is an open item that doesn't block). A non-blocking **[Response](#the-inbox-tab)** item (a run ended with unreviewed output) counts here **exactly like a Warning** — and because it **coalesces** (one item per agent; a second unseen run *updates* it, never adds another), it moves the count by at most one while the **status badge stays plain *idle*** (no fifth badge state, no state-dependent click). Because the footer and the badge are **two views of the same item**, resolving via *either* completes it and clears it from *both*; an item leaves **only when completed**, never on a glance (no read-tracking for now). The drawer is **independent** of the subagent drawer (both can be open, each grows the card downward), and its clicks are **isolated** from card-select (`stopPropagation`, like the subagent badges). Built from the same request data as the Inbox tab, so the two can't drift.
- **Scales past what fits.** The graph shows many agents (not just a few); cards lay out in a grid and the view scrolls.
- **Linking starts here.** Select the agents you want to connect and use **Link Agents** to open the Link Config drawer — see [Linking & context-sharing](#linking--context-sharing).
- **Links as edges *(planned)*.** Active links are intended to render as **directed arrows between cards**; the current mockup doesn't draw them yet (linking lives in the drawer + footer count for now).

### Team Feed (right, top)

The top panel of the right column (the Feed sits above [Prompt](#prompts-right-bottom)): a real-time, cross-agent view of communication, attention, and events, as four tabs over a **shared agent From/To filter** — a header selector (in a sub-bar, labelled **From/To**) whose **in-flow accordion drawer** (it pushes the tab content below down rather than overlaying it — a sticky toggle: no outside-click dismiss, Esc closes, chevron right → down) is a **2-level tree**: agent identity rows on top (including a **User** row, with the reserved **System** row **second, after User** — see [Agent identity & naming](#agent-identity--naming)), and any agent that has spawned subagents carries an **expand chevron + a selected-of-total sub-count badge** (e.g. `2/4` — selected leaves over total, live-updating as leaf selections change) revealing its **indented subagent rows** (run-state badged, e.g. `A2 · Explore`). **Every surface that lists agents/subagents** — this filter tree, the Prompt **To**/**From** lists, the graph cards, the Agent panel's Subagents audit — **renders from or reconciles to one shared roster**, with the **Team Graph cards as ground truth** where surfaces conflict, so rosters and counts can't drift. **Parent-select = the whole subtree**, **leaf-select = just that subagent**; a single **contextual All/None** toggle still spans the lot. Selected agents fill the trigger as identity **badges** (a selected subagent reads compactly as `parent › A2`; collapsing to a **+N** chip past a cap). The filter is **one shared field across all four tabs and applies universally** — a subagent selection is simply **inert on tabs with no subagent traffic (Scratch, Inbox)**, matching nothing there, rather than being toggled per-tab. **Filter-only, never addressable:** subagents appear here (sender-side) but **not** in the [Prompt](#prompts-right-bottom) compose-**To** — you can't send *to* a helper (To stays parents + User/Scratch); the reserved **System** identity follows the same rule (present in this filter tree only — excluded from Compose To, Compose From, and the History From filter).

Tab order puts the everyday streams first and the **Inbox** (approvals) last (Messages · Scratch · Log · Inbox).

| Tab | What it shows |
|-----|----------------|
| **Messages** | The team's traffic, attributed and color-coded by agent, merged into one stream with two in-tab filter groups — a **Type** segment for **direction** (Sent / Received) and a **Content** segment for how much of the message to include (**Text** / Thoughts / Read / Write / Bash / Diffs / Meta — **Text** leads the row and toggles the main reply text, Claude Code's `text` content block, **default on**). Direction is anchored to **you, the operator**: **Sent** = prompts you pushed to the team; **Received** = everything coming back from agents. (So **User** only ever reads as *Sent* — it never carries an incoming tag.) **Subagent events stream nested/indented under their parent** — visible by default, the Type/Content filters applying as normal, and always **Received** (helpers never receive operator sends; the parent badge leads, then the run-state sub-id badge); a *collapse subagent chatter* toggle is an allowed fast-follow. The direction tag on each message is intentionally neutral — no special color. Each card's **header** leads with the **agent badge** (at the full `.badge` size, the reviewer-chip standard), then a **"→" recipient mini-badge** (the addressing model, docs/ARCHITECTURE.md §7.2) for who the message is **addressed to** — a deliberately-smaller identity badge (User, an agent, or Scratch; a multi-target send collapses to **+N**) — then, for **received** messages, a **status badge** (**Active** in-flight / **Complete** / **Failed** red), then the **Sent/Recv** direction tag (order **agent → recipient → status → dir**). The recipient is **routing (addressed-to), not visibility** — every message shows regardless: an agent reply's recipient defaults to **User**, a user-**Sent** message carries its **To/Target** selection (multi), a link fire targets the linked peer, and a scratchpad post reads **→ Scratch**. The expanded body is a **contiguous Library-style rail** that **fills the card's content surface** (rows butt together with hairline dividers, the rail cell flush against the content — no inner bordered panel inset within the card; the Messages card's body padding is dropped so the rail reaches all four edges, with each row's own cell padding giving the text its breathing room): a **title row** holds the **turn number** ("Turn N", doc-title style) in its content with an **empty rail box**, then the **reply text** ("txt" — Claude Code's `text` content block, toggleable like any block via the Content filter's **Text** toggle), then **each included tool block**, each rail box carrying a tight 3-char type tag (`txt · tht · rd · bsh · dif · wrt · mta`). Cards use the shared **select-to-act** model (the same language as the [Library](#library-middle-bottom) doc rail), **teal throughout** (no pink): each block rail **toggles** its block and **multi-selects within a card** — a contiguous run reads as **one unbroken light-teal band** (rail + content, no seam); the **title rail (or the card header) select-alls the whole message**; the rail tracks the active **Content** filter so only currently-visible blocks are selectable. **Hover** lights the row (or, from the title rail, the whole card) canvas cream. The **chevron** expands separately, so *select ≠ expand*. (Whole-message → Attach; a block run → Embed, wired in [Linking & context-sharing](#linking--context-sharing).) The shared **Messages footer** carries **select/deselect-all** · the **[merged Export control](#sharing--exporting--the-merged-export-control)** (one dropdown — **Export**: Copy selected · Export selected → file; **Add to prompt**: Embed · Attach — each item selection-gated, export-to-file lands the selection in [Library → Documents](#library-middle-bottom) as a new doc) · **Summarize** · plus a rightmost icon-only **Stop** (solid danger — stops the agent's run); it wraps to a second row on a narrow column. **Scratch / Log / Inbox cards share the same selection model** — a **header-click** selects the whole card (light teal), a **separate chevron** toggles the dropdown, and the card attaches via the [Embed/Attach control](#linking--context-sharing); they have **no checkbox** and **no internal sub-selection** (Inbox is one request per card, so whole-card select is for attach parity, not multi-item grouping). |
| **Scratch** | A live view of the [shared scratchpad](#the-shared-scratchpad): each post stamped with agent ID + time. |
| **Log** | A timestamped system-event stream (started, committed, flagged, link fired, permission/approval requested…), color-coded by agent; platform-level events carry the reserved **System** badge. |
| **Inbox** | The team-wide **requests inbox** (last tab) — typed sections Error · Warning · Permission · Plan · Decision · **Response**; see [The Inbox tab](#the-inbox-tab) just below. |

#### The Inbox tab

The team-wide **requests inbox** — the human-control surface that keeps agents unblocked without dropping to a terminal. Because a Claude Code agent is single-threaded, **it can only ever be blocked on one thing at a time**, so this tab shows **one card per agent that's waiting on you** (narrowed by the same agent chips as the rest of the feed), and the tab's badge is the **fleet total** — how many open items need you right now. The blocking one-thing-at-a-time model has one **non-blocking complement**: the **Response** section at the bottom — a run that **ended with output you haven't reviewed** blocks nothing, it just waits for a look. Cards are **grouped into typed sections** — **Error · Warning · Permission · Plan · Decision · Response** (ordered most-to-least urgent; **Response**, the one non-blocking, non-request section, sits last) — so the **section header carries the type** and the cards themselves stay clean (**no per-card type badge**). Each card carries the **agent's identity** (icon + role + name); an **Error** *or* **Warning** card also carries a short **subtype** label in the section's colour (Error: the failure class, e.g. *Connection*, on `--danger`; Warning: the limit hit, e.g. *Max turns*, on `--warning`):

| Section | For | Controls |
|---------|-----|----------|
| **Error** | Any **abnormal block to execution** — API/model, tool/MCP, environment/connection, config, or a stall (distinct from a Permission prompt, a Decision, or a normal Lifecycle auto-stop); carries a short **subtype**. **System-wide failures** — infrastructure (tmux/WSL2/sidecar down), account-level (rate/usage caps, auth expiry), shared services (a global MCP outage) — front the reserved **[System](#agent-identity--naming)** identity instead: one card for the fleet, not N identical per-agent cards | inline error text · **Retry** · **Dismiss** · Reply *(no View, no Forward; on a **System** card Reply is **disabled** — greyed, not removed, the Export-menu convention)* |
| **Warning** | An **attention-needed notice that isn't a hard block** — e.g. nearing the context limit (compaction imminent), an approaching rate/usage cap, or a **hit auto-stop limit** (e.g. *Max turns*); carries a short **subtype** | **Dismiss · Reply (the simplified Warning treatment — docs/ARCHITECTURE.md §7.9).** A Warning is a **plain FYI**: **Dismiss** (its **sole completion action** — clears the warning from the Inbox; pink `btn-main`, this card's one primary "commit here", *not* danger) and the shared **Reply**. It carries **no cap-specific actions** — the old *Continue / Raise cap / Stop / Acknowledge* set was **removed** (no per-warning dependency) — and it **never auto-clears**: a Warning leaves the Inbox only on a **manual Dismiss** |
| **Permission** | A tool/command the agent wants to run (the same prompts you'd normally approve in the CLI) | Approve · Deny · Reply |
| **Plans & Docs** | A plan/handoff — or a doc-edit hand-off — to review before it proceeds | **Review** (→ Library → Plans, or → Documents for a `doc-*` hand-off) · Reply |
| **Decision** | The **`AskUserQuestion`** surface — **one question per card**, with candidate options to choose between | pick an option, then an explicit **Approve** (disabled until you've selected) · Reply |
| **Response** | **Non-blocking** — a run **ended with output you haven't reviewed**: the answer to *your* prompt (the five sections above are agent *requests* of you; this is the lone *response*). **One coalesced card per agent** — a second unseen run **updates** the card (an **×N runs** marker), never stacks. Opens on run end; **stays open (coalescing)** if a new prompt fires unseen; clears on **Retire/Delete** | **View** (→ Team Feed → Messages with the filter scoped to that agent, scroll + **flash the run's final reply** — the plan-flash jump pattern; **completes the item**) · Reply (**also completes** here) — **no Dismiss**, and **no read-tracking**: like every card it "leaves only when completed, never on a glance" (reading Messages organically leaves it open — an explicit to-review queue) |

**Plan cards drop Approve/Reject** — all plan approval and the agent-review verdicts live in the [Library Plans tab](#library-middle-bottom); the Inbox **Plans & Docs** card only **Reviews** (jumps to the matching Library tab — **Plans**, or **Documents** for a `doc-*` hand-off — expanding + highlighting the matching plan/doc) or **Replies**. Colour: **Deny / Stop** and the **Error card's Dismiss** (a give-up / clear) use the **danger** colour; **Approve** and the **Warning's Dismiss** (its sole completion action, per the Warning simplification) stay **pink-primary**. *(The old Continue / Raise cap are retired with the Warning simplification.)* **Permission is binary Approve / Deny (+ Reply)** — "Always allow" was **removed entirely** (docs/ARCHITECTURE.md §7.11 Permissions): no always-allow button and no always-allow rule-persistence, now or later. **Section headings carry an escalating severity ramp** — the heading **label text + its matching count badge** run **Error** (`--danger`, red) → **Warning** (`--warning`, amber) → **Permission** (`--inbox-permission`, blackish-orange), while **Plan + Decision + Response** stay the neutral `--muted` tone. Only Permission needs a dedicated token (`--inbox-permission` `#a9710f`); Warning **reuses the existing `--warning`** and Error **reuses `--danger`**. **Error stays the single alarm**: it's the only section whose cards also carry a **danger left-edge** (Warning and the rest are heading-coloured only). This supersedes the old per-card reddish→copper request-type ramp (it ranks severity by section heading now, not a per-card badge).

- **Cards are expandable + selectable.** Like the Log/Messages cards, each Inbox card **collapses to its identity row + title** and **expands via its chevron** to reveal the request detail + its controls; a **header-click** selects the whole card (light teal) into the shared Copy/Export/Attach strip — there is **no checkbox** (the same select-to-act model as Messages/Scratch/Log). A **Pending** status badge on a [Team Graph](#team-graph-middle-top) card jumps here and **scrolls to, expands, selects, and highlights** that agent's card (e.g. a failed run surfaces as that agent's **Error** card).
- **Review routes to the plan or doc** *(Plans & Docs cards)*. The **Review** button **hands you to the matching Library tab** — the **Plans** tab for a plan, or the **Documents** tab for a doc-edit hand-off (a `doc-*` entry) — expanding and briefly highlighting the matching plan/doc so you can read/edit the full proposal there — rather than embedding an editor inside the card. This is the deliberate cross-link between the two surfaces: the *plan* (or *doc*) is a document (it lives with the docs); its *approval* lives in the Library tab, and only its **review hand-off** surfaces here.
- **Reply is generalized and routes to the Editor.** Every card's **Reply** (far right, set off from the other controls) **pre-fills the [Prompts](#prompts-right-bottom) Editor with a frozen reference block** quoting the card's contents — built from the shared [inserted-block primitive](#the-templates-flow) (an `embed` block, *not* a bespoke widget): a **Decision** embeds the question + its options; **Plan / Permission / Error** the request or error detail. It also **pre-targets that agent**, so a free-form reply is composed and sent from the one place all prompts originate. Change the **To** to forward — which is why there's no separate Forward. *(Error's **Retry** likewise loads the last command into the Editor, routing through it rather than the Console.)*
- **Mirrored on each agent's card.** Every open request also surfaces on its [Team Graph card](#team-graph-middle-top) footer — the `node-inbox` envelope+count (the agent's **full open queue**, including non-blocking Warnings and **Responses**; a **System** card has no graph card, so it has no envelope mirror). The two are **parallel views of one item**: resolving it here *or* from the card clears **both**, and an item leaves only when **completed** (there's no read-tracking — the count is open items, not unread). A per-agent count can therefore exceed the binary status badge, which still points only at the one *blocking* thing.

### Agent (left pane)

Everything about **one** agent — whichever is selected in the graph. Three tabs share the pane.

- **Details** — opens with a **header** (agent identity — tile · role · name — with the **status badge** and the agent's **created-time**, a datetime + auto-scaling "ago" duration, stacked right-aligned beside it), then the agent's configuration and live readouts in three deliberately-separated bands:
  1. **On-demand config** (Role, No., Name, Description, Model, Skills, Tools, **MCP servers**, **Plugins**, **Deny rules**, Color, Icon). These are *greyed by default*; click a field's **pencil** to edit, then **save** — so configuration can't be changed by accident mid-run. **Role** is a dropdown (with free-text entry) populated from the user- and project-level `agent.md` files; picking one in **Create** prepopulates the related fields from its front matter. **Skills** and **Tools** are multi-select dropdowns (skills from the skills files; tools = the native Claude Code tool set) that keep floating chips. **MCP servers**, **Plugins**, and **Deny rules** (docs/ARCHITECTURE.md §7.15 Settings) are the same multi-select primitive, exposing this agent's **per-agent scoping** that the engine accepts at create but didn't surface before: which **MCP servers / plugins** it may use (none = all available; these take effect at the agent's **next launch/restart**, so editing them on a *live* agent means "applies on next start") and the **deny rules** that hard-block it (per-agent tool scoping leans **deny-based** — deny is the reliable hard-block; `--allowedTools` is ignored under bypass, a claude bug). This is deliberately distinct from [Settings](#settings-step-into-view)' *global* registry ("is server/plugin X available at all" vs. "may agent Y use server X"). *(Model is intentionally editable, not locked: an agent's model can be changed mid-run.)*
  2. **Always-editable while running** — **Mode**, **Effort**, and **Lifecycle** auto-stop limits, the knobs you reach for during a run, so they stay live. The **Mode** block carries the permission-mode segmented control (**Plan · Ask · Edit · Auto · Bypass**, where Bypass is the bypass-permissions danger option) and, beneath it, two **labeled toggle-buttons** on one evenly-spaced row (icon + label) — **Opus Fast-Mode** (maps to `/fast`) and **Thinking Mode** (extended thinking). Each is a button whose label reads its own state (*Opus Fast-Mode Off/On* · *Thinking Mode Off/On*) and whose whole face fills teal when on. **Opus Fast-Mode is gated** (dimmed + inert) **to the Opus model** — enabled only when this panel's **Model** is **Opus** (`/fast` is an Opus mode); **Thinking Mode** has no gate. See [Lifecycle & autonomy](#lifecycle--autonomy).
  3. **Live readout** — the **health-colored** context-usage bar (with a Compact action) that **expands to a per-category breakdown** whose head carries a native **turn-scope `<select>`** (*Total* or a specific *Turn n* — descending to Turn 1, n = the agent's turn count): *Total* is today's whole-window snapshot (categories as % of the window, the cutoff line, both loaded-context sub-sections); picking *Turn n* rescopes the header / bar / category rows to that turn's contribution (no Free space, no cutoff; rows sum to ~100% of the turn) while the *Memory files* / *Custom agents* sub-sections stay put (they describe loaded context, not a per-turn quantity). The two denominators differ on purpose — the header % is share of the **window**, the row %s are share of the **turn**. **Context refresh + on-demand pull** *(design contract — the refresh/pull endpoint itself is backend work, to be implemented later)*: context can't be read while the agent is mid-run, so the readout **refreshes automatically between runs**, and **opening the accordion triggers a direct pull** — while a pull is in flight the breakdown shows the **loading-strip** loading state (the bar swaps to the teal shimmer; the stale category rows + head total dim) and the fresh data renders when it lands. A **Turns** readout (the count inline beside the label like Context) that **uses the same accordion primitive**, expanding to a **by-tool** breakdown (Read/search · Edit · Bash · MCP · Subagent · Web, plus a **Coordinating** slice for cross-agent work and a *Remaining* segment closing the turn budget) — the multi-agent parallel to Context's by-category view, and the categorical summary to the Timeline's per-turn log. Then, under its own **"Timeline"** section heading (the standard `.sec-h`, matching the Context/Turns section labels), the **Timeline**: a point-by-point list of the messages sent to the model that fills the rest of the panel, with two actions over the same list — **Rewind** (roll this agent back to the chosen point) and **Handoff** (branch from that point into a *new* agent). See [Rewind & Handoff](#rewind--handoff). Then, pinned **below the Timeline and above the Retire/Delete footer**, a collapsed **Subagents** accordion — a **read-only audit history** of this agent's subagents, its header carrying a **total-subagent count badge**; it expands to rows **grouped by run** (A·B·C…), each: id + type (`A2 · Explore`) · task · status (running / done / error) · usage (tokens · tools · duration) · model · a link to its own transcript. (The [Team-Graph](#team-graph-middle-top) card's subagent strip stays the glance view; this is the audit home, and the subagent-badge click lands here.)
  - **Footer:** **Retire** + **Delete** — Retire (soft red-text-on-cream) ends the session; **Delete** (a solid-red-fill / white-text variant, to its right) **permanently** wipes the agent — its configuration and transcripts — and removes it from the roster/graph and any links. Because it's irreversible, Delete is gated behind an inline **"this can't be undone" confirm** (the same confirm pattern as Retire, in the danger tone).
- **Create** — the new-agent wizard: Role, No., Name (with a randomize affordance), Description, Skills, Tools, **MCP servers**, **Plugins**, **Deny rules** (the same per-agent scoping as Details), Color, Icon, Model, Mode (with the same Opus Fast-Mode / Thinking Mode toggles), Effort, Lifecycle. **Handoff** lands here with the source agent's settings prepopulated (and editable).
  - **Footer:** **Create · Reset · Cancel**.
- **Console** — the focused agent's **raw Claude Code terminal feed** paired with a **slash-command runner**, both scoped to the single selected agent (so a command's target is unambiguous — *no* multi-select, which is exactly why this lives here and not in the multi-target Prompt or the multi-filter Feed). The feed **faithfully mimics a real Claude Code terminal** — a dark monospace surface with the native markers (`>` input · `●` tool calls with `⎿` results · `✻ Thinking…` · a model/status line · `+`/`−` diffs · a native permission-prompt box) — so it reads as the agent's actual CLI output, the raw counterpart to the Team Feed's rendered [Messages](#team-feed-right-top). Because a native terminal is *not* the dashboard's neobrutalism surface, the feed uses a **self-contained `--term-*` palette** — a **documented exception** to the otherwise-strict "every value comes from `tokens.css`" rule. The exception is scoped to the feed surface only: the surrounding **agent bar, command catalog, run bar, and Expand control stay neobrutalism**. The command surface is the **single home for every slash command** — *not* split into agent-specific vs. shared, because everything simply runs on the focused agent: a **scrollable catalog grouped into conceptual clusters** (Session & context · Model & behavior · Info & status · Tools & integrations · Project & custom · System), each entry a command + a one-line description, with a filter box. Commands that already have a dedicated home (e.g. `/model` in Details, `/mcp` in [Settings](#settings-step-into-view)) are still listed here, tagged *also-available-there* — the catalog is deliberately complete. Picking an entry stages it into the run bar; a typed-or-staged command runs against the focused agent and its output lands in the feed.
  - **Width via Expand.** Because raw output and a grouped catalog both want more room than the narrow Agent column gives, the Console carries an **Expand** control that pops it into a **step-into view that covers only the left + middle columns** — its right edge stops at the left edge of the right column, so the **Team Feed + Prompt stay visible** while you work against the live terminal (the edge is recomputed on window resize and splitter drags). This is a **deliberate difference from [Settings](#settings-step-into-view)**, whose step-into covers the *whole* body: the Console is a working surface you reference against the live feed, where Settings is a configuration modal you step fully into. Both share the step-into mechanics (replace their region, return on **Close** / Esc) and the [bias toward on-screen surfaces over floating ones](#purpose--vision). The pieces **reflow** between states: in-column the feed leads and the catalog rides as a **slide-up panel** over it (full panel width); expanded, the catalog becomes a persistent **left rail** beside a wide feed, with the run bar across the bottom — the rail-beside-body shape [Library](#library-middle-bottom)'s entry-nav already uses.
  - **Footer:** the **run bar** itself (the command input + Run) — the Console's bottom strip, in place of a button row.
  - *(The Console reflects whichever agent is focused, like the rest of this pane: selecting a graph card repaints its identity bar, feed header, and run-bar target — the same selection-drives-the-panel wiring as Details.)*

### Prompts (right, bottom)

The compose-first heart of the app — the bottom panel of the **right** column, beneath the Team Feed. Three tabs sit over a **shared sub-header**, so the way you address a prompt is consistent no matter which tab you're on.

- **Shared From + To** — a header **sub-bar** of two **accordion selectors** taking **equal space** (in-flow drawers, like the Feed filter), so Compose and the streams span the full width. **To persists across the tabs; From forks per tab** — Compose's single-select *source* vs History's multi-select *filter*, a deliberate asymmetry: multiple sources only make sense for filtering history, never for sending:
  - **From / Source** (single-select, **Compose**) — who the prompt is *from*: **User**, or an agent (sending *as* that agent for coordination). A compact **accordion selector** whose trigger shows only the active selection as a collapsed identity badge; no All/None (single-select needs neither). **On the History tab the From column swaps to a multi-select *filter*** — the same list as To **minus the Scratch row** (agents only): agent-sourced prompts gate on the selection, while your own (User-sent) prompts always show — User isn't a row there. *(The reserved **System** identity appears in none of these — neither Compose From, Compose To, nor the History From filter: it's [filter-only](#agent-identity--naming), feed-filter-tree only.)*
  - **To / Target** (multi-select) — who it goes *to*: a selector whose **drawer** is the full agent **identity-row list** (recolorable tile + two-line role·name, no truncation), led by a **Scratch** row that posts to the [shared scratchpad](#the-shared-scratchpad), with a single **contextual All/None** toggle (shows *All* until everything is selected, then *None*). Picked targets fill the trigger as identity **badges** and collapse to a **+N** chip past a cap; multi-target is simply picking several — there's no separate "broadcast" mode.
- **Tabs** (just **Compose · History** now — Templates folded into Compose):
  - **Compose** — the prompt-composing surface, top to bottom: a horizontal **attachment-chip strip** (under an **"Attachments"** section heading), then the **Editor**, then the **Templates** section. The **attachment strip** (above the Editor) holds the prompt's attachments as small chips styled like the [Library](#library-middle-bottom) nav cards — each with a **remove ×** and **click-to-open in the Library**; it's hidden when empty. The **Editor** (its heading reads **"Editor"**, renamed from "Compose") is a `contenteditable` rich field under that label; its header carries a **mic** ghost button (per-field voice dictation, right after the "Editor" label — red while recording) plus **copy / clear** ghost icons; it holds **free prose plus stacked, delimited inserted blocks** (the shared [inserted-block primitive](#the-templates-flow)) — **frozen `embed` blocks** (quoted content with a "from \<source\>" header + click-to-source) and **interactive `template` blocks**. Below the Editor, the **Templates** section holds a single-select **dropdown** (default **None** + the saved templates) and a **fill input** (greyed until a template is active): picking a template **inserts a selectable `template` block** at the cursor (not raw inline text), and the **active** template block is the one whose placeholder **pills** the fill input drives. **None** deselects the active template + re-greys the input; the box stays fully editable. See [the Templates flow](#the-templates-flow).
  - **History** — past prompts for the focused agent, attributed and **right-aligned timestamped** (the time pins to the card's right edge in both the collapsed *and* expanded states, matching Team Feed cards), and narrowed by the **History From multi-select filter** in the shared sub-bar above (the To list minus Scratch; User-sent prompts always show). Cards take the **same select-to-act model as the Team Feed cards** — a **header-click** selects the whole card (light teal), a **separate flush chevron** expands (*select ≠ expand*; **no checkbox**). **Edit** is a **ghost icon in the card header** (after the attachment chip, before the timestamp), and a prompt that **hasn't run yet** — any status but **Active** / **Complete** (Queued · Next · Held) — carries a **danger trash ghost** immediately after Edit, deleting the prompt (through the standard inline confirm) before it ever runs; the **footer** mirrors the feed strip — **select/deselect-all · [merged Export control](#sharing--exporting--the-merged-export-control) (Copy · Export → file · Embed · Attach, each selection-gated) · Retry · Stop** (icon-only; Stop is a **solid `--danger` fill with a white square icon**), with export-to-file landing the selection in [Library → Documents](#library-middle-bottom). A prompt that carried **attachments** shows a small **paperclip + count chip** in the card header (only when there are any); clicking it opens a popover listing each attachment, and clicking an entry **opens it in the Library** — images in **Assets**, docs in **Documents**. *(History-only for now; the same chip is built to extend to Team Feed → Messages later.)*
- **Action row** (the shared Prompts footer) — leads with a **paperclip attach** button that picks an image from the [Library → Assets](#library-middle-bottom) list — both directions register there, so Assets stays the source of truth — then the Revise and Response-format controls, and the Send split button at the right. *(Voice dictation is **not** in this row: each Editor — Compose, Plans, Documents — carries its **own** mic ghost button in its Editor header, bound to that field.)* Sending uses a pair of **split buttons** (a button joined to a dropdown chip), styled as a **pink primary** (Send) next to a **teal secondary** (Revise):
  - **Send** (primary) with a **timing** chip — **Now · Inject · Next · Queue** (no "Hold" or "Piggyback"; those only make sense for a link). The chip carries the delivery timing, reusing the same vocabulary as link [Triggers](#linking--context-sharing). **Now** interrupts and delivers immediately; **Inject** delivers to a *running* agent without stopping it (a one-shot message applied at the next safe boundary); **Next** waits for the next turn boundary; **Queue** appends to the agent's prompt queue.
  - **Revise** with a **scope** chip — **Grammar · Language · Refactor** (default Grammar): runs an AI cleanup/rewrite pass on the prompt before you send it, scoped from a light spelling/grammar fix up to a full restructure.
  - **Scope:** Compose offers Revise + Send (plus an **attach** paperclip; the voice **mic** lives in the **Editor header** as a per-field ghost button, not here). History swaps the row for its own footer (**select/deselect-all · the [merged Export control](#sharing--exporting--the-merged-export-control) · Retry · Stop**, icon-only; Edit rides in the card header) and hides the action row.

### Library (middle, bottom)

The home for **organizing and reviewing all of a project's documents and reference assets** — three tabs: **Plans · Documents · Assets**. The **Plans** *and* **Documents** tabs each carry a **count badge** (same rounded-square style as the Team Feed's Inbox badge) signalling **new items not yet reviewed** (in *review* or *draft* state) — Plans and Documents are the same reviewable-document surface, so both count; each badge hides when its count is zero. It's the other half of the middle column, beneath the Team Graph. (The panel was named "Documentation" while it held only docs; "Library" reflects that it now also holds images.)

- **The shared doc editor.** The **Plan** and **Documents** tabs render their files as **line-numbered markdown** whose line numbers match the raw file, so "see line N" is a real reference. **Authoring convention:** prose is soft-wrapped, **one line per paragraph** (a paragraph wraps to the container rather than carrying hard intra-paragraph line breaks; headings, list/checklist items, blank-line paragraph gaps, and code keep their own lines) — so the per-line anchors (gutter selection, `data-secid`, "see line N") stay stable, since re-flowing the container never renumbers a hard-wrapped paragraph's tail. An **interactive left rail** indexes every line and section: click a rail cell to **select a line, a whole section, or the entire document** (so a chunk can be shared or commented on). The rail cells are **color-coded by what a click selects** — pink = the **title** (selects the whole doc), dark teal = a **section**, light teal = a **line**; the selected text highlights in **light pink**, and re-clicking the active cell clears it. Inline code is tinted so it reads apart from body text. This **select-to-act** model is **shared with the Team Feed's [Messages](#team-feed-right-top) cards** (whole = pink, block = teal), so a selection means the same thing — *this is what you'll Embed or Attach* — in both surfaces.
- **The Plans tab is the review surface.** It lists the native Claude Code plan files (`~/.claude/plans/*.md`, shown under a single directory line + a "*N plans · N awaiting review*" count) as **expandable cards**. Each card's header is **two rows**: ① owner **agent badge** · title · a **promoted "done/total steps" chip** inline right after the title (hidden when there are no steps, which naturally makes it Plans-only) · then, right-aligned, the **feedback tally badges** (Approve/Revise/Block counts — the glance-altitude `cnt-strip`) and the **lifecycle state** chip (e.g. *In review*) at the far right; ② filename · **Created / Edited** date-time with a relative "ago". (The tally chips and the **Reviewers roster** coexist deliberately — the chips are the glance count, the roster the per-reviewer detail.) Expanding a card reveals the line-numbered plan beside a **nav rail**, then the shared footer.
- **The nav rail is a three-tab lens switch** — **Outline · Feedback · Authors** (the **V1** tab layout: Outline keeps its text label and takes the first half of the strip; Feedback + Authors are **icon + count** tabs splitting the second half equally). Selecting a tab is a **lens**: it repaints the rail's **card list**, the editor's **gutter badges**, and the docked **box** together. **Outline** (body caption **Table of contents**) leads with a **nested, level-generic table of contents** — parsed from every `##`–`####` heading, indented by heading level (so a future `#####` is a threshold bump, not a refactor) and each row anchored to the heading's **line index** (never its text, so repeated sub-headings can't collide) — each carrying a **worst-verdict dot** (revise/block only — an approved section reads as a **neutral** dot) + comment count, and it shows **no gutter badges**. Below the TOC sit **two rosters, Authors then Reviewers** (each a heading + count badge over identity-badge rows): the **Authors** roster is the document's distinct authors; the **Reviewers** roster is its distinct reviewers, each row trailing its **worst revise/block verdict** (block > revise) — or a lone **Approve** for a reviewer who left only approvals. The rosters encode the **two-level model**: a reviewer's verdict is a *response to* the document and is **deliberately distinct from the document's own lifecycle** (the header lifecycle badge and the footer **Approve** — the doc's own draft → in-review → approved arc). A level-L heading's section spans from its heading down to the next heading of level ≤ L, so selecting/hovering a `##` also covers its nested `###`/`####` children. **Feedback** (body caption **Feedback**) is one **card per response** — the reviewer's agent badge + a thumbs **agree** toggle, a color-coded **verdict badge** (Approve/Revise/Block) + a **section badge**, and a clamped comment summary — with verdict-coloured gutter badges. **Authors** surfaces authorship metadata as one **card per edit** — the agent's identity badge + a **section badge** + the edit **timestamp** — with neutral **users**-glyph gutter badges. Selecting an annotation lens (Feedback or Authors) moves **three indicators together**: the **card** fills (light teal), its **section text** highlights (teal, linking card ↔ text), and the **comment popout** opens. Closing the popout or deselecting clears all three, and opening a different comment switches all three — driven from either a Feedback card or the in-text rail-gutter badge. The rail **resizes with the text column** (it grows when the popout opens, so cards aren't clipped). The **comment popout itself is never crushed**: docked open it keeps its natural height — the editor body above yields and scrolls internally to make room — and past its own cap the popout's **body** scrolls rather than clipping (so the fit holds in both the Plans cards and the Documents tab).
- **Comments dock under the text.** Section feedback also surfaces as a **verdict badge in the rail gutter** at that section's line — rendered **only for revise/block**: an approved section shows **no gutter chip** (absence = approved, which frees rail space), while the section's comment *count* still stays coherent (it tallies every verdict, approves included). The rail badge is a **section anchor**: its count is **all** the comments on that section (every verdict), and clicking it (or any Feedback card for that section) opens **one comment popout** listing **all of them** — each row is **reviewer badge · verdict badge · thumbs agree toggle · timestamp** (the time right-aligned at the row's end), so the count always matches the contents; **each row is click-to-select**, feeding the [merged Export control](#sharing--exporting--the-merged-export-control) so a comment can be embedded/copied into a prompt for follow-up. The **Authors lens** reuses this section-anchored pattern with a **neutral gutter badge** (a users glyph + count, distinct from Feedback's verdict colour) that opens the docked **author box** listing **one entry per count** — each row uses the same layout grammar *minus the thumbs*: **agent badge · a Drafted / Edited / Revised action badge** (a neutral `dbadge` variant — an authorship action, not a verdict colour) **· timestamp** (right-aligned) over the edit's **summary** line; a document-wide edit (no section) reads as **Whole document**. Its rows are **click-to-select** into the same Export control. (Selecting a section moves the three indicators together: the matching Feedback card(s), the highlighted section text, and the popout.) You add your own with the **Comment** button — enabled once you select a line/section — which opens a **composer** in the same popout: your badge, a **Mark as** verdict dropdown (Approve/Revise/Block), a thumbs agree default, a note, and **Save**. A **Revise or Block verdict requires a non-empty comment** — **Save stays disabled** (with a "needs a comment" hint) until you type one, and the submit path enforces it even if the control is bypassed; **Approve is comment-optional**, and a comment-less approve surfaces only in the **Reviewers roster** (it needs no popout entry).
- **An "Editor" header over the content; hand-off/decision in the footer.** Plans (and Documents) carry an **"Editor" header** above the content field, mirroring the Compose Editor header: the **Editor** label — with a **mic** ghost button right after it (per-field voice dictation, red while recording) — and right-aligned **Copy · Edit · Comment** ghost icons (the same ghost icon-buttons Compose's copy/clear use). The card's **footer** then holds only the hand-off + decision controls — for a Plan, the **[merged Export control](#sharing--exporting--the-merged-export-control)** **leads**, then the single-agent **Review chip** (a reviewer badge select + an icon-only review button) — both left-aligned — then, right-justified, the decision trio: **Revise** (send the flagged sections back to the authoring agent) · **Reject** · **Approve**. (It wraps to a second row on a narrow column.)
- **Cross-linked with the Inbox.** A plan or document is reviewable here; its *approval* surfaces in the Team Feed's [Inbox](#the-inbox-tab). The Inbox's **Review** button jumps to the matching tab — **Plans** for a plan, **Documents** for a doc (routed by the card's id prefix) — expanding and briefly highlighting the matching card; the card's **Approve/Reject** is the same handoff in reverse.
- **Documents are the same reviewable-document component as Plans.** A project doc is now a full **plan-style card** — the same 2-row header (owner **agent badge** · title · right-aligned tally chips + **lifecycle** state / filename · dates; the steps chip is naturally absent since a doc has no checklist), the same **Outline · Feedback · Authors** nav rail over the line-numbered editor, the same **"Editor" header** (Editor label + per-field **mic** + right-aligned **Copy · Edit · Comment** ghosts, the Edit ghost toggling the rendered view ⇄ a raw-markdown textarea — **now the same working toggle on both plan and doc cards**), and the **full decision footer**: the **[merged Export control](#sharing--exporting--the-merged-export-control)** + single-agent **Review chip** lead (left), then the right-justified decision trio — **Revise** · **Reject** · **Approve** (collapsing to a static *Approved* stamp once approved) — exactly as Plans. A doc carries the **full lifecycle** (*draft · in review · approved*) and the whole review flow, so it can be **approved** just like a plan (an earlier model treated approval as Plans-only — that distinction is gone). The only intended divergences from a Plan: a doc has no checklist (so the *done/total steps* count is naturally absent), and the Inbox **Review** action routes to the Documents tab. **Assets keep the simplified card shape** (a Plans card *without* the owner/title/tally/steps rows): a basic **header** with the file **path** + **Created / Edited** dates, then the image **preview**, then the plain **[Export control](#sharing--exporting--the-merged-export-control) + Remove** footer — no Editor header (an image isn't text-editable), no Review chip / Revise; an Asset is whole-file, so only **Attach** is enabled on it.
- **Both the Plans and Documents tabs are a two-column layout** — a left **entry-nav** column beside the right card list. Each nav entry is a **mini-card**: a **per-file-type icon** · name (**· path for docs**, which need it to disambiguate same-named files) over its own **lifecycle badge** (plans list by title). Clicking a nav entry **opens/scrolls/flashes** its card, and an open card **highlights** its nav entry — the two stay in sync both ways. Plans lists **plans**; Documents lists **docs** and keeps the **Add** affordance; **Assets** reuses the same nav shape. Each **Documents/Assets** nav row carries a ghost **rename** (pencil) icon; the **Add** affordance at the foot of the Documents/Assets nav offers **Add file** (opens the explorer) or **Paste** (creates a new row **opened with its name in inline-rename** so you type it immediately — this is also where the Team Feed / History / editor **Export → "Export selected → file"** lands the selection, as a new pasted doc).
- **The Documents tab** folds every project doc into **one** tab (it replaces the old per-file Readme / Claude tabs). The nav **path is load-bearing** — it disambiguates same-named files (the project `CLAUDE.md` vs the user-level `~/.claude/CLAUDE.md`). Selecting a nav entry **opens that doc's card** — the **same line-numbered editor + rail** the Plans tab uses. Only **README**, the project **CLAUDE.md**, and the user-level **CLAUDE.md** are listed by default; the nav's **Add** affordance adds more.
- **The Assets tab** holds **reference images** as a **rail + preview**: a left list of **per-file-type icon + name**, with the selected asset's **preview** filling the right (Documents once shared this single-pane shape; it's now card-based, and only Assets keeps it). Assets is the **single source of truth** for media — an image added here, *or* attached while composing via the [Prompts paperclip](#prompts-right-bottom), both register in this one list.

### Settings (step-into view)

A top-level **Settings** surface for the whole workspace — the configuration that isn't about any one agent. It's opened from a **gear in the title bar** (right cluster, beside the WSL2 / tmux / Connected chips) and presents as a **step-into full-window view**: it replaces the 3-pane body in place, toggles in and out, and returns to the 3-pane on **Close** (or Esc). It is deliberately **not a fourth always-on column** and **not a floating window** — it steps into the same frame rather than overlaying it, in keeping with the dashboard's [bias toward on-screen surfaces over floating ones](#purpose--vision).

**Tabs are by subject, not by elevation.** Six subjects — **Projects · Setups · Usage · MCP · Plugins · Config** (Projects leads as the first, default-shown tab) — and where a subject spans config scopes, **scope is a secondary segment inside the tab** (never a tab of its own), so a single subject is never split across tabs:

| Tab | What it owns | Scope segment |
|-----|--------------|---------------|
| **Projects** | The **project registry + open/close flow** — the repo/folder a fleet works in; open, switch, and register projects, with **one project open at a time**. See [Projects](#the-projects-tab) below. | — |
| **Setups** | The full **Save / Load setup** flow (agents + links). | — |
| **Usage** | **Account** (email / org / plan, read-only from local creds), **Limits** (session / weekly % + resets, fetched live from the API — degrades gracefully when unreachable), and token consumption — the `/stats` · `/status` surface. **Usage only**; per-agent cost/dollar spend stays [out of scope](#purpose--vision). | — |
| **MCP** | The global server registry — enable/disable, connection & OAuth health, and the disabled-server **parking** state. | user / project |
| **Plugins** | Installed plugins + enabled state, and marketplaces. | user / project / local |
| **Config** | Default model, permission mode, sandbox, hooks, env, CLAUDE.md, plans. Each setting tagged **Live** (takes effect now) or **New session** (needs a restart). | global / project |

Cross-cutting rules:

- **Read-only vs editable, separated.** Within every tab, read-only status/health/usage is kept in its own band, visually distinct from editable config — so glancing at state never reads as a control.
- **Global edits are gated.** Changing global (`~/.claude`) config — which affects every project — is held behind an explicit **confirm**, with a standing warning on the Global scope.
- **It owns the global registry, not per-agent scope.** Enablement here is "is server/plugin X available at all," a different thing from "may agent Y use server X," which lives in the [Agent panel](#agent-left-pane) — **now surfaced there as the per-agent MCP servers / Plugins / Deny-rules scoping controls**. The two are deliberately not conflated.
- **The footer keeps a glanceable shortcut.** A token-usage summary lives in the [status footer](#layout--the-three-column-model) for quick access (it jumps straight into Usage); the panel holds the full detail. (The footer's former Save/Load shortcut was removed — the Setups tab is now the single home for that flow.) *(Voice dictation is no longer a single universal footer mic: each Editor — Compose, Plans, Documents — now carries its **own** mic ghost button bound to that field, enabled only while its field is editable + visible, red while recording.)*

#### The Projects tab

The **first** (default-shown) Settings tab. A **project** is the repo/folder a fleet of agents works in; the dashboard runs against **one project at a time**, and this tab is where you **open, switch, and register** projects. It reuses the Settings kit (`settings-note` · `set-sec` / `settings-section-header` · `settings-row` · `registry-row` · `connector-health-badge`) plus a small project-specific set. Top to bottom:

- **Active project** (the `active-project-card`) — pinned on top, shown only while a project is open: an "Open" [connector-health-badge](#component-system--names-status-and-variants), rows for Project / Repo path / Agents / Last opened, and a **Close** row whose action is a **Close project** button. When no project is open this card is replaced by the quiet `project-empty` placeholder (a dashed muted band, deliberately not a card).
- **Close semantics — two choices.** "Close project" reveals the `project-close-confirm` strip: **exactly two** actions, each with a one-line note — **Close** (`btn-main`) leaves the project's agents **running in tmux** (reopening the project **reattaches** them), and **Close & stop agents** (`btn-danger`) **ends every agent session in this project first**. It is **warning-toned** — an attention decision that sits between [Retire](#agent-left-pane)'s soft confirm and [Delete](#agent-left-pane)'s danger gate in severity. A **ghost-x** in its header and the **Esc** key cancel it (it never toggles — dismiss is explicit, the [confirm-gate](#component-system--names-status-and-variants) precedent).
- **Known projects** — the registry: one `registry-row` per project (name · path · agent count · last-opened) with an **Open** action. The open project's row takes the `.proj-open` **selection fill** and shows the "Open" badge; **every other row's Open is disabled** while a project is open (the one-at-a-time rule — close the current project first).
- **Register** — an **Open other folder…** action registers any folder as a new project; a freshly-registered row **flashes** (`.proj-flash`) once as it joins the list. *(In the real app this opens the OS folder picker.)*
- **Active-project chip.** The [title bar](#layout--the-three-column-model) carries a small `active-project-chip` naming the open project — **display + a shortcut affordance only** (clicking it jumps to Settings → Projects), consistent with the other title-bar chips; it reads "No project" when nothing is open.

## How it works — concepts that span panels

### Execution vocabulary — session, run, turn, step

A precise ladder for talking about *what an agent is doing*, so the [Run strip](#team-graph-middle-top), the Turns bar, [Lifecycle](#lifecycle--autonomy) limits, and the [Console](#agent-left-pane) all use the same words:

- **Session** — the agent's whole lifetime, from create/launch to retire/delete; the persistent Claude Code process behind the card.
- **Run** — one stretch of autonomous work **between your prompt inputs**: it begins when you send the agent a prompt and ends when it goes idle awaiting the next. A run is what the **Run strip** measures, and what **Retry** redoes from the top (so the strip's segments **reset**).
- **Turn** — one **model round** within a run (a single model request/response, including its tool calls). The **Turns** load bar counts these against the agent's **Max-turns** Lifecycle limit.
- **Step** — one item on the agent's **self-reported checklist** within a run — the granularity the **segmented** Run strip renders as *done ÷ total*. A step spans however many turns it takes; steps are the agent's own operational milestones, not model rounds.

*"**Exchange**" is deliberately **not** in this ladder — it's reserved for the agent-to-agent sense (the End-After exchange rule — docs/ARCHITECTURE.md §7.6): **one message each direction** across a [link](#linking--context-sharing) (a round-trip), the unit a link's **End After** limit counts. **Amended for one-way links:** on a link that only fires in one direction, **each fire counts as an exchange** — without this, End After could never trigger on a one-way link. Inter-agent traffic is measured in **Exchanges**; a single agent's own work is measured in **Run / Turn / Step**.*

### Agent identity & naming

Every agent has a stable identity made of a **role**, a per-role **number**, a short **human name**, a **color**, and an **icon** — e.g. *researcher · 01 · sandy* → `researcher-01-sandy`. Human names (sandy, kai, drew, rowan…) are used because they're easy to say and remember when you're talking about agents. Color and icon are chosen per agent (not fixed by role) and then appear *everywhere* that agent does, which is what makes the whole UI scannable.

Two **reserved pseudo-identities** sit beside the agent roster. **User** is you — the slate `--ag-user` tile. **System** is the app/runtime itself — a **gear glyph on a navy tile** (the one additive `--ag-system` token, kept in the `--foreground` **ink register** and deliberately **never a jewel colour**: System is the app's own ink, not an agent — the gear reads as *who*, not as the Settings control). System speaks only when the *platform* fails or acts: it fronts Inbox **[Error](#the-inbox-tab)** cards for **system-wide failures** — infrastructure (the tmux/WSL2 bridge or sidecar down), account-level (rate/usage caps, auth expiry), and shared services (a global MCP server failing) — one System card standing in for N identical per-agent cards, and it badges **Log** lines for system events. Like subagents, System is **filter-only, never addressable**: its row sits **second, after User**, in the feed From/To filter tree, and it is **excluded** from Compose To, Compose From, and the History From filter (addressability is encoded in the shared roster data). On a System Error card **Reply is disabled** — greyed, not removed (the Export-menu "listed but greyed" convention) — and since System has no graph card, there is no `node-inbox` envelope mirror.

### Linking & context-sharing

The defining capability: a persistent **link** that forwards context from one agent to another so they can collaborate (including back-and-forth "conversation" between two agents). You create a link from the [Team Graph](#team-graph-middle-top) and configure it in the **Link Config drawer** — a sheet that opens immediately to the **right of the Team Graph panel** (anchored to the left edge of the right pane, sliding out from behind the graph) so the whole Team Graph stays visible while you set the link up:

The drawer's fields run in a **fixed order**, top→bottom: **agent-pair + direction → Relationship → Trigger → End After → Save/Delete → the links list (Active / Expired sections)**. The **agent pair** is two **single-select agent dropdowns** (the same `source-dropdown` primitive as Compose's **From**, one per endpoint) flanking the direction cycler — the Team Graph selection **prepopulates** them, and either endpoint can be re-picked from inside the drawer.

- **Direction** — a 3-state toggle: A→B, B→A, or A↔B (both — the **default**). Its arrow glyphs are **Lucide icons** (`arrow-right` / `arrow-left` / `arrow-left-right`) at the standard UI-icon stroke, inked navy `--foreground` and flex-centred in the control — not unicode text glyphs. *(On the graph itself, links are intended to read as directed arrows — see [Team Graph](#team-graph-middle-top).)*
- **Relationship** (the one-relationship-per-link decision, **amended** — docs/ARCHITECTURE.md §7.6) — *what flows*; this **replaces the old Payload field** (what's sent is now **derived from the relationship**, not picked as a separate knob). It's a **single-select** button group (a two-option segmented control: **Direct messaging | Shared context**) — **one relationship per link**; wanting both behaviors = **two links**. *(This supersedes the earlier "a link can be both" multi-select: bundling two behaviors with different natural delivery models under one shared Trigger kept making "what does Trigger mean for a share?" ambiguous — one relationship per link dissolves that, and today's piggybacked-share behavior demotes from a hardcoded property to just the default Trigger value.)* Picking a relationship **re-defaults the Trigger** below to its natural delivery — **Direct messaging → Queue · Shared context → Piggyback** — and either can still pick any trigger:
  | Relationship | Meaning |
  |--------------|---------|
  | **Direct messaging** | A reply-to conversation between the two agents (the direct-messaging link relationship, docs/ARCHITECTURE.md §7.6). A **fire = the completion of a reply** — the source finishing the turn that answers the peer's inbound, routed back to that inbound's sender — **not** a blind broadcast on every idle; strict **one inbound in flight** per agent keeps the pairing unambiguous. No extra config. |
  | **Shared context** | Passive awareness — the target sees the source's selected context, **delivered per the link's Trigger**. The default is **Piggyback** (the payload rides the target's next inbound message, never initiating a turn), but SC keeps the **full** Trigger menu — no gating. The honest tradeoff is written into the model: an **actively-delivered** share (Now / Inject / Next / Queue) is a real message that **costs the target a turn to ingest**, which is why Piggyback stays the SC default. Reveals a nested **content-type multi-select** (the Messages **Content** taxonomy — **Text** / Thoughts / Read / Write / Bash / Diffs / Meta; the two surfaces deliberately share one taxonomy, and **Text** — the agent's actual reply, default on — is what makes a share carry what the agent *said*, not just its tool trail) filtering *what* is shared, plus a **"share all prior context" backfill** switch (a one-time receiver backfill, **default off**) — both scoped to SC links. |
  *(The removed Message / Transcript / Manual payload options fold in: Message/Transcript → the two relationship types; Manual → just sending a prompt by hand. One-shot context sharing is deliberately **not** a link preset — no "Once" — the operator covers that via Messages → Embed/Attach.)*
- **Trigger** — *when/how* the message is delivered. A **dropdown** (a menu selector — six options each carrying a one-line description exceed an inline segment, per the [inline-vs-menu rule](#design-system)); its menu entries are styled like the Send split-button's timing-menu items (**bold label + one-line description**, check on the selected row). The default is **per-relationship** — **Direct messaging → Queue · Shared context → Piggyback** — and picking a relationship re-defaults it:
  | Trigger | Behavior |
  |---------|----------|
  | **Now** | Interrupt the target and deliver immediately. |
  | **Inject** | Deliver to a *running* target without stopping it — a one-shot message applied at the next safe boundary (between tool calls). |
  | **Next** | Wait for the target's current response to finish, then deliver — ahead of its queue. |
  | **Queue** | Wait for the current response *and* let the existing queue drain first (the polite default; the **Direct-messaging** default). |
  | **Hold** | Stage the message for your manual approval before it's sent. |
  | **Piggyback** | **Never initiates** — the payload rides the **next message delivered to the target from any source**. **Link-only** (like Hold, it isn't offered on the Send timing chip), and available on any link type (a piggybacked DM reads as a deferred reply — coherent, if niche). The **Shared-context** default. |
- **End After** — optional safety limits so bidirectional links can't run away: **Exchanges / Tokens** as independent toggles, each with its own value (**default 25 exchanges**). None on = no limit; if both are on, the link ends at the first one reached. An **Exchange = one message each direction** (a round-trip) — explicitly **not** an agent's internal turns/steps, which are [Lifecycle](#lifecycle--autonomy)'s scope. **Amended:** on a **one-way** link, **each fire counts as an exchange** — without this, End After would be dead on every one-way link.
- **The links list** (the link-tracking decision, **amended** — docs/ARCHITECTURE.md §7.6) — pinned to the **bottom** of the drawer (below Save/Delete), split into two **collapsible sections**: **Active Links** and **Expired Links** (a link expires when an End After limit is reached or it's ended manually). The **section header carries the state** — expired rows get **no gray-out** (the Inbox typed-section precedent: the header carries the type/state, the rows stay clean and legible — which also keeps the future master/detail "click an expired row to load + **re-arm** it" extension open). Within a section, links **group by agent**: because a link joins two agents, **each link is double-listed under both agents' groups**, entries **sorted peer-adjacent** (rows sharing the same other agent sit together, in roster order). Each group header is the agent's identity badge carrying a **corner count badge** — the **overlay badge family** (a small count square overhanging the badge's top-right corner; **teal fill >0 / muted at none**, the node-inbox envelope convention). Each entry shows a **direction arrow relative to that group's agent** — the **Lucide** icons **`arrow-right`** (to) / **`arrow-left`** (from) / **`arrow-left-right`** (both), navy-inked and flex-centred, reusing the same arrow vocabulary as the pair row — then the **other agent** + the link's **full relationship label** ("Direct messaging" / "Shared context" — a link has exactly one, so one label says it all). *(Natural extension, not required for v1: clicking a list entry loads it into the fields above to edit — the master/detail pattern.)* On-graph link **edges** and per-card link **badges** stay deferred (`link-edges`, `planned`); basic link tracking lives entirely in this panel for now.

**How messages read (sender + trigger).** A message an agent receives carries lightweight metadata — who sent it and which trigger delivered it — embedded in the message for the *receiving agent's* benefit. The dashboard hides those tags and renders the human-facing version: a color-coded sender heading, a small trigger badge, and the body text. One message, two presentations. User-sent prompts go through as plain text.

#### Sharing & exporting — the merged Export control

Export and content-sharing live in **one** dropdown — the **Export control** — reused on the **Library** footers (Plans / Documents / Assets), the **Team Feed** footer, and the **History** footer. It routes content to the [Prompts Editor](#prompts-right-bottom) (Embed/Attach) and to the clipboard / [Library → Documents](#library-middle-bottom) (Copy/Export), replacing the older *separate* Output Export and Embed/Attach dropdowns. The trigger is a single download-icon dropdown-with-chevron whose menu has **two headed sections**:

- **Copy & Export** — **Copy selected** (to the clipboard) · **Export selected → file** (a new Documents doc from the selection; you stay put).
- **Add to prompt** — **Embed in prompt** · **Attach as file**.

*(Cut was dropped in the consolidation. The word **Link** stays reserved for inter-agent [links](#linking--context-sharing); content-sharing is **Embed** or **Attach**.)*

- **Per-item selection-gating** via the shared [select-to-act model](#library-middle-bottom): the trigger enables when **any** item does, and each item gates on the selection — **Embed in prompt** is enabled for **any** selection (whole card/doc or a sub-block/line/section); **Attach as file** is **whole-only** (a whole card/doc/title selection or an Asset); **Copy** / **Export → file** act on **whole feed/History cards** or on the **editor line/section/doc** selection. Disabled items stay listed but greyed, so the menu only ever fires the applicable action. **Assets are Attach-only** (an image isn't copyable-as-text, embeddable as a quote, or a `.md`).
- **Embed** drops a **frozen `embed` block** into the Editor — the selected text (a doc section, a feed block run, *or* a whole card/doc) under a "from \<source\>" header with click-to-source (the [inserted-block primitive](#the-templates-flow)).
- **Attach** **saves the selection as a real file**, drops a **path chip** into the Editor's [attachment strip](#prompts-right-bottom) (a path reference + a hardcoded "read this"), then **switches to Compose and reveals** (flashes) the chip. Content that has its own file (an Asset, an existing Document, a Plan) is chipped at its own path; **materialized** content (a feed message, a History prompt) is first **written to a new Library → Documents doc** (the same `createDoc` the Add-document Paste uses) — distinct from **Export → file**, which creates that doc and leaves you *in* Documents.
- **Citations** — from an attachment chip's link icon, with the cursor in the prose, you insert an inline **`citation` pill** pointing at that attachment (same payload as Attach). Deleting a citation does **not** delete its attachment; deleting an attachment **cascades** to remove its citations; you can only cite something already attached.

**The Review chip.** On the Plans and Documents footers, **Review** is a **single-agent reviewer select** — the trigger shows the chosen reviewer's full agent badge, paired with an **icon-only review button** (no text). Picking *who* reviews is the only choice; the icon sends. Its reviewer **picker opens *upward*** as a raised overlay above the footer (matching the Export menu), so the editor card's bounds never clip it — the shared rule for the Plans/Documents footer popovers (reviewer picker, merged Export, verdict). *(Groundwork for the single-reviewer-agent model; the deeper review/verdict workflow stays **deferred**.)*

### The shared scratchpad

A single shared markdown document agents post to as a **living** workspace — each post attributed to an agent and timestamped, viewable live in the Team Feed's [Scratch tab](#team-feed-right-top) and writable from [Prompts → Target → Scratch](#prompts-right-bottom). The attribution/timestamps exist because it's a running log of who-said-what; a clean, stateless document can be produced from it at the end. *(Posting in, reading out is the whole interaction for now; richer per-post selecting/commenting is **deferred**.)*

### Rewind & Handoff

Both act on the **Timeline** in the Agent → Details tab — the list of points (messages sent to the model) in the focused agent's run:

- **Rewind** — roll *this* agent back to a chosen point and resume from there.
- **Handoff** — branch from a chosen point into a *new* agent: it opens the Create tab with the source agent's settings prepopulated (editable), so you can carry the work onward without disturbing the original. *(Richer handoff artifacts/summaries are **deferred**.)*

### Lifecycle & autonomy

The project leans toward letting an agent "go as far as it can" on a task rather than babysitting it — so agents carry **auto-stop limits** (per-agent **Max turns** and **Context %**) that end a run safely when hit. These are deliberately a **different scope** from a link's [End After](#linking--context-sharing) limits: **Lifecycle bounds a single agent's run; End After bounds an inter-agent exchange.** Keep the two distinct.

### The Templates flow

The Prompts **Editor** holds **inserted blocks** alongside free prose — one reusable **block primitive** with three variants, stacked vertically as delimited blocks (the boundary uses the **card-outline** colour, not the tan divider):

- **`embed`** — *frozen* quoted content (a doc section, a feed message or block): a muted, small-font body under a static **"from \<source\>"** header, with **click-to-source**, a **remove ×**, and **clamp + expand** when long. Not editable.
- **`template`** — the same shell but *interactive*: **selectable**, and the **active** one (the selected block) is what the Templates fill input drives.
- **`citation`** — an inline pill in the prose pointing at an [attachment](#prompts-right-bottom) chip (see [Linking & context-sharing](#linking--context-sharing)).

Templates turn reusable prompt files into fill-in-the-blank forms *(Templates was a separate tab, then inlined into Compose, and now lives **below** the Editor)*:

- A single-select **dropdown** (default **None** + the saved templates) sits in the **Templates** section below the Editor, with the **fill input** below it (greyed until a template block is active).
- Picking a template **inserts a `template` block** into the Editor at the cursor (or appends if the box isn't focused), *without wiping* existing text; the template's placeholders come in as **clickable pills** (filled pills show their value; unfilled show the bare tag, e.g. `{severity_threshold}`).
- **Selecting** a template block (clicking it, or a pill inside it) makes it the **active** one; click a pill → the **fill input** activates (a single-line field that auto-grows) with **Reset** and **Apply** as trailing icon buttons *outside* the field; Apply pushes the value into the pill, which **stays editable/re-selectable** (Reset clears it back to the bare tag). *(The input isn't tinted to the pill's color — the placeholder simply names the active tag.)*
- Filled vs. unfilled pills are visually distinct, and the active pill gets a selected style, so it's clear what's left to fill before you Send. **None** deselects the active template block + re-greys the input; the Editor stays **fully editable** throughout.

## Design system

> **Where the values live.** Every raw value — colour, **type (families + weights)**, **spacing**, **sizing**, **radius**, **border-width**, shadow — lives once in **[`tokens.css`](tokens.css)**, the single source of truth, exposed as CSS custom properties (`--main`, `--border`, `--border-width` / `--divider-width` / `--ring-width`, `--radius-base` / `--radius-sm` / `--radius-xs`, the `--size-*` and `--space-*` scales, `--font-sans` / `--font-mono`, the 25 `--ag-*` agent colours, …). Both [`mockup.html`](mockup.html) and the shared [`styles.css`](styles.css) reference the tokens via `var()` — never a hardcoded literal (no inline `2px`/`3px` borders, badge radii, font-family strings, or recurring padding/gap/size values remain).
>
> **Token shape (the locked rule).** Tokens are **primitive** — used directly, by value, with *no* semantic layer — for **colour, radius, border-width, and sizing**. The one place **semantic alias** tokens are allowed is **padding / gaps / spacing**; the mockup currently uses the primitive `--space-*` scale directly. What stays **out** of `tokens.css`: per-instance bindings (the per-card `--nc` that binds one `--ag-*` colour to a single agent) and data/runtime values (a progress bar's inline `width:68%`, and the resizable panes' **default divider positions** — each panel's inline `flex:` ratio) — those reflect state, not design, and stay inline. *(The panes' **min/max size constraints** are a different matter — a stable design decision, not runtime state — so the fixed dev-window size, the pinned column widths, and each vertical pane's height range DO live in `tokens.css` as the **Layout** set: `--win-*` / `--col-*` / `--pane-*`; consumed via `var()` on `.app` and the panel inline styles, with only the `flex:` ratio left inline.)* *(The retired `--req-*` Inbox request-type ramp was **removed** in the token pass — unreferenced after the Inbox went to typed sections.)*
>
> This section owns the **rules** — which token to reach for, and why — and names tokens rather than repeating their hexes. To change a value, edit `tokens.css`; to change a *rule*, edit here.
>
> **Token-check (run after any `design/` change).** `grep -nE 'border-radius:[0-9]' design/styles.css design/mockup.html` — every hit must be an **intentional exception** (a `0` corner, a `50%` circle, or a documented one-off like the `.swh` 11px pill); anything matching an existing token must reference it via `var()`. This is the fast catch for a hardcoded radius that should have been a token.
>
> **Variant-check (run alongside it).** `grep -hoE 'data-comp="[^"]+"( data-variants="[^"]+")?' design/mockup.html design/behavior.js | sort | uniq -c | sort -rn` — one row per tagged component root (static markup + template-literal builder emissions), with its `data-variants` declaration paired beside it where one exists. Review the components you touched: **if you built or reshaped a component that has appearance states the mockup doesn't show, declare them on the element before you're done** (`data-variants` immediately after `data-comp` — the adjacency is what lets this one-liner pair them); see [Variant declarations](#variant-declarations-data-variants). *Known blind spots:* slugs emitted via `setAttribute` or string concatenation don't match the pattern — today `toast`, `hover-card`, `inline-confirm` (both the History-delete and footer-confirm builders in `behavior.js` tag it via `setAttribute`), and the six inbox-card types (whose concatenated `_inboxComp` emission shows up as one junk row — though that row does pair the six cards' shared `data-variants` declaration). The **label overlay** (`Ctrl+L`) reads the live DOM, so its enumeration is the complete one; this grep is the fast static check, and the **Variant-lint** below covers the blind spots for declared pairs.
>
> **Variant-lint (the enforcement tier — run before calling variant work done).** `node dev/tools/variant-lint/variant-lint.mjs` — **fails** on malformed `data-variants` (an empty value, duplicate or non-kebab state names), a declaration not immediately after its `data-comp`, declared **interaction states** (forbidden by the [state ladder](#variant-declarations-data-variants)), and two diverging declarations for one slug — and it boots the **[states page](states.html)** headless to report every declared state the page cannot render mechanically (the honest-gap list; `--strict` fails on those too, `--static` skips the browser and runs the source checks alone). It parses the `setAttribute` emission form and reads the live-DOM-derived states-page report, so the one-liner's blind spots don't apply to declared pairs. Its own limits are documented in the script header — chiefly that **it cannot see states nobody declared** (the per-touch rule and review are the only guard there), and that the name `active` can't be interaction-linted (the run-state sense is legal).

The dashboard speaks the **[neobrutalism.dev](https://www.neobrutalism.dev)** visual language — thick **2px navy borders** (`--border`), **hard offset shadows** with no blur (`--shadow` / `--shadow-sm`) on raised/interactive elements, **flat fills**, a uniform **tight radius** (`--radius-base`), and the **Archivo** type family (heading 800 / body 500) with **JetBrains Mono** for metrics. The palette keeps the **"Happy Hues 17"** core (cream / navy / pink) and puts that palette's teal to work as a secondary. Token names mirror neobrutalism's Tailwind theme (`bg-main`, `bg-secondary-background`, `border-border`, `shadow-shadow`, `rounded-base`, `font-heading` / `font-base`) so the static mockup ports ~1:1 to React/shadcn.

**Accents — one emphasis ladder** (grounded in Material 3 color roles + Carbon button hierarchy): **pink = primary · teal = secondary · cream = low-emphasis · red = danger.** Read any button by asking *what does clicking it do to this content?* — **pink** commits it *here* (one per surface), **teal** sends it *elsewhere* (hand-off / outbound to another agent or surface), **cream** is a quiet local utility that acts in place, **red** destroys, and **white is not a button** (reserved for fillable inputs + selector triggers). The one nuance is that teal carries two jobs of the same emphasis tier: a **hand-off action** *and* anything **selected/active** (segmented values, toggles, the selection fill). They rarely sit side by side — state lives in the config panel, hand-off actions in footers — so teal carrying both is correct, not muddy.

Form **inputs** and **selector-field triggers** (Color, Icon, Role, Source, Skills, Tools) stay white (`--secondary-background`) — fillable surfaces read distinct from clickable low-emphasis buttons; the accent rides the *value/selected option*, not the field chrome.

**Selectors — inline vs. menu.** *Inline* selectors (all options visible — segmented controls, tabs, toggles) show the chosen option as a **teal fill** (`--secondary`) in place. *Menu* selectors (options hidden behind a trigger — Color/Source/Skills/Tools, **Response**, and the link **Trigger** dropdown) keep a **neutral** trigger; teal rides the selected option(s) inside the open menu plus a small **teal count badge** for multi-selects. The dividing line is option count × description weight: the link Trigger left the inline segment for a menu when it reached **six options each carrying a one-line description**.

**Split buttons** read **light chip + full action**: the dropdown chip is a *lighter tint* of the action's accent (it recedes; the action leads). Send = full-pink action (`--main`) + light-pink chip (`--main-dim`); Revise = full-teal action (`--secondary`) + light-teal chip (`--secondary-dim`). The chip is a *parameter of that action*, so it stays in the action's hue family rather than going teal.

**Inbox — typed sections with an escalating heading ramp.** The Inbox groups its cards into **typed sections** (**Error · Warning · Permission · Plan · Decision · Response**, ordered most-to-least urgent — **Response**, the one non-blocking, non-request section, sits last); the section *header* carries the type, so cards drop the per-card type badge. The section **heading label + its matching count badge** carry an **escalating severity ramp** — **Error** (`--danger`, red) → **Warning** (`--warning`, amber) → **Permission** (`--inbox-permission`, blackish-orange) — while **Plan + Decision + Response** stay the neutral `--muted` tone. Only **Permission** carries a dedicated token (`--inbox-permission` `#a9710f`); **Warning reuses `--warning`** and **Error reuses `--danger`**, so no new signal colours were minted. **Error stays the single alarm**: it's the only section whose cards also carry a **danger left-edge** (the rest are heading-coloured only). This replaces the old per-card reddish→copper request-type ramp — the `--req-*` tokens were **removed** from `tokens.css` (retired and unreferenced), and severity is ranked by section heading now, not a per-card badge. `--warning` doubles as the **pending** signal elsewhere (it tints the graph-card **status badge**), which is consistent — both are "attention, not an error". The Team Feed's Inbox tab carries a **teal count badge** (`--secondary`, the fleet total) matching the Library → Plans tab count — the teal **tab count badge** family, distinct from the per-section heading badge.

**Status:** shown per graph card as a rectangular **text badge** in-flow at the right end of the card's top meta strip (a button that jumps to Inbox / History / Compose) — active `--success` · idle a solid `--muted` (slate-gray) fill with **white text** · pending → the warning tone above · **error a solid `--danger` fill with white text** (the 4th run state — a failed run; jumps to that agent's Inbox **Error** card) — **all four** use the same solid-fill + white-text treatment so they read as one badge family (the lighter `--ag-user` / `--muted-2` tone is reserved for the User row and the footer idle dot). The History/Plan **`db-*` status badges** mirror this: `db-error` is likewise a **solid `--danger` fill + white text** (not a soft tint), matching the Team Feed Messages **Error** status badge. The **context bar** on graph cards and in the Agent panel is colored by **health** (`--success` → `--warning` → `--danger`), not by the agent's identity color — matching the Timeline's context-% coloring.

**Agent identity palette** — **25** colors assigned per agent (tokens `--ag-crimson` … `--ag-rose` in `tokens.css`) and kept in a different *register* from the UI accents above (deeper and more saturated than the light pink/teal/status tints) so identity never collides with meaning. The two reserved pseudo-identities deliberately sit **outside** the jewel ring: `--ag-user` is the slate User tone, and `--ag-system` (additive) is **navy — the `--foreground` ink register** — because [System](#agent-identity--naming) is the app, not an agent. It's the **"Jewel" family**: one **OKLCH** recipe — a fixed deep lightness/chroma (`oklch 0.52 0.15`, gamut-clamped per hue) spread around the wheel in spectral **ROYGBIV** order, so every agent reads as equally weighted and none as a UI signal. The first **16** are spaced ~evenly; the **+9** (added for the agent-identity expansion, docs/ARCHITECTURE.md §7.5) **densify the widest gaps** to a **25**-colour ring rather than re-spacing the originals (additive — no existing token renamed). Past ~16 colours the **icon becomes the primary disambiguator** and colour a soft "family" signal, so assignment is round-robin (`color = n mod 25`, `icon = n mod 50` — every (colour, icon) pair unique for the first 50 agents, reused beyond). Defined and listed (and shown in the mockup's picker) in spectral order: crimson · scarlet · vermilion · amber · gold · topaz · citron · chartreuse · lime · peridot · fern · emerald · teal · turquoise · cyan · cerulean · azure · cobalt · sapphire · indigo · violet · orchid · fuchsia · magenta · rose. UI (non-agent) icons are **Lucide** (`assets/icons/ui/`), drawn at ~2.25px stroke to suit the heavy borders.

**Badge/chip size standard — S/M/L (landed 2026-07-05).** Every badge/chip family sits on one of **three fixed size tiers**, minted in [`tokens.css`](tokens.css) as the `--badge-{s,m,l}-{h,fs,px}` trios (height · font-size · horizontal padding): **S** (12px) for pure **numeric count squares and micro pills**, **M** (18px) for **small id cards and the semantic label badges** (status / lifecycle / verdict / direction / type, plus the icon+count tally chips), **L** (22px — ratified over 20 by the 2026-07-05 headed fit test) for the **generic and Settings chip/pill cluster**. Exempt as not-badges: unboxed icon+text readouts (the node settings chips, chromeless count texts) and the inline editor pills.

## Component system — names, status, and variants

The component vocabulary is governed in four pieces — the first three carried **on the elements themselves**: a **canonical name** on every instance (`data-comp`), a **variant declaration** for the appearance states an instance doesn't show (`data-variants`), a **status marker** for dormant or unresolved UI (`data-status`), plus a small **Open Questions register** (below, in this doc) for design calls deliberately left open. With the [token rules](#design-system) above, these are the rules the **five** design-source files follow: **`tokens.css`** (values) · **`styles.css`** (shared component CSS) · **`behavior.js`** (shared component **behavior** — the interaction logic) · **`mockup.html`** (the app surface + the on-element tags) · this doc (the rules).

### The catalog is retired — the mockup is the inventory

The interactive component catalog (`gallery.html`) and the per-component registry list this section used to carry were **retired on 2026-07-05**: both were hand-maintained parallel inventories of what the mockup already contains, and both had drifted. The gallery's final state is preserved as a **frozen, self-contained snapshot** at [`archive/design/design-v12p7-gallery-retirement/gallery.html`](../archive/design/design-v12p7-gallery-retirement/gallery.html) — it carries its own copies of the shared design files with the CDN scripts vendored, so it renders identically offline forever, and it is **never edited**. That snapshot stays the frozen visual record of every component and variant **as of the retirement date**; the living render of today's declared states is the **[states page](states.html)**.

What replaces them is **on-element tagging: the mockup plus its `data-comp` tags IS the component inventory.** There is no separate registry to maintain. To enumerate the components, toggle the **label overlay** (`Ctrl+L` on the mockup) — it reads the live DOM, so it sees every tagged component, however its slug is emitted — or run the **Variant-check** one-liner in [Design system](#design-system) for a fast static listing (it has a few named blind spots; see the check's note) — and open the **[states page](states.html)** to see every declared `data-variants` state rendered side by side, generated entirely from the live tags at load. *(The states page was deliberately built ahead of the second design sprint, bundled with the 2026-07-05 conversion sweep — sprint 2's badge rework will churn some declarations, each redo a one-attribute edit.)*

**Dev-only view tooling (outside the design system).** [`mockup-toolkit.js`](mockup-toolkit.js) (the `Ctrl+G` measuring/annotation overlay), [`label-overlay.js`](label-overlay.js) (the `Ctrl+L` component-label overlay), and [`states.html`](states.html) (the auto-generated **component-states page** — it boots the real mockup in an off-screen iframe, walks the live DOM for `data-comp`/`data-variants` plus a `behavior.js` source scan for declarations whose builders don't run at boot, and renders every declared state beside its default with the exact mapping captioned; states it cannot render mechanically fail loudly as red placeholders, and that gap list feeds the Variant-lint via `window.__statesReport`; serve `design/` over localhost — ui-verify — to view it, since the iframe needs same-origin access) are development tools that *read* the mockup: they are **not** part of the five-file design system, carry no `data-comp` tags of their own, and are **exempt from the propagation rule** — a change to them owes no other file (linking the real `tokens.css`/`styles.css` to render specimens faithfully is expected and fine). They store nothing per component (the label overlay and the states page recompute from the live DOM on every draw/load), so they cannot go stale: renames, new declarations, and layout changes flow through automatically.

### Canonical component names (`data-comp`)

Every component instance carries a single **`data-comp`** attribute on its **root element** — the canonical name as a **kebab-case slug** (`status-badge`, `agent-node-card`, `export-control`). Sub-parts are *not* tagged; they're identified by their existing classes (a node card carries `data-comp`; its `node-bars` sub-part does not — `node-chip` is the deliberate boundary case: **one** canonical chip on node-1 carries the tag so the slug stays enumerable, and its sibling chips stay untagged) — but a composite-internal primitive that carries its own root class + an independent state set (`node-subagents`, `markdown-row`, `message-rail-row`, `msg-block`, …) IS tagged as a component in its own right. The slug is the one canonical label — single-sourced on the element; where a doc needs a readable name it spells the same slug in words rather than tracking a second string. Many components are JavaScript-generated (the `.db-*` lifecycle family, the verdict badge, the count chips, the identity badges, all six inbox card types), so the slug is **emitted from inside the builder functions** — every rendered instance carries it, not just the static markup. New components are slugified with the same kebab rule and tagged at creation; **there is no list to append to** (the retired registry is not to be recreated — enumerate via the Variant-check line or the label overlay).

**Kept slugs with no live mockup root.** Three slugs have no rendered element in [`mockup.html`](mockup.html) today: `comment-split` (the `.cmt-ctl` split-button variant, CSS-only), `verdict-chip` (the `.vchip`/`.vtally` tally chips — though the family's `.rd` gutter-dot variant, including the neutral Authors-lens `.rd--author`, IS live, built by `railBadge`/`authorBadge` in `behavior.js`), and `verdict-popover` (`.vpop`, kept defensively in `styles.css` and included in `behavior.js`'s `closeAllPopups` query). At the gallery retirement their CSS/JS was deliberately **kept, not deleted** — none is provably unreferenced — and their rendered look is preserved in the [archive snapshot](../archive/design/design-v12p7-gallery-retirement/gallery.html). Don't expect to find these three tagged in `mockup.html`; if one gains a live instance later, tag it then.

**Intentionally untagged — excluded scaffolding** (pure background scaffolding / layout hosts; they get **no** `data-comp` tag, only this line):

- **Resizable splitter** — `.rz-group` / `.rz-panel` / `.rz-handle` (drag-resize splitters + the cursor-following size readout).
- **Generic panel shell** — `.pcard-head` / `.pcard-body` / `.pcard-foot` (the panel chrome the components sit inside).

### Variant declarations (`data-variants`)

**Variant declarations are a requirement, not a suggestion.** When you build or reshape a component that has appearance states the mockup does not currently show — a verdict badge that can be Approve, Revise, or Block but only ever renders one; a disabled, empty, or error look; a status ramp — you declare those states in a **`data-variants`** attribute on the **same root element** that carries `data-comp` (placed **immediately after** `data-comp`, so the Variant-check one-liner in [Design system](#design-system) pairs them), as **space-separated state names**, **emitted from the builder** where the component is JavaScript-generated. One attribute, on the element you are already editing, no second file. The **[states page](states.html)** renders every declared variant side by side, and the **Variant-lint** (both in [Design system](#design-system) / [dev tooling](#the-catalog-is-retired--the-mockup-is-the-inventory)) makes malformed or doctrine-violating declarations fail loudly; the [archive snapshot](../archive/design/design-v12p7-gallery-retirement/gallery.html) stays the visual record as of the retirement date. **Hidden is judged per slug, against the booted mockup:** a state counts as *shown* if any tagged instance of the slug carries it in the live DOM after `boot()` (instances inside non-active tabs included — DOM presence is the criterion, not static markup alone); only states hidden at the slug level are declared, on the first/most canonical static instance, or emitted from the builder for JS-generated slugs.

**What counts as a variant** (the conceptual state ladder): **default** → **hover** → **focus / focus-within** → **active / pressed** → **on / selected / checked** → **open / expanded** → **disabled / gated / off** → **status variants** (the semantic ramp a component encodes — run-state `active · idle · pending · error`, verdict `approve · revise · block`, or health `success · warning · danger`) → **empty / placeholder** → **recording / live / flash / loading** (transient feedback — *loading* is the in-flight data-pull state the **loading-strip** primitive renders, e.g. the context accordion's `.is-loading`). The **interaction** states (hover · focus · active · open/expanded · selected) are reachable by interacting with the live component and are **never declared**; `data-variants` is for the rest — the status ramps, emphasis tiers, and colour sets, and the **non-interaction data-states** (disabled · empty · error · transient) — whenever the mockup doesn't already show them.

**Do not create prose inventories.** Variants are defined on the element (plus at most a one-line code comment). Do not write per-component variant lists into this doc or any other — a hand-maintained parallel inventory is exactly the artifact that just failed and got retired.

**The remaining backfill is scheduled work, not ambient duty.** The **variant conversion sweep** landed 2026-07-05 — every pre-existing component with hidden appearance states now carries its `data-variants` declaration (the sweep's per-slug judgments are logged in DEVLOG). The **token-hygiene audit's conversion pass also landed 2026-07-05** (the token run: every hardcoded design value with an existing-token match converted across `styles.css` / `mockup.html` / `behavior.js`, the badge/chip **S/M/L size standard** minted, the component inventory re-trued, and the proven-dead CSS removed — per-decision record in DEVLOG). What remains is **named future work, still not ambient duty**: **colour-token minting** for the no-match colours the run deliberately kept as flagged literals, the **palette revisit** that would follow it, a **semantic alias layer** (deliberately not built — tokens stay primitive per the locked rule above), and the **badge/chip visual-consolidation pass** ([ND] #5 territory). Do not attempt any of these ambiently while doing unrelated work — half-converted-everywhere is worse than scheduled-and-done. The per-touch duties are unchanged: declare on the components you *touch* (the sweep's backfill doesn't retire that rule), and **when you pass a hardcoded value that should be a token, flag it — or fix it if you're confident — never silently ignore it**; deferring the *audit* is not license to walk past what's in front of you.

### Status markers (`data-status`)

Dormant-but-shipped UI and unresolved design questions are marked on the element itself with a **`data-status`** attribute, so they show up both as a tag on the thing and (via the register below) as a scannable list. Live, wired components carry no marker.

- **`data-status="planned"`** — shipped code/markup that doesn't do its real job yet (a `toast()` or counter stands in). The marker is the attribute alone (optionally a short plain-language code comment); it does **not** cross-reference any external backlog. Current `planned` items: the **link edges** (`drawEdges` / `LINKS` ship but draw nothing — the edge-layer host was removed), the **Link Agents** drawer **Save / Delete** (toast + counter only), the **MCP / plugin enable switches** (scripted demo), and **review / citation / attachment routing** (`sendReview`, `gotoCitation`, `composeAttach`, `openAttachment` — toast or navigation only). *(The Console decision — docs/ARCHITECTURE.md §7.13 — flipped the **Console run bar** planned→built: the Console surface — feed · catalog · run bar — is design-final, so its `data-status="planned"` was removed; `runConsoleCmd` remains a static-mockup demo and the live raw-terminal feed + slash-command route are **backend** wiring tracked in the backend stream, not a design status.)*
- **`data-status="undecided"`** — an element tied to an open design question (see the register). Resolving it means deleting the register line **and** removing the marker; nothing forces a resolution to ship.

### Behavior: wired vs planned

The mockup is heavily wired, not a static prototype — `boot()` renders every list from data arrays (including every **agent-selector roster** — the feed filter tree, Prompt To, and the From lists all mount from the one shared `AG` roster via `data-agroster`, with the Team Graph cards as ground truth) and wires the real interaction model: **tab switching**, the **column-tie selection** (clicking a Team Graph card repaints the Agent panel), **select-to-act**, **dropdown/popover** open-close, **accordions**, **empty-state gating**, **cross-panel hand-off**, **live form state**, **data-driven rendering**, **resize/layout**, and transient **`toast`** feedback. The surface that is **not** wired carries `data-status="planned"` (the list above) so dormant code isn't read as live behaviour; `toast()` is the line between "wired" and "acknowledged but mock."

### Open Questions register

Design calls deliberately left open. Each has a stable id, a one-line question, and `status: undecided`; where it attaches to a concrete element, that element carries `data-status="undecided"`. Resolving an item = delete its line here and remove the marker.

- **OQ-2 — Inbox section: fold vs. catalog — RESOLVED (folds).** The Inbox's typed sections (the `inbox-section` wrapper) are **collapsible accordions, expanded by default** (every section open for scanning): a neutral full-width `--surface-3` band with a **leading** chevron — deliberately distinct from the `.fcard` accordion's bordered right-side chevbtn — carrying the section label + its ramped count, and a `--rule` hairline into the cards; clicking the band collapses/expands it. The wrapper no longer carries `data-status="undecided"`. *(status: resolved)*

*(OQ-1 — the subagent-badge click behaviour — was **resolved by the subagent model** and removed from this register: clicking a subagent badge now focuses the parent agent, opens its Details → Subagents accordion to that row, and scopes the Team Feed to that subagent. The subagent badge no longer carries `data-status="undecided"`.)*
