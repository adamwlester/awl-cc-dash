# AWL Multi-Agent Dashboard — Design Reference

A single-window desktop app for running and coordinating many Claude Code agents at once. This document is the ground-truth reference for the dashboard's **UI/UX intent** — what each part is *for* and *why it behaves the way it does*.

It speaks the **[neobrutalism.dev](https://www.neobrutalism.dev)** visual language — token names + utility classes mirror that library's Tailwind theme so the static mockup ports ~1:1 to a React/shadcn build. This README avoids pixels/hexes/labels: the **values** all live in **[`tokens.css`](tokens.css)** (the single source of truth), and the [Design system](#design-system) section below holds the *rules* for using them and points to that file.

## Purpose & vision

Running several Claude Code sessions in tmux means constant tab-switching, copy-pasting prompts between terminals, and no real visibility into what each agent is doing or how they relate. The dashboard replaces that with **one window** where you can see every agent, its state and context, the messages flowing between agents, and compose/send prompts — **without ever touching the raw CLI.**

The thing that makes this more than "several terminals in a grid" is **context-sharing between agents** — links, a shared scratchpad, and agent-to-agent conversation. That is the defining feature; everything else exists to support it.

Guiding principles that shape the whole UI:

- **Bias toward on-screen, not floating.** Prefer giving a surface a pane or step-into view over floating it. Anchored, transient bits — menus, pickers, hover popovers, quick confirms — are fine; the thing to avoid is movable or stateful floating windows. When something wants that, make it a pane instead.
- **Compose-first.** Typing into a raw CLI is the thing this tool is meant to replace, so the prompt-composing surface is always present and is the primary action.
- **One identity per agent, everywhere.** Each agent gets a color + icon used consistently across the graph, feeds, log, history, and chips, so you can track an agent at a glance — see [Agent identity & naming](#agent-identity--naming).
- **Route control through the GUI, including approvals.** Anything you'd normally approve in the CLI — permission prompts especially — surfaces in the dashboard so you never drop back to a terminal to unblock an agent. (This is why approvals get a first-class home; see [the Inbox tab](#the-inbox-tab).)
- **Bridge is the backbone; the dashboard is the skin.** Agent lifecycle/send/read already exist below the UI; the dashboard is a control layer on top.

> **Out of scope (deliberately):** per-agent cost/spend tracking, a diff viewer, a terminal-UI (TUI) version, and keyboard-only (F-key) control — all considered and explicitly dropped. *(A raw live-CLI/terminal feed was previously dropped here too, on the grounds that the Team Feed's Messages tab covers what each agent is doing. That was reversed: a raw feed is now provided **on-demand** by the [Agent panel's Console tab](#agent-left-pane) — an expandable surface scoped to one agent, **not** an always-on pane. The Messages tab remains the curated, cross-agent **rendered** view; the Console is the raw, single-agent one.)*

## What it physically is

A desktop application (Electron + React) presenting one window. Each agent is a real Claude Code session running in tmux on WSL2, driven by a local Python (FastAPI) sidecar through the tmux bridge that already handles create/send/read/status; tmux also serves as the crash-recovery safety net. The tmux bridge is the primary path the dashboard is built around, and the design target throughout. The in-process Claude Agent SDK is not a whole-system fallback; it is reserved for specific non-interactive tasks that need no real terminal (such as the Revise / Summarize utility passes). Agents themselves always run on the bridge. That's the whole stack the UI sits on; this README stays focused on UX.

## Layout — the three-column model

A title bar on top, a status footer on the bottom, and three vertical columns between them. The **Agent** column (left) is the narrow one, full-height; the **middle** and **right** columns are the wide ones, each split top/bottom into two stacked panels — **Team Graph** over **Library** in the middle, **Team Feed** over **Prompt** on the right. Columns and their stacked sections are separated by draggable splitters. The whole frame is **full-bleed** — the header, the three-pane body, and the footer meet flush and run edge-to-edge, with no outer margin.

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│ Title bar — app name · version · agent count │ clock · WSL2 · tmux · status     │
├────────────────────┬────────────────────────────┬───────────────────────────────┤
│ LEFT (narrow)      │ MIDDLE (wide)              │ RIGHT (wide)                  │
│                    │                            │                               │
│ Agent              │ Team Graph                 │ Team Feed                     │
│ ├ Details          │   (agent cards)            │ ├ Messages                    │
│ ├ Create           │   + Link Config drawer     │ ├ Scratch                     │
│ └ Console          │ ────────────────────────── │ ├ Log                         │
│                    │ Library                    │ └ Inbox                       │
│                    │ ├ Plans ├ Documents        │   + shared agent filter       │
│                    │ └ Assets                   │     (persists · subagent tree)│
│                    │   + doc switcher,          │ ───────────────────────────── │
│                    │     line-numbered editor   │ Prompt                        │

```

> *(Layout note — the schematic above is deliberately simplified; each panel's full behavior lives in its own section under [Panels](#panels). A few cross-panel specifics worth flagging here: the always-on agent-selector columns (Prompt **From/To**, Feed **Filter**) are **header dropdowns** — the multi-selects fill with **two-line identity badges** (full-height tile + role over number·name, mirroring the list rows) and collapse to a `+N` chip, so Compose and the feed streams span the full panel width. The **Plans** tab renders the native Claude Code plan files (`~/.claude/plans/*.md`) as **line-numbered markdown** with a per-card review system — see [Library](#library-middle-bottom). Feed/History cards are **expandable** and use **[select-to-act](#team-feed-right-top)** throughout — a header-click selects the whole card, a **separate flush chevron** expands (*select ≠ expand*; **no checkbox** on any tab) — over a shared footer: **select/deselect-all · a merged Export control · Summarize** (the single Export dropdown folds Copy/Export-to-file together with Embed/Attach; Summarize opens a slide-over summary — available on **Messages, Scratch, and Log**, each with its own; the Messages tab also adds a rightmost **Stop**). Major panels are separated by a clear **navy divider** (no grip nub; the strip is still drag-resizable). The [current mockup](mockup.html) is the visual authority.)*

The columns are tied together by **selection**: clicking an agent card in the Team Graph loads that agent into the **Agent** panel, so the graph highlight and the Agent panel always describe the same agent in focus.

The **title bar** also carries a **Settings** gear at its right end (beside the status chips) that opens the [Settings](#settings-step-into-view) step-into view — a full-window surface that replaces the body and returns to the 3-pane on exit.

## Panels

Each panel below is the single source of truth for its own behavior. Cross-cutting ideas (identity, linking, the scratchpad, lifecycle) are described once in [How it works](#how-it-works--concepts-that-span-panels).

### Team Graph (middle, top)

The roster of live agents, one **card** per agent. Each card is a **fixed square** with a single-column layout, top → bottom: a thin **top meta strip** (the agent's **age stamp** — created datetime + auto-scaling "ago" duration, small + left-aligned — paired with the **status badge** in-flow at the right end); an **identity row** (icon + color, role, number + name on the left; the **model** right-justified and bottom-aligned to the name, with an **opus-only FAST lightning bolt** trailing it — bright when on, muted when off, hidden for non-opus); a full-bleed divider; three **settings chips** (mode · effort · think — icon heading + value, spread full width); **two labeled health-colored bars** — a **Ctx** bar (context usage %) and, beneath it, a **Turns** bar (live count against the agent's Max turns, e.g. *Turns 34/50*), each with a leading label and an inline value; an **activity band pinned to the bottom** (a textless **Run strip** + a two-track **Marquee** of the agent's current activity, each separated by full-bleed dividers); and a **subagent strip** of small **rectangular** clickable id-badges (each **filled with its run-state status colour with a bold white id** — no dot — shown collapsed to **one row** with a chevron; clicking expands an **in-flow accordion drawer** that grows the card downward to reveal the rest). The currently-selected card is filled (light teal) and marked as the one being viewed.

- **Selection drives the app.** Clicking a card focuses that agent in the Agent panel.
- **Status at a glance.** A single **status badge** (in-flow at the right end of the top meta strip) shows one of **four** run states — **active / idle / pending / error** — where *pending* (the warm attention tone) means "this agent is waiting on you" and *error* (the danger tone) means "this agent's run failed." It's deliberately **binary, not a count** — an agent can only be blocked on one thing at a time (see [Inbox](#the-inbox-tab)), so the *number* of agents awaiting you lives on the Team Feed's Inbox tab, not on each card. The badge is also a **shortcut button**: it jumps to **Inbox** when pending **or error** (both surface in the Inbox — a failed run as that agent's **Error** card), **Prompts → History** when active, or **Prompts → Compose** when idle.
- **Run + activity, pinned to the bottom.** The **Run strip** is a textless, single-flat-colour progress bar of *opposite polarity* to the load bars (no danger ramp for the live states) keyed off status — active green · pending warm · idle muted/empty · **error a solid danger bar** (a halted/failed run, static — no barber-pole, since it isn't working) — using a **barber-pole indeterminate** animation for "working, % unknown" (the honest fallback until the real Run-% source is decided). Beneath it, a neobrutalism two-track **Marquee** streams the agent's current activity as one scrolling line, pausing on hover and going static + muted when idle.
- **Health coloring.** Both the Turns bar and the Ctx bar use the same good→bad ramp the Timeline uses for context % (green → amber → red), so load reads at a glance without decoding the agent's identity color.
- **Subagents** appear as small **rectangular clickable** id-badges on the parent card — each **filled with its run-state status colour** (active / pending / idle / error, reusing the `--success` / `--warning` / `--muted` / `--danger` status tokens) with a **bold white id** (`A1`, `A2`, `B1`, … — **group + member**: the parent run lettered A·B·C… in occurrence order, then the spawn order within that run; an identifier, *not* a count, and **no `s` prefix** — full sender form `coder-01 › A2`), matching the larger status-badge treatment. The strip is an **in-flow accordion drawer** (like the Inbox cards): collapsed it shows only the **first row** of badges (vertically centred in the trigger) with a **chevron** action icon on the right; clicking the strip toggles it **open**, growing the card downward to reveal the remaining badges wrapped onto new rows (chevron right → down). The state **sticks** (a toggle, not a hover popover), and the chevron/drawer appear **only when the badges wrap past one row**. The section is **reserved at a fixed size even when empty** (0-subagent cards show a muted "no subagents" placeholder), so collapsed every card stays the same square height — a card only grows past its square while its drawer is open (its grid row grows; square row-mates stay square). **A badge click is isolated from the card's own select/drawer** (per-badge `stopPropagation`), then runs the **subagent-badge action** (OD-13, which **resolved OQ-1**): it **(1)** focuses the **parent** agent in the Agent panel, **(2)** opens that agent's Details **[Subagents](#agent-left-pane)** accordion scrolled + highlighted to that subagent's row, and **(3)** scopes the Team Feed filter to that subagent on the **Messages** tab — clicking a helper selects its parent and scopes everything to the helper. (Its 4th **error** state — `sb-error` — mirrors the status badge.)
- **Scales past what fits.** The graph shows many agents (not just a few); cards lay out in a grid and the view scrolls.
- **Linking starts here.** Select the agents you want to connect and use **Link Agents** to open the Link Config drawer — see [Linking & context-sharing](#linking--context-sharing).
- **Links as edges *(planned)*.** Active links are intended to render as **directed arrows between cards**; the current mockup doesn't draw them yet (linking lives in the drawer + footer count for now).

### Team Feed (right, top)

The top panel of the right column (the Feed sits above [Prompt](#prompts-right-bottom)): a real-time, cross-agent view of communication, attention, and events, as four tabs over a **shared agent From/To filter** — a header **dropdown** (in a sub-bar, labelled **From/To**) whose popover is a **2-level tree** (OD-13): agent identity rows on top (including a **User** row), and any agent that has spawned subagents carries an **expand chevron + sub-count** revealing its **indented subagent rows** (run-state badged, e.g. `A2 · Explore`). **Parent-select = the whole subtree**, **leaf-select = just that subagent**; a single **contextual All/None** toggle still spans the lot. Selected agents fill the trigger as identity **badges** (a selected subagent reads compactly as `parent › A2`; collapsing to a **+N** chip past a cap). The filter is **one shared field across all four tabs and applies universally** — a subagent selection is simply **inert on tabs with no subagent traffic (Scratch, Inbox)**, matching nothing there, rather than being toggled per-tab. **Filter-only, never addressable:** subagents appear here (sender-side) but **not** in the [Prompt](#prompts-right-bottom) compose-**To** — you can't send *to* a helper (To stays parents + User/Scratch).

Tab order puts the everyday streams first and the **Inbox** (approvals) last (Messages · Scratch · Log · Inbox).

| Tab | What it shows |
|-----|----------------|
| **Messages** | The team's traffic, attributed and color-coded by agent, merged into one stream with two in-tab filter groups — a **Type** segment for **direction** (Sent / Received) and a **Content** segment for how much tool detail to include (Thoughts / Read / Write / Bash / Diffs / Meta). Direction is anchored to **you, the operator**: **Sent** = prompts you pushed to the team; **Received** = everything coming back from agents. (So **User** only ever reads as *Sent* — it never carries an incoming tag.) **Subagent events stream nested/indented under their parent** (OD-13) — visible by default, the Type/Content filters applying as normal, and always **Received** (helpers never receive operator sends; the parent badge leads, then the run-state sub-id badge); a *collapse subagent chatter* toggle is an allowed fast-follow. The direction tag on each message is intentionally neutral — no special color. Each card's **header** leads with the **agent badge** (at the full `.badge` size, the reviewer-chip standard), then — for **received** messages — a **status badge** (**Active** in-flight / **Complete** / **Failed** red), then the **Sent/Recv** direction tag (order agent → status → dir); user-**Sent** messages keep just the direction tag. The expanded body is a **contiguous Library-style rail** that **fills the card's content surface** (rows butt together with hairline dividers, the rail cell flush against the content — no inner bordered panel inset within the card; the Messages card's body padding is dropped so the rail reaches all four edges, with each row's own cell padding giving the text its breathing room): a **title row** holds the **turn number** ("Turn N", doc-title style) in its content with an **empty rail box**, then the **default message text** ("msg"), then **each included tool block**, each rail box carrying a tight 3-char type tag (`msg · tht · rd · bsh · dif · wrt · mta`). Cards use the shared **select-to-act** model (the same language as the [Library](#library-middle-bottom) doc rail), **teal throughout** (no pink): each block rail **toggles** its block and **multi-selects within a card** — a contiguous run reads as **one unbroken light-teal band** (rail + content, no seam); the **title rail (or the card header) select-alls the whole message**; the rail tracks the active **Content** filter so only currently-visible blocks are selectable. **Hover** lights the row (or, from the title rail, the whole card) canvas cream. The **chevron** expands separately, so *select ≠ expand*. (Whole-message → Attach; a block run → Embed, wired in [Linking & context-sharing](#linking--context-sharing).) The shared **Messages footer** carries **select/deselect-all** · the **[merged Export control](#sharing--exporting--the-merged-export-control)** (one dropdown — **Export**: Copy selected · Export selected → file; **Add to prompt**: Embed · Attach — each item selection-gated, export-to-file lands the selection in [Library → Documents](#library-middle-bottom) as a new doc) · **Summarize** · plus a rightmost icon-only **Stop** (stops the agent's run); it wraps to a second row on a narrow column. **Scratch / Log / Inbox cards share the same selection model** — a **header-click** selects the whole card (light teal), a **separate chevron** toggles the dropdown, and the card attaches via the [Embed/Attach control](#linking--context-sharing); they have **no checkbox** and **no internal sub-selection** (Inbox is one request per card, so whole-card select is for attach parity, not multi-item grouping). |
| **Scratch** | A live view of the [shared scratchpad](#the-shared-scratchpad): each post stamped with agent ID + time. |
| **Log** | A timestamped system-event stream (started, committed, flagged, link fired, permission/approval requested…), color-coded by agent. |
| **Inbox** | The team-wide **requests inbox** (last tab) — typed sections Error · Warning · Permission · Plan · Decision; see [The Inbox tab](#the-inbox-tab) just below. |

#### The Inbox tab

The team-wide **requests inbox** — the human-control surface that keeps agents unblocked without dropping to a terminal. Because a Claude Code agent is single-threaded, **it can only ever be blocked on one thing at a time**, so this tab shows **one card per agent that's waiting on you** (narrowed by the same agent chips as the rest of the feed), and the tab's badge is the **fleet total** — how many agents need you right now. Cards are **grouped into typed sections** — **Error · Warning · Permission · Plan · Decision** (ordered most-to-least urgent) — so the **section header carries the type** and the cards themselves stay clean (**no per-card type badge**). Each card carries the **agent's identity** (icon + role + name); an **Error** *or* **Warning** card also carries a short **subtype** label in the section's colour (Error: the failure class, e.g. *Connection*, on `--danger`; Warning: the limit hit, e.g. *Max turns*, on `--warning`):

| Section | For | Controls |
|---------|-----|----------|
| **Error** | Any **abnormal block to execution** — API/model, tool/MCP, environment/connection, config, or a stall (distinct from a Permission prompt, a Decision, or a normal Lifecycle auto-stop); carries a short **subtype** | inline error text · **Retry** · **Dismiss** · Reply *(no View, no Forward)* |
| **Warning** | An **attention-needed notice that isn't a hard block** — e.g. nearing the context limit (compaction imminent), an approaching rate/usage cap, or a **hit auto-stop limit** (e.g. *Max turns*) that paused the run and needs your call to resume; carries a short **subtype** | **Acknowledge** (clears it) · Reply |
| **Permission** | A tool/command the agent wants to run (the same prompts you'd normally approve in the CLI) | Approve · Deny · Reply |
| **Plan** | A plan/handoff to review before it proceeds | **Review** (→ Library → Plans) · Reply |
| **Decision** | The **`AskUserQuestion`** surface — **one question per card**, with candidate options to choose between | pick an option, then an explicit **Approve** (disabled until you've selected) · Reply |

**Plan cards drop Approve/Reject** — all plan approval and the agent-review verdicts live in the [Library Plans tab](#library-middle-bottom); the Inbox Plan card only **Reviews** (jumps there, expanding + highlighting the matching plan) or **Replies**. Colour: **Deny / Dismiss** use the **danger** colour; **Approve** stays pink-primary. **Permission is binary Approve / Deny (+ Reply)** — "Always allow" was **removed entirely** (OD-14): no always-allow button and no always-allow rule-persistence, now or later. **Section headings carry an escalating severity ramp** — the heading **label text + its matching count badge** run **Error** (`--danger`, red) → **Warning** (`--warning`, amber) → **Permission** (`--inbox-permission`, blackish-orange), while **Plan + Decision** stay the neutral `--muted` tone. Only Permission needs a dedicated token (`--inbox-permission` `#a9710f`); Warning **reuses the existing `--warning`** and Error **reuses `--danger`**. **Error stays the single alarm**: it's the only section whose cards also carry a **danger left-edge** (Warning and the rest are heading-coloured only). This supersedes the old per-card reddish→copper request-type ramp (it ranks severity by section heading now, not a per-card badge).

- **Cards are expandable + selectable.** Like the Log/Messages cards, each Inbox card **collapses to its identity row + title** and **expands via its chevron** to reveal the request detail + its controls; a **header-click** selects the whole card (light teal) into the shared Copy/Export/Attach strip — there is **no checkbox** (the same select-to-act model as Messages/Scratch/Log). A **Pending** status badge on a [Team Graph](#team-graph-middle-top) card jumps here and **scrolls to, expands, selects, and highlights** that agent's card (e.g. a failed run surfaces as that agent's **Error** card).
- **Review routes to the plan** *(Plan cards only)*. The **Review** button **hands you to the Library panel's Plans tab**, expanding and briefly highlighting the matching plan so you can read/edit the full proposal there — rather than embedding a plan editor inside the card. This is the deliberate cross-link between the two surfaces: the *plan* is a document (it lives with the docs); its *approval* lives in the Plans tab, and only its **review hand-off** surfaces here.
- **Reply is generalized and routes to the Editor.** Every card's **Reply** (far right, set off from the other controls) **pre-fills the [Prompts](#prompts-right-bottom) Editor with a frozen reference block** quoting the card's contents — built from the shared [inserted-block primitive](#the-templates-flow) (an `embed` block, *not* a bespoke widget): a **Decision** embeds the question + its options; **Plan / Permission / Error** the request or error detail. It also **pre-targets that agent**, so a free-form reply is composed and sent from the one place all prompts originate. Change the **To** to forward — which is why there's no separate Forward. *(Error's **Retry** likewise loads the last command into the Editor, routing through it rather than the Console.)*

### Agent (left pane)

Everything about **one** agent — whichever is selected in the graph. Three tabs share the pane.

- **Details** — opens with a **header** (agent identity — tile · role · name — with the **status badge** and the agent's **created-time**, a datetime + auto-scaling "ago" duration, stacked right-aligned beside it), then the agent's configuration and live readouts in three deliberately-separated bands:
  1. **On-demand config** (Role, No., Name, Description, Model, Skills, Tools, **MCP servers**, **Plugins**, **Deny rules**, Color, Icon). These are *greyed by default*; click a field's **pencil** to edit, then **save** — so configuration can't be changed by accident mid-run. **Role** is a dropdown (with free-text entry) populated from the user- and project-level `agent.md` files; picking one in **Create** prepopulates the related fields from its front matter. **Skills** and **Tools** are multi-select dropdowns (skills from the skills files; tools = the native Claude Code tool set) that keep floating chips. **MCP servers**, **Plugins**, and **Deny rules** (OD-18) are the same multi-select primitive, exposing this agent's **per-agent scoping** that the engine accepts at create but didn't surface before: which **MCP servers / plugins** it may use (none = all available; these take effect at the agent's **next launch/restart**, so editing them on a *live* agent means "applies on next start") and the **deny rules** that hard-block it (per-agent tool scoping leans **deny-based** — deny is the reliable hard-block; `--allowedTools` is ignored under bypass, a claude bug). This is deliberately distinct from [Settings](#settings-step-into-view)' *global* registry ("is server/plugin X available at all" vs. "may agent Y use server X"). *(Model is intentionally editable, not locked: an agent's model can be changed mid-run.)*
  2. **Always-editable while running** — **Mode**, **Effort**, and **Lifecycle** auto-stop limits, the knobs you reach for during a run, so they stay live. The **Mode** block carries the permission-mode segmented control (**Plan · Ask · Edit · Auto · Bypass**, where Bypass is the bypass-permissions danger option) and, beneath it, two **labeled toggle-buttons** on one evenly-spaced row (icon + label) — **Opus Fast-Mode** (maps to `/fast`) and **Thinking Mode** (extended thinking). Each is a button whose label reads its own state (*Opus Fast-Mode Off/On* · *Thinking Mode Off/On*) and whose whole face fills teal when on. **Opus Fast-Mode is gated** (dimmed + inert) **to the Opus model** — enabled only when this panel's **Model** is **Opus** (`/fast` is an Opus mode); **Thinking Mode** has no gate. See [Lifecycle & autonomy](#lifecycle--autonomy).
  3. **Live readout** — the **health-colored** context-usage bar (with a Compact action) that **expands to a per-category breakdown** whose head carries a native **turn-scope `<select>`** (*Total* or a specific *Turn n* — descending to Turn 1, n = the agent's turn count): *Total* is today's whole-window snapshot (categories as % of the window, the cutoff line, both loaded-context sub-sections); picking *Turn n* rescopes the header / bar / category rows to that turn's contribution (no Free space, no cutoff; rows sum to ~100% of the turn) while the *Memory files* / *Custom agents* sub-sections stay put (they describe loaded context, not a per-turn quantity). The two denominators differ on purpose — the header % is share of the **window**, the row %s are share of the **turn**. A **Turns** readout (the count inline beside the label like Context) that **uses the same accordion primitive**, expanding to a **by-tool** breakdown (Read/search · Edit · Bash · MCP · Subagent · Web, plus a **Coordinating** slice for cross-agent work and a *Remaining* segment closing the turn budget) — the multi-agent parallel to Context's by-category view, and the categorical summary to the Timeline's per-turn log. Then the **Timeline**: a point-by-point list of the messages sent to the model that fills the rest of the panel, with two actions over the same list — **Rewind** (roll this agent back to the chosen point) and **Handoff** (branch from that point into a *new* agent). See [Rewind & Handoff](#rewind--handoff). Then, pinned **below the Timeline and above the Retire/Delete footer**, a collapsed **Subagents** accordion (OD-13) — a **read-only audit history** of this agent's subagents, its header carrying a **total-subagent count badge**; it expands to rows **grouped by run** (A·B·C…), each: id + type (`A2 · Explore`) · task · status (running / done / error) · usage (tokens · tools · duration) · model · a link to its own transcript. (The [Team-Graph](#team-graph-middle-top) card's subagent strip stays the glance view; this is the audit home, and the subagent-badge click lands here.)
  - **Footer:** **Retire** + **Delete** — Retire (soft red-text-on-cream) ends the session; **Delete** (a solid-red-fill / white-text variant, to its right) **permanently** wipes the agent — its configuration and transcripts — and removes it from the roster/graph and any links. Because it's irreversible, Delete is gated behind an inline **"this can't be undone" confirm** (the same confirm pattern as Retire, in the danger tone).
- **Create** — the new-agent wizard: Role, No., Name (with a randomize affordance), Description, Skills, Tools, **MCP servers**, **Plugins**, **Deny rules** (the same per-agent scoping as Details, OD-18), Color, Icon, Model, Mode (with the same Opus Fast-Mode / Thinking Mode toggles), Effort, Lifecycle. **Handoff** lands here with the source agent's settings prepopulated (and editable).
  - **Footer:** **Create · Reset · Cancel**.
- **Console** — the focused agent's **raw Claude Code terminal feed** paired with a **slash-command runner**, both scoped to the single selected agent (so a command's target is unambiguous — *no* multi-select, which is exactly why this lives here and not in the multi-target Prompt or the multi-filter Feed). The feed **faithfully mimics a real Claude Code terminal** — a dark monospace surface with the native markers (`>` input · `●` tool calls with `⎿` results · `✻ Thinking…` · a model/status line · `+`/`−` diffs · a native permission-prompt box) — so it reads as the agent's actual CLI output, the raw counterpart to the Team Feed's rendered [Messages](#team-feed-right-top). Because a native terminal is *not* the dashboard's neobrutalism surface, the feed uses a **self-contained `--term-*` palette** — a **documented exception** to the otherwise-strict "every value comes from `tokens.css`" rule. The exception is scoped to the feed surface only: the surrounding **agent bar, command catalog, run bar, and Expand control stay neobrutalism**. The command surface is the **single home for every slash command** — *not* split into agent-specific vs. shared, because everything simply runs on the focused agent: a **scrollable catalog grouped into conceptual clusters** (Session & context · Model & behavior · Info & status · Tools & integrations · Project & custom · System), each entry a command + a one-line description, with a filter box. Commands that already have a dedicated home (e.g. `/model` in Details, `/mcp` in [Settings](#settings-step-into-view)) are still listed here, tagged *also-available-there* — the catalog is deliberately complete. Picking an entry stages it into the run bar; a typed-or-staged command runs against the focused agent and its output lands in the feed.
  - **Width via Expand.** Because raw output and a grouped catalog both want more room than the narrow Agent column gives, the Console carries an **Expand** control that pops it into a **step-into view that covers only the left + middle columns** — its right edge stops at the left edge of the right column, so the **Team Feed + Prompt stay visible** while you work against the live terminal (the edge is recomputed on window resize and splitter drags). This is a **deliberate difference from [Settings](#settings-step-into-view)**, whose step-into covers the *whole* body: the Console is a working surface you reference against the live feed, where Settings is a configuration modal you step fully into. Both share the step-into mechanics (replace their region, return on **Close** / Esc) and the [bias toward on-screen surfaces over floating ones](#purpose--vision). The pieces **reflow** between states: in-column the feed leads and the catalog rides as a **slide-up panel** over it (full panel width); expanded, the catalog becomes a persistent **left rail** beside a wide feed, with the run bar across the bottom — the rail-beside-body shape [Library](#library-middle-bottom)'s doc-switcher already uses.
  - **Footer:** the **run bar** itself (the command input + Run) — the Console's bottom strip, in place of a button row.
  - *(The Console reflects whichever agent is focused, like the rest of this pane: selecting a graph card repaints its identity bar, feed header, and run-bar target — the same selection-drives-the-panel wiring as Details.)*

### Prompts (right, bottom)

The compose-first heart of the app — the bottom panel of the **right** column, beneath the Team Feed. Three tabs sit over a **shared sub-header**, so the way you address a prompt is consistent no matter which tab you're on.

- **Shared From + To** — a header **sub-bar** of two dropdowns (the selection persists across all three tabs), so Compose and the streams span the full width:
  - **From / Source** (single-select) — who the prompt is *from*: **User**, or an agent (sending *as* that agent for coordination). A compact **dropdown** whose trigger shows only the active selection as a collapsed identity badge; no All/None (single-select needs neither).
  - **To / Target** (multi-select) — who it goes *to*: a **dropdown** whose popover is the full agent **identity-row list** (recolorable tile + two-line role·name, no truncation), led by a **Scratch** row that posts to the [shared scratchpad](#the-shared-scratchpad), with a single **contextual All/None** toggle (shows *All* until everything is selected, then *None*). Picked targets fill the trigger as identity **badges** and collapse to a **+N** chip past a cap; multi-target is simply picking several — there's no separate "broadcast" mode.
- **Tabs** (just **Compose · History** now — Templates folded into Compose):
  - **Compose** — the prompt-composing surface, top to bottom: a horizontal **attachment-chip strip** (under an **"Attachments"** section heading), then the **Editor**, then the **Templates** section. The **attachment strip** (above the Editor) holds the prompt's attachments as small chips styled like the [Library](#library-middle-bottom) nav cards — each with a **remove ×** and **click-to-open in the Library**; it's hidden when empty. The **Editor** (its heading reads **"Editor"**, renamed from "Compose") is a `contenteditable` rich field under that label; its header carries a **mic** ghost button (per-field voice dictation, right after the "Editor" label — red while recording) plus **copy / clear** ghost icons; it holds **free prose plus stacked, delimited inserted blocks** (the shared [inserted-block primitive](#the-templates-flow)) — **frozen `embed` blocks** (quoted content with a "from \<source\>" header + click-to-source) and **interactive `template` blocks**. Below the Editor, the **Templates** section holds a single-select **dropdown** (default **None** + the saved templates) and a **fill input** (greyed until a template is active): picking a template **inserts a selectable `template` block** at the cursor (not raw inline text), and the **active** template block is the one whose placeholder **pills** the fill input drives. **None** deselects the active template + re-greys the input; the box stays fully editable. See [the Templates flow](#the-templates-flow).
  - **History** — past prompts for the focused agent, attributed and **right-aligned timestamped** (the time pins to the card's right edge in both the collapsed *and* expanded states, matching Team Feed cards). Cards take the **same select-to-act model as the Team Feed cards** — a **header-click** selects the whole card (light teal), a **separate flush chevron** expands (*select ≠ expand*; **no checkbox**). **Edit** is a **ghost icon in the card header** (after the attachment chip, before the timestamp); the **footer** mirrors the feed strip — **select/deselect-all · [merged Export control](#sharing--exporting--the-merged-export-control) (Copy · Export → file · Embed · Attach, each selection-gated) · Retry · Stop** (icon-only; Stop is danger-red), with export-to-file landing the selection in [Library → Documents](#library-middle-bottom). A prompt that carried **attachments** shows a small **paperclip + count chip** in the card header (only when there are any); clicking it opens a popover listing each attachment, and clicking an entry **opens it in the Library** — images in **Assets**, docs in **Documents**. *(History-only for now; the same chip is built to extend to Team Feed → Messages later.)*
- **Action row** (the shared Prompts footer) — leads with a **paperclip attach** button that picks an image from the [Library → Assets](#library-middle-bottom) list — both directions register there, so Assets stays the source of truth — then the Revise and Response-format controls, and the Send split button at the right. *(Voice dictation is **not** in this row: each Editor — Compose, Plans, Documents — carries its **own** mic ghost button in its Editor header, bound to that field.)* Sending uses a pair of **split buttons** (a button joined to a dropdown chip), styled as a **pink primary** (Send) next to a **teal secondary** (Revise):
  - **Send** (primary) with a **timing** chip — **Now · Inject · Next · Queue** (no "Hold"; that only makes sense for a link). The chip carries the delivery timing, reusing the same vocabulary as link [Triggers](#linking--context-sharing). **Now** interrupts and delivers immediately; **Inject** delivers to a *running* agent without stopping it (a one-shot message applied at the next safe boundary); **Next** waits for the next turn boundary; **Queue** appends to the agent's prompt queue.
  - **Revise** with a **scope** chip — **Grammar · Language · Refactor** (default Grammar): runs an AI cleanup/rewrite pass on the prompt before you send it, scoped from a light spelling/grammar fix up to a full restructure.
  - **Scope:** Compose offers Revise + Send (plus an **attach** paperclip; the voice **mic** lives in the **Editor header** as a per-field ghost button, not here). History swaps the row for its own footer (**select/deselect-all · the [merged Export control](#sharing--exporting--the-merged-export-control) · Retry · Stop**, icon-only; Edit rides in the card header) and hides the action row.

### Library (middle, bottom)

The home for **organizing and reviewing all of a project's documents and reference assets** — three tabs: **Plans · Documents · Assets**. The **Plans** tab carries a **count badge** (same rounded-square style as the Team Feed's Inbox badge) signalling **new plans** — those not yet reviewed (in *review* or *draft* state); it hides when the count is zero. It's the other half of the middle column, beneath the Team Graph. (The panel was named "Documentation" while it held only docs; "Library" reflects that it now also holds images.)

- **The shared doc editor.** The **Plan** and **Documents** tabs render their files as **line-numbered markdown** whose line numbers match the raw file, so "see line N" is a real reference. An **interactive left rail** indexes every line and section: click a rail cell to **select a line, a whole section, or the entire document** (so a chunk can be shared or commented on). The rail cells are **color-coded by what a click selects** — pink = the **title** (selects the whole doc), dark teal = a **section**, light teal = a **line**; the selected text highlights in **light pink**, and re-clicking the active cell clears it. Inline code is tinted so it reads apart from body text. This **select-to-act** model is **shared with the Team Feed's [Messages](#team-feed-right-top) cards** (whole = pink, block = teal), so a selection means the same thing — *this is what you'll Embed or Attach* — in both surfaces.
- **The Plans tab is the review surface.** It lists the native Claude Code plan files (`~/.claude/plans/*.md`, shown under a single directory line + a "*N plans · N awaiting review*" count) as **expandable cards**. Each card's header is **three rows**: ① owner **agent badge** · title · state (e.g. *In review*); ② the **feedback tally badges** (Approve/Revise/Block counts) · *done/total steps*; ③ filename · **Created / Edited** date-time with a relative "ago". Expanding a card reveals the line-numbered plan beside a **nav rail**, then the shared footer.
- **The nav rail** has two modes: **Outline** (the section list, each with a worst-verdict dot + comment count) and **Feedback** (one **card per response** — the reviewer's agent badge + a thumbs **agree** toggle, a color-coded **verdict badge** (Approve/Revise/Block) + a **section badge**, and a clamped comment summary). Selecting feedback moves **three indicators together**: the **card** fills (light teal), its **section text** highlights (teal, linking card ↔ text), and the **comment popout** opens. Closing the popout or deselecting clears all three, and opening a different comment switches all three — driven from either a Feedback card or the in-text rail-gutter badge. The rail **resizes with the text column** (it grows when the popout opens, so cards aren't clipped).
- **Comments dock under the text.** Section feedback also surfaces as a **verdict badge in the rail gutter** at that section's line. The rail badge is a **section anchor**: its count is **all** the comments on that section (every verdict), and clicking it (or any Feedback card for that section) opens **one comment popout** listing **all of them** — each row carrying its **own verdict badge** + reviewer badge · time · a thumbs agree toggle, so the count always matches the contents. (Selecting a section moves the three indicators together: the matching Feedback card(s), the highlighted section text, and the popout.) You add your own with the **Comment** button — enabled once you select a line/section — which opens a **composer** in the same popout: your badge, a **Mark as** verdict dropdown (Approve/Revise/Block), a thumbs agree default, an optional note, and **Save**.
- **An "Editor" header over the content; hand-off/decision in the footer.** Plans (and Documents) carry an **"Editor" header** above the content field, mirroring the Compose Editor header: the **Editor** label — with a **mic** ghost button right after it (per-field voice dictation, red while recording) — and right-aligned **Copy · Edit · Comment** ghost icons (the same ghost icon-buttons Compose's copy/clear use). The card's **footer** then holds only the hand-off + decision controls — for a Plan, the **[merged Export control](#sharing--exporting--the-merged-export-control)** **leads**, then the single-agent **Review chip** (a reviewer badge select + an icon-only review button) — both left-aligned — then, right-justified, the decision trio: **Revise** (send the flagged sections back to the authoring agent) · **Reject** · **Approve**. (It wraps to a second row on a narrow column.)
- **Cross-linked with the Inbox.** A plan is a *document* (it lives here); its *approval* surfaces in the Team Feed's [Inbox](#the-inbox-tab). The Inbox's **Review** button jumps here, expanding and briefly highlighting the matching plan; a plan's **Approve/Reject** is the same handoff in reverse.
- **Documents and Assets share a card shape** (a simplified Plans, *without* Plans' owner/title/tally/steps rows): a basic **header** with the file **path** + **Created / Edited** dates, then the body (the line-numbered editor for docs, the image preview for assets), then a **footer**. **Documents** also carry the **"Editor" header** (Editor label + a per-field **mic** + right-aligned **Copy · Edit · Comment** ghost icons) above the editor — same treatment as Plans — so their footer holds the **[merged Export control](#sharing--exporting--the-merged-export-control)** (left-aligned, the leading spacer dropped) + **Remove** (danger, icon-only, right). **Assets drop Copy · Edit · Comment entirely** (no Editor header — an image isn't text-editable), leaving just the path/dates header, the preview, and the same **Export control + Remove** footer (an Asset is whole-file, so only **Attach** is enabled on it). Both tabs use the same **nav rail** (a **per-file-type icon** · name · path), and each nav row carries a ghost **rename** (pencil) icon; an **Add** affordance at the foot of the rail offers **Add file** (opens the explorer) or **Paste** (creates a new row **opened with its name in inline-rename** so you type it immediately — this is also where the Team Feed / History / editor **Export → "Export selected → file"** lands the selection, as a new pasted doc).
- **The Documents tab** folds every project doc into **one** tab (it replaces the old per-file Readme / Claude tabs). The nav **path is load-bearing** — it disambiguates same-named files (the project `CLAUDE.md` vs the user-level `~/.claude/CLAUDE.md`). Selecting a row loads that doc into the **same line-numbered editor + rail** the Plans tab uses. Only **README**, the project **CLAUDE.md**, and the user-level **CLAUDE.md** are listed by default; the rail's **Add** affordance adds more.
- **The Assets tab** holds **reference images** as a **rail + preview** (the same shape as Documents): a left list of **per-file-type icon + name**, with the selected asset's **preview** filling the right. Assets is the **single source of truth** for media — an image added here, *or* attached while composing via the [Prompts paperclip](#prompts-right-bottom), both register in this one list.

### Settings (step-into view)

A top-level **Settings** surface for the whole workspace — the configuration that isn't about any one agent. It's opened from a **gear in the title bar** (right cluster, beside the WSL2 / tmux / Connected chips) and presents as a **step-into full-window view**: it replaces the 3-pane body in place, toggles in and out, and returns to the 3-pane on **Close** (or Esc). It is deliberately **not a fourth always-on column** and **not a floating window** — it steps into the same frame rather than overlaying it, in keeping with the dashboard's [bias toward on-screen surfaces over floating ones](#purpose--vision).

**Tabs are by subject, not by elevation.** Five subjects — **Setups · Usage · MCP · Plugins · Config** (Setups leads as the first, default-shown tab) — and where a subject spans config scopes, **scope is a secondary segment inside the tab** (never a tab of its own), so a single subject is never split across tabs:

| Tab | What it owns | Scope segment |
|-----|--------------|---------------|
| **Setups** | The full **Save / Load setup** flow (agents + links). | — |
| **Usage** | **Account** (email / org / plan, read-only from local creds), **Limits** (session / weekly % + resets, fetched live from the API — degrades gracefully when unreachable), and token consumption — the `/stats` · `/status` surface. **Usage only**; per-agent cost/dollar spend stays [out of scope](#purpose--vision). | — |
| **MCP** | The global server registry — enable/disable, connection & OAuth health, and the disabled-server **parking** state. | user / project |
| **Plugins** | Installed plugins + enabled state, and marketplaces. | user / project / local |
| **Config** | Default model, permission mode, sandbox, hooks, env, CLAUDE.md, plans. Each setting tagged **Live** (takes effect now) or **New session** (needs a restart). | global / project |

Cross-cutting rules:

- **Read-only vs editable, separated.** Within every tab, read-only status/health/usage is kept in its own band, visually distinct from editable config — so glancing at state never reads as a control.
- **Global edits are gated.** Changing global (`~/.claude`) config — which affects every project — is held behind an explicit **confirm**, with a standing warning on the Global scope.
- **It owns the global registry, not per-agent scope.** Enablement here is "is server/plugin X available at all," a different thing from "may agent Y use server X," which lives in the [Agent panel](#agent-left-pane) — **now surfaced there as the per-agent MCP servers / Plugins / Deny-rules scoping controls (OD-18)**. The two are deliberately not conflated.
- **The footer keeps a glanceable shortcut.** A token-usage summary lives in the [status footer](#layout--the-three-column-model) for quick access (it jumps straight into Usage); the panel holds the full detail. (The footer's former Save/Load shortcut was removed — the Setups tab is now the single home for that flow.) *(Voice dictation is no longer a single universal footer mic: each Editor — Compose, Plans, Documents — now carries its **own** mic ghost button bound to that field, enabled only while its field is editable + visible, red while recording.)*

## How it works — concepts that span panels

### Agent identity & naming

Every agent has a stable identity made of a **role**, a per-role **number**, a short **human name**, a **color**, and an **icon** — e.g. *researcher · 01 · sandy* → `researcher-01-sandy`. Human names (sandy, kai, drew, rowan…) are used because they're easy to say and remember when you're talking about agents. Color and icon are chosen per agent (not fixed by role) and then appear *everywhere* that agent does, which is what makes the whole UI scannable.

### Linking & context-sharing

The defining capability: a persistent **link** that forwards context from one agent to another so they can collaborate (including back-and-forth "conversation" between two agents). You create a link from the [Team Graph](#team-graph-middle-top) and configure it in the **Link Config drawer** — a sheet that opens immediately to the **right of the Team Graph panel** (anchored to the left edge of the right pane, sliding out from behind the graph) so the whole Team Graph stays visible while you set the link up:

- **Direction** — a 3-state toggle: A→B, B→A, or A↔B (both). *(On the graph itself, links are intended to read as directed arrows — see [Team Graph](#team-graph-middle-top).)*
- **Trigger** — *when/how* the message is delivered:
  | Trigger | Behavior |
  |---------|----------|
  | **Now** | Interrupt the target and deliver immediately. |
  | **Inject** | Deliver to a *running* target without stopping it — a one-shot message applied at the next safe boundary (between tool calls). |
  | **Next** | Wait for the target's current response to finish, then deliver — ahead of its queue. |
  | **Queue** | Wait for the current response *and* let the existing queue drain first (the polite default). |
  | **Hold** | Stage the message for your manual approval before it's sent. |
- **Payload** — *what* is sent:
  | Payload | Meaning |
  |---------|---------|
  | **Message** | The source's output, forwarded as a single rendered message. |
  | **Transcript** | The source agent's full conversation/context, export-style (*intent: drawn from the agent's transcript; exact source TBD*). |
  | **Manual** | No automatic content — you compose it by hand each time the link fires. |
- **End After** — optional safety limits so bidirectional links can't run away: **Turns / Tokens** as independent toggles, each with its own value. None on = no limit; if both are on, the link ends at the first one reached.

**How messages read (sender + trigger).** A message an agent receives carries lightweight metadata — who sent it and which trigger delivered it — embedded in the message for the *receiving agent's* benefit. The dashboard hides those tags and renders the human-facing version: a color-coded sender heading, a small trigger badge, and the body text. One message, two presentations. User-sent prompts go through as plain text.

#### Sharing & exporting — the merged Export control

Export and content-sharing live in **one** dropdown — the **Export control** — reused on the **Library** footers (Plans / Documents / Assets), the **Team Feed** footer, and the **History** footer. It routes content to the [Prompts Editor](#prompts-right-bottom) (Embed/Attach) and to the clipboard / [Library → Documents](#library-middle-bottom) (Copy/Export), replacing the older *separate* Output Export and Embed/Attach dropdowns. The trigger is a single download-icon dropdown-with-chevron whose menu has **two headed sections**:

- **Copy & Export** — **Copy selected** (to the clipboard) · **Export selected → file** (a new Documents doc from the selection; you stay put).
- **Add to prompt** — **Embed in prompt** · **Attach as file**.

*(Cut was dropped in the consolidation. The word **Link** stays reserved for inter-agent [links](#linking--context-sharing); content-sharing is **Embed** or **Attach**.)*

- **Per-item selection-gating** via the shared [select-to-act model](#library-middle-bottom): the trigger enables when **any** item does, and each item gates on the selection — **Embed in prompt** is enabled for **any** selection (whole card/doc or a sub-block/line/section); **Attach as file** is **whole-only** (a whole card/doc/title selection or an Asset); **Copy** / **Export → file** act on **whole feed/History cards** or on the **editor line/section/doc** selection. Disabled items stay listed but greyed, so the menu only ever fires the applicable action. **Assets are Attach-only** (an image isn't copyable-as-text, embeddable as a quote, or a `.md`).
- **Embed** drops a **frozen `embed` block** into the Editor — the selected text (a doc section, a feed block run, *or* a whole card/doc) under a "from \<source\>" header with click-to-source (the [inserted-block primitive](#the-templates-flow)).
- **Attach** **saves the selection as a real file**, drops a **path chip** into the Editor's [attachment strip](#prompts-right-bottom) (a path reference + a hardcoded "read this"), then **switches to Compose and reveals** (flashes) the chip. Content that has its own file (an Asset, an existing Document, a Plan) is chipped at its own path; **materialized** content (a feed message, a History prompt) is first **written to a new Library → Documents doc** (the same `createDoc` the Add-document Paste uses) — distinct from **Export → file**, which creates that doc and leaves you *in* Documents.
- **Citations** — from an attachment chip's link icon, with the cursor in the prose, you insert an inline **`citation` pill** pointing at that attachment (same payload as Attach). Deleting a citation does **not** delete its attachment; deleting an attachment **cascades** to remove its citations; you can only cite something already attached.

**The Review chip.** On the Plans footer, **Review** is a **single-agent reviewer select** — the trigger shows the chosen reviewer's full agent badge, paired with an **icon-only review button** (no text). Picking *who* reviews is the only choice; the icon sends. Its reviewer **picker opens *upward*** as a raised overlay above the footer (matching the Export menu), so the editor card's bounds never clip it — the shared rule for the Plans footer's popovers (reviewer picker, merged Export, verdict). *(Groundwork for the single-reviewer-agent model; the deeper review/verdict workflow stays **deferred**.)*

### The shared scratchpad

A single shared markdown document agents post to as a **living** workspace — each post attributed to an agent and timestamped, viewable live in the Team Feed's [Scratch tab](#team-feed-right-top) and writable from [Prompts → Target → Scratch](#prompts-right-bottom). The attribution/timestamps exist because it's a running log of who-said-what; a clean, stateless document can be produced from it at the end. *(Posting in, reading out is the whole interaction for now; richer per-post selecting/commenting is **deferred**.)*

### Rewind & Handoff

Both act on the **Timeline** in the Agent → Details tab — the list of points (messages sent to the model) in the focused agent's run:

- **Rewind** — roll *this* agent back to a chosen point and resume from there.
- **Handoff** — branch from a chosen point into a *new* agent: it opens the Create tab with the source agent's settings prepopulated (editable), so you can carry the work onward without disturbing the original. *(Richer handoff artifacts/summaries are **deferred**.)*

### Lifecycle & autonomy

The project leans toward letting an agent "go as far as it can" on a task rather than babysitting it — so agents carry **auto-stop limits** (per-agent **Max turns** and **Context %**) that end a run safely when hit. These are deliberately a **different scope** from a link's [End After](#linking--context-sharing) limits: **Lifecycle bounds a single agent's run; End After bounds an inter-agent exchange.** Keep the two distinct.

### The Templates flow

The Prompts **Editor** holds **inserted blocks** alongside free prose — one reusable **block primitive** with three variants, stacked vertically as delimited blocks (the boundary uses the **card-outline** colour, not the tan divider):

- **`embed`** — *frozen* quoted content (a doc section, a feed message or block): a muted, small-font body under a static **"from \<source\>"** header, with **click-to-source**, a **remove ×**, and **clamp + expand** when long. Not editable.
- **`template`** — the same shell but *interactive*: **selectable**, and the **active** one (the selected block) is what the Templates fill input drives.
- **`citation`** — an inline pill in the prose pointing at an [attachment](#prompts-right-bottom) chip (see [Linking & context-sharing](#linking--context-sharing)).

Templates turn reusable prompt files into fill-in-the-blank forms *(Templates was a separate tab, then inlined into Compose, and now lives **below** the Editor)*:

- A single-select **dropdown** (default **None** + the saved templates) sits in the **Templates** section below the Editor, with the **fill input** below it (greyed until a template block is active).
- Picking a template **inserts a `template` block** into the Editor at the cursor (or appends if the box isn't focused), *without wiping* existing text; the template's placeholders come in as **clickable pills** (filled pills show their value; unfilled show the bare tag, e.g. `{severity_threshold}`).
- **Selecting** a template block (clicking it, or a pill inside it) makes it the **active** one; click a pill → the **fill input** activates (a single-line field that auto-grows) with **Reset** and **Apply** as trailing icon buttons *outside* the field; Apply pushes the value into the pill, which **stays editable/re-selectable** (Reset clears it back to the bare tag). *(The input isn't tinted to the pill's color — the placeholder simply names the active tag.)*
- Filled vs. unfilled pills are visually distinct, and the active pill gets a selected style, so it's clear what's left to fill before you Send. **None** deselects the active template block + re-greys the input; the Editor stays **fully editable** throughout.

## Design system

> **Where the values live.** Every raw value — colour, **type (families + weights)**, **spacing**, **sizing**, **radius**, **border-width**, shadow — lives once in **[`tokens.css`](tokens.css)**, the single source of truth, exposed as CSS custom properties (`--main`, `--border`, `--border-width` / `--divider-width` / `--ring-width`, `--radius-base` / `--radius-sm` / `--radius-xs`, the `--size-*` and `--space-*` scales, `--font-sans` / `--font-mono`, the 25 `--ag-*` agent colours, …). Both [`mockup.html`](mockup.html) and the shared [`styles.css`](styles.css) reference the tokens via `var()` — never a hardcoded literal (no inline `2px`/`3px` borders, badge radii, font-family strings, or recurring padding/gap/size values remain).
>
> **Token shape (the locked rule).** Tokens are **primitive** — used directly, by value, with *no* semantic layer — for **colour, radius, border-width, and sizing**. The one place **semantic alias** tokens are allowed is **padding / gaps / spacing**; the mockup currently uses the primitive `--space-*` scale directly. What stays **out** of `tokens.css`: per-instance bindings (the per-card `--nc` that binds one `--ag-*` colour to a single agent) and data/runtime values (a progress bar's inline `width:68%`, resizable pane widths) — those reflect state, not design, and stay inline. *(The retired `--req-*` Inbox request-type ramp was **removed** in the token pass — unreferenced after the Inbox went to typed sections.)*
>
> This section owns the **rules** — which token to reach for, and why — and names tokens rather than repeating their hexes. To change a value, edit `tokens.css`; to change a *rule*, edit here.

The dashboard speaks the **[neobrutalism.dev](https://www.neobrutalism.dev)** visual language — thick **2px navy borders** (`--border`), **hard offset shadows** with no blur (`--shadow` / `--shadow-sm`) on raised/interactive elements, **flat fills**, a uniform **tight radius** (`--radius-base`), and the **Archivo** type family (heading 800 / body 500) with **JetBrains Mono** for metrics. The palette keeps the **"Happy Hues 17"** core (cream / navy / pink) and puts that palette's teal to work as a secondary. Token names mirror neobrutalism's Tailwind theme (`bg-main`, `bg-secondary-background`, `border-border`, `shadow-shadow`, `rounded-base`, `font-heading` / `font-base`) so the static mockup ports ~1:1 to React/shadcn.

**Surfaces** — deliberately just **three warm surfaces**, plus a button tint and a hairline:

| Role | Token | Used for |
|------|-------|----------|
| **canvas** | `--background` | the app background; the Team Graph scroll-well |
| **card** | `--secondary-background` | cards, inputs, panel bodies, **and the Team Feed tab wells** (Messages/Scratch/Log/Inbox, white like Prompts → History) — white pops against the cream canvas |
| **chrome** | `--surface-3` | panel headers/footers, toolbars, segmented tracks |
| **button** | `--surface-btn` | low-emphasis **action buttons** — a warm cream, lighter than chrome so buttons still pop on `--surface-3` footers (form inputs/selector fields stay white) |
| hairline | `--rule` | the tan outline on **inline-enclosing atoms only** — inline code chips and count/verdict-badge pills. **Structural card dividers** (section rules, row separators, rail/gutter separators, popover/section headers) recolour to the navy `--border` instead, so a card's internal structure reads in the same ink as its outline. **Every navy structural divider stays navy in every state** — the hover (cream) and selection/feedback fills (`--select` / `--rail-section`) show **behind** the line, never repainting it. (This retired the older "seam-blend," which used to recolour a rail/gutter separator to the band fill within a contiguous selection so the highlight read as one unbroken band; structure-visible won out — a selected Messages/editor band, a run of stacked selected rows, and a Team Feed / History card's header→body divider all keep their navy lines) |

Border + heading ink is navy (`--border` / `--foreground`, 2px everywhere). Muted text ramps `--muted` → `--muted-2`.

**Accents — one emphasis ladder** (grounded in Material 3 color roles + Carbon button hierarchy): **pink = primary · teal = secondary · cream = low-emphasis · red = danger.** Read any button by asking *what does clicking it do to this content?* — **pink** commits it *here* (one per surface), **teal** sends it *elsewhere* (hand-off / outbound to another agent or surface), **cream** is a quiet local utility that acts in place, **red** destroys, and **white is not a button** (reserved for fillable inputs + selector triggers). The one nuance is that teal carries two jobs of the same emphasis tier: a **hand-off action** *and* anything **selected/active** (segmented values, toggles, the selection fill). They rarely sit side by side — state lives in the config panel, hand-off actions in footers — so teal carrying both is correct, not muddy.

| Accent | Token | Meaning (tier) |
|--------|-------|---------|
| **pink** | `--main` | **PRIMARY** — the one primary action per surface (Send · Apply · Create · Approve · Save) · active **panel tab** · the token pill · title bar |
| **teal** | `--secondary` | **SECONDARY** — **hand-off / outbound** *actions* (Revise · Reply · Review · the **Embed/Attach** action · Load — they send the content to another agent or surface) · active value in segmented controls / model·trigger tabs / toggles · selection **rings** · the **tab count badges** (the Library → Plans + Team Feed → Inbox counts) and the "N selected" count badge |
| **light teal** | `--select` | **selection FILL** (the "secondary container") — selected list rows *and* cards (graph nodes, History, Messages, plan **Feedback** cards), *and* the user's **line / section / whole-doc selection** in the Plans/Documents editor render (rail cell + content as one band; replaced the old light-pink doc-render highlight) |
| **dark teal** | `--rail-section` | the Plans/Documents editor's **feedback/badge highlight** — clicking a Feedback card fills its section dark teal with white ink (`--icon-fg`), the one place this colour is used, distinct from the user's light-teal selection |
| **cream** | `--surface-btn` | **LOW-EMPHASIS** action buttons — quiet local utilities that act in place (Copy · Edit · Comment · Retry · Summarize · Reset · mic · attach), **including the icon-only buttons** (these sit on cream too, so the neutral tier is *one* surface — white is reserved for fillable inputs/selectors, not buttons) |
| **success** | `--success` | success / active |
| **warning** | `--warning` | attention / pending |
| **danger** | `--danger` | destructive (Retire, **Delete**, Reject, Deny, Dismiss, Stop, Bypass) **and the Error/alarm state** — the Inbox **Error** section/cards, the Team Feed Messages / History **Error** status badges, and the graph-card **Error** run-state badge. Most danger controls are **red-text-on-cream**; the irreversible **Delete agent** button is the one **solid-red-fill / white-text** variant (heavier, to read distinctly from the softer Retire beside it) |

Form **inputs** and **selector-field triggers** (Color, Icon, Role, Source, Skills, Tools) stay white (`--secondary-background`) — fillable surfaces read distinct from clickable low-emphasis buttons; the accent rides the *value/selected option*, not the field chrome.

**Selectors — inline vs. menu.** *Inline* selectors (all options visible — segmented controls, tabs, toggles) show the chosen option as a **teal fill** (`--secondary`) in place. *Menu* selectors (options hidden behind a trigger — Color/Source/Skills/Tools and **Response**) keep a **neutral** trigger; teal rides the selected option(s) inside the open menu plus a small **teal count badge** for multi-selects.

**Split buttons** read **light chip + full action**: the dropdown chip is a *lighter tint* of the action's accent (it recedes; the action leads). Send = full-pink action (`--main`) + light-pink chip (`--main-dim`); Revise = full-teal action (`--secondary`) + light-teal chip (`--secondary-dim`). The chip is a *parameter of that action*, so it stays in the action's hue family rather than going teal.

**Inbox — typed sections with an escalating heading ramp.** The Inbox groups its cards into **typed sections** (**Error · Warning · Permission · Plan · Decision**, ordered most-to-least urgent); the section *header* carries the type, so cards drop the per-card type badge. The section **heading label + its matching count badge** carry an **escalating severity ramp** — **Error** (`--danger`, red) → **Warning** (`--warning`, amber) → **Permission** (`--inbox-permission`, blackish-orange) — while **Plan + Decision** stay the neutral `--muted` tone. Only **Permission** carries a dedicated token (`--inbox-permission` `#a9710f`); **Warning reuses `--warning`** and **Error reuses `--danger`**, so no new signal colours were minted. **Error stays the single alarm**: it's the only section whose cards also carry a **danger left-edge** (the rest are heading-coloured only). This replaces the old per-card reddish→copper request-type ramp — the `--req-*` tokens were **removed** from `tokens.css` (retired and unreferenced), and severity is ranked by section heading now, not a per-card badge. `--warning` doubles as the **pending** signal elsewhere (it tints the graph-card **status badge**), which is consistent — both are "attention, not an error". The Team Feed's Inbox tab carries a **teal count badge** (`--secondary`, the fleet total) matching the Library → Plans tab count — the teal **tab count badge** family, distinct from the per-section heading badge.

**Status:** shown per graph card as a rectangular **text badge** in-flow at the right end of the card's top meta strip (a button that jumps to Inbox / History / Compose) — active `--success` · idle a solid `--muted` (slate-gray) fill with **white text** · pending → the warning tone above · **error a solid `--danger` fill with white text** (the 4th run state — a failed run; jumps to that agent's Inbox **Error** card) — **all four** use the same solid-fill + white-text treatment so they read as one badge family (the lighter `--ag-user` / `--muted-2` tone is reserved for the User row and the footer idle dot). The History/Plan **`db-*` status badges** mirror this: `db-error` is likewise a **solid `--danger` fill + white text** (not a soft tint), matching the Team Feed Messages **Error** status badge. The **context bar** on graph cards and in the Agent panel is colored by **health** (`--success` → `--warning` → `--danger`), not by the agent's identity color — matching the Timeline's context-% coloring.

**Agent identity palette** — **25** colors assigned per agent (tokens `--ag-crimson` … `--ag-rose` in `tokens.css`) and kept in a different *register* from the UI accents above (deeper and more saturated than the light pink/teal/status tints) so identity never collides with meaning. It's the **"Jewel" family**: one **OKLCH** recipe — a fixed deep lightness/chroma (`oklch 0.52 0.15`, gamut-clamped per hue) spread around the wheel in spectral **ROYGBIV** order, so every agent reads as equally weighted and none as a UI signal. The first **16** are spaced ~evenly; the **+9** (added for OD-03) **densify the widest gaps** to a **25**-colour ring rather than re-spacing the originals (additive — no existing token renamed). Past ~16 colours the **icon becomes the primary disambiguator** and colour a soft "family" signal, so assignment is round-robin (`color = n mod 25`, `icon = n mod 50` — every (colour, icon) pair unique for the first 50 agents, reused beyond). Defined and listed (and shown in the mockup's picker) in spectral order: crimson · scarlet · vermilion · amber · gold · topaz · citron · chartreuse · lime · peridot · fern · emerald · teal · turquoise · cyan · cerulean · azure · cobalt · sapphire · indigo · violet · orchid · fuchsia · magenta · rose. The mockup's identity JS reads each agent's colour from these tokens, so the picker, the agent tiles, and the agent roster all draw from one place. Each agent also picks an **icon** from a **curated 50-icon set** (drawn from the **game-icons.net** pool of 167 in `assets/icons/agents/`, embedded in the mockup as a recolorable sprite); these render as **recolorable tiles** — the tile background takes the agent's color and the glyph is a white knockout (`--icon-fg`). UI (non-agent) icons are **Lucide** (`assets/icons/ui/`), drawn at ~2.25px stroke to suit the heavy borders.

**Core components** (behavior the mockup styles): **Resizable** panel groups (drag-resizable, but major panels read as a clear **3px navy divider** rather than a grip nub; while a splitter is dragged a small **cursor-following readout** shows the two adjacent panels' live sizes in px — left/top panel first, then right/bottom — and hides on release), heading strips, tabbed panels with a persistent shared sub-header, segmented controls (single-select), **split buttons** (a pink primary or teal secondary joined to a dropdown chip — Send+timing, Revise+scope; light chip + full action), agent **identity rows** (recolorable tile + two-line role·name; selected = light-teal fill + check) used for the Source / Target / Filter selectors with a **contextual All/None** toggle, **color & icon pickers** (current selection always visible; sized so both dropdowns are the same height), the **context accordion** (a usage bar that expands to a per-category breakdown; its head carries a native **turn-scope `<select>`** — *Total* or a specific *Turn n* — that rescopes the header / bar / category rows to one turn's contribution while the loaded-context sub-sections stay put, the header % then reading share-of-window vs. the rows' share-of-turn), **labeled status bars** (a leading label · health-colored bar · inline value, used for Turns + Context on graph cards; in the **Agent panel** both Turns and Context reuse the same context-accordion primitive — the Turns bar expands to a **by-tool** breakdown, the multi-agent parallel of Context's by-category one), the **doc editor** (line-numbered markdown with a plain tan line/section selection rail whose rows are split by **navy (`--border`) horizontal dividers confined to the rail — never across the text** — so each click-target is easy to eyeball; the Editor header sits over the editor box while the Outline/Feedback nav rail rises full-height; the editor and nav are **height-capped as one row** and each scrolls internally, so the empty **rail track continues to the bottom of the visible editor** (no line numbers fabricated for the empty space — real numbers stop at the last file line; the tan track continues, ruled with **continued navy (`--border`) hairline row-dividers** — a lighter 1px echo of the real per-row dividers — so the empty rail reads as a ruled gutter, closed by the navy gutter edge) even when the nav is taller than the text; a click selects the line / section / whole-doc as a light-teal fill across rail + content, with the navy rail/gutter separators **staying navy** (the fill shows behind them) — + a docked comment popout), selectable cards (graph nodes · History · Messages · plan **Feedback** cards — light-teal fill), cards with a colored left accent stripe (Inbox), the Timeline point-list, palette-matched scrollbars, and a **universal jump-to-end pill** — a small floating cream pill auto-attached to **every scroll region** (feeds, editors, the Console feed, catalogs, Settings, lists — including each **expanded feed/history card body**, which caps its height and scrolls internally; the contenteditable Compose field gets its pill via an outer overlay so the field text isn't disturbed): it appears in the region's corner only when scrolled away from an edge, snaps to the very top/bottom on click, and hides at the edge (Slack/Discord/terminal behavior), with native scrollbars still handling incremental movement. The **Console** adds a **terminal feed** that **faithfully mimics a real Claude Code CLI** — a dark monospace surface with native markers (`>` input · `●`/`⎿` tool calls + results · `✻` thinking · a model/status line · `+`/`−` diffs · a permission box) via a self-contained **`--term-*` palette**, a documented exception to the token rule, while the surrounding chrome stays neobrutalism — + a **command catalog** (conceptual groups under sticky hairline-ruled headers; **two-line rows** — bold-mono command over a muted one-line description — so the command column scans cleanly; a faint flat *also-in-X* tag; light-teal hover) + a run bar, and an **Expand** step-into that covers the **left + middle columns only** (so the right column stays visible — a deliberate difference from Settings' full-window step-into; catalog as a slide-up panel in-column, as a left rail when expanded). The **Settings step-into view** adds its own small kit: subject tabs over a secondary scope segment, **on/off switches** (teal-when-on) for MCP/plugin enablement, read-only vs editable section headers, lifecycle **Live / New-session** tags, health-colored usage bars, and an explicit global-edit confirm.

**Hover cards (documentation-through-design).** Major components carry an **info popover** — an anchored, transient hover/focus card (an allowed anchored-transient surface, **not** a stateful floating window, per the [bias toward on-screen surfaces](#purpose--vision)) that documents what the component does and why. It's **pure documentation**: no clickable links inside, so routing stays on the real controls. The trigger model is **hybrid** — a small **info-glyph** that **trails immediately after** panel-header labels / tabs / the Link drawer label (a **no-chrome muted glyph** that brightens on hover — no circle or border, vertically centered to the title; the header's right-hand controls stay pushed to the end), and **whole-element hover** on small atoms (the status badge, the Ctx/Turns bars, the Trigger/Payload segments) where a glyph would clutter. One shared primitive drives them all from a `data-hc` attribute. This is the "documentation through the design process" surface — behavior that can't be *clicked* in the static mockup is captured *on the component it describes*.

**Conventions:** one uniform tight radius (`--radius-base` / `rounded-base`; small badges/chips use a slightly tighter radius but are still **rounded squares** — no pills); hard offset shadows (`--shadow` / `--shadow-sm`) are reserved for raised / interactive elements, while inert rows and **non-interactive badges are flat** (border-only). Identity badges used as labels (card headers, feed/history/nav cards, the comment popout) are flat; the raised treatment is kept only for the interactive From/To/Filter dropdown trigger. Single-line inputs carry an inline clear "**X**". The repeated **Edit / Comment / Retry** actions — plus **Stop** (icon-only **danger**) — are **icon-only** buttons (+ a hover tooltip) wherever they appear, in one of two styles: the **cream mic-button style** (`.icon-btn`) in action strips (the Team Feed + History strips — where the **[merged Export control](#sharing--exporting--the-merged-export-control)** now carries Copy / Export-to-file *and* Embed / Attach in one selection-gated dropdown, **History's Edit** rides in the card header, and the Messages footer's **Stop** sits rightmost), and the **ghost icon style** (`.ghost-ic`, matching the Compose Editor's mic/copy/clear) in the **Editor headers** of Compose, Plans, and Documents, where a per-field **mic** + **Copy · Edit · Comment** live above the content (Assets drops them — an image isn't text-editable). Labeled buttons (Summarize, Revise, Reject, Approve) keep their text; the **merged Export control** is a **download-icon dropdown with a chevron** (its actions are menu items under **Copy & Export** / **Add to prompt** headings — no on-control labels), and the **Review chip** is a reviewer badge + an icon-only send. (The token values themselves live in [`tokens.css`](tokens.css); the mockup no longer carries an in-page Palette Reference legend.) **Badge families** are kept to a small, named set so the rounded-square vocabulary reads as a system rather than a zoo: **count** chips (Inbox/Plans counts, the teal "N selected"), **status** badges (graph-card active/idle/pending; History `db-*` lifecycle states), **identity** badges (the recolorable tile + role·name used as labels everywhere — now at **one consistent full size** across the Team Feed (Messages · Scratch/Log/Inbox), Prompt → History, and Plans review/feedback rows, matching the reviewer-chip standard; sized so a full dropdown/selector badge sits in the **same height footprint as a standard footer action button**, with the colored agent **tile filling the badge edge-to-edge** so there are no blank strips above/below the icon; the deliberately-smaller **subagent / recipient mini-badges** are the one exception). *(The old per-card Inbox **request-type** badges are retired — the Inbox now groups by [typed section](#the-inbox-tab) instead.)* The **full named badge catalog** (all ten families and every variant) — and every other component — is shown once in [`gallery.html`](gallery.html); the canonical names live in [Component system](#component-system--names-status-and-the-catalog) below. A one-pass *visual* consolidation across the existing badge styles remains a future cleanup.

## Component system — names, status, and the catalog

The component vocabulary is governed in four pieces: a **visual catalog** (the gallery), a **canonical name** on every instance (`data-comp`), a **status marker** for dormant or unresolved UI (`data-status`), and a small **Open Questions register** for design calls deliberately left open. With the [token rules](#design-system) above, these are the rules the **six** design-source files follow: **`tokens.css`** (values) · **`styles.css`** (shared component CSS) · **`behavior.js`** (shared component **behavior** — the interaction logic, loaded by *both* the mockup and the gallery) · **`mockup.html`** (the app surface) · **`gallery.html`** (the interactive catalog) · this doc (the rules).

### The gallery is the visual catalog — an interactive catalog

[`gallery.html`](gallery.html) is an **INTERACTIVE CATALOG**: every reusable component + basic building block is shown as the **real, live component**, grouped by canonical name, with a one-line purpose blurb and the component's `data-comp` slug. It shares [`tokens.css`](tokens.css), [`styles.css`](styles.css) **and** [`behavior.js`](behavior.js) with the mockup (no divergent component CSS or behavior of its own), so each operable specimen *is* the genuine component driven by the genuine interaction code — it can't drift from the app, because there is no second copy. It exists because the running app can't show everything at once — most **badge** variants, all five **inbox card** types, and every component are one tab or data-state away from being invisible, so the gallery is the only place to see them all side by side. **`DESIGN.md` no longer carries a per-component inventory** — the gallery owns the catalog; this doc owns the rules.

**How states are shown (the model that replaced the old static one):** a component with several **looks** (button emphasis tiers, badge colours, the lifecycle ramp) shows **every variant side by side**, each labelled; a control with **menus/toggles** shows **one live instance you actually drive** (click it, open its menu — `behavior.js` makes it work, and an open menu is transient, dismissing on outside-click just like the app); a **data-state you can't reach by interacting** (disabled · empty · error · "just-attached") is shown as an **additional labelled specimen** — that labelled-instance convention is the single, uniform way non-interactive states appear. A handful of **sub-popovers** with no standalone trigger (`format-popover`, `verdict-popover`, …) are shown **open and contained** in their own card. The gallery is **bootstrapped from the live mockup DOM** (each `data-comp` instance harvested as its real markup); to add a component, tag it in the mockup and add a `gx-card` here with its real markup — behaviour comes free from `behavior.js`.

**Group order** (the gallery's five sections, in order): **Primitives** (cross-cutting controls & elements) · **Form fields** (text entry, selection, the per-field edit lock — the white fillable register, isolated from the cream/teal control register) · **Badge catalog** (every badge family + atom, every variant) · **Inbox card types** (the five typed request cards) · **Composites** (multi-part cards, editors, rows, panels).

**State vocabulary** (the conceptual reference for the states a component can be in): **default** → **hover** → **focus / focus-within** → **active / pressed** → **on / selected / checked** → **open / expanded** → **disabled / gated / off** → **status variants** (the semantic ramp it encodes — run-state `active · idle · pending · error`, verdict `approve · revise · block`, or health `success · warning · danger`) → **empty / placeholder** → **recording / live / flash** (transient feedback). In the interactive gallery these split two ways: the **interaction** states (hover · focus · active · open/expanded · selected) are reached by **interacting** with the live specimen — they are *not* given standing specimens — while **variants** (status ramps, the emphasis tiers, colour sets) and the **non-interaction** states (disabled · empty · transient) get their own labelled specimens. Any standing specimen's label is its precise class combo / state name.

**Interaction is real, not faked (the `behavior.js` share).** The gallery does **not** simulate states with helper classes — it loads the same [`behavior.js`](behavior.js) the mockup does, so hovering, focusing, opening a menu, toggling, and switching tabs all run the genuine interaction code on the genuine markup. `behavior.js`'s `boot()` no-ops when there is no `.app` root, so it stays inert on the gallery (which runs its own tiny init for icon rendering); the shared dismiss / Escape / hover-card handlers install on both pages by design. The **only** gallery-only code is the `gx-*` **page-layout scaffolding** (the spec-card grid, headers, name/slug/blurb typography) in `gallery.html`'s own `<style>` — **no component CSS or behavior is copied into the gallery**, and `styles.css` / `behavior.js` are untouched by it. **One necessary asset exception:** the **agent-icon sprite** is embedded in `gallery.html` as a **synced copy of the mockup's** `<symbol>` defs — SVG `<use href="#ag-…">` can't resolve across documents, so the gallery needs its own sprite for the icon-picker / identity-badge / agent-tile / source-dropdown specimens to render real glyphs (rather than empty colour tiles). It is kept in lockstep with the mockup's sprite whenever the curated icon set changes (OD-03). Disabled / empty / error specimens carry the real state class or HTML attribute (e.g. a real `disabled`), so they render exactly as in the app with nothing faked.

### Canonical component names (`data-comp`)

Every component instance carries a single **`data-comp`** attribute on its **root element** — the canonical name as a **kebab-case slug** (`status-badge`, `agent-node-card`, `export-control`). Sub-parts are *not* tagged; they're identified by their existing classes (a node card carries `data-comp`; its `node-bars` / `node-chip` sub-parts do not). The slug is the one canonical label — single-sourced on the element; where a doc or the gallery needs a readable name it spells the same slug in words rather than tracking a second string. Many components are JavaScript-generated (the `.db-*` lifecycle family, the verdict badge, the count chips, the identity badges, all five inbox card types), so the slug is **emitted from inside the builder functions** — every rendered instance carries it, not just the static markup.

**The registry** (canonical name → slug) — the set the gallery enumerates in full, grouped by its five sections. Sub-parts of a composite are *not* registry entries; the composite-internal primitives below (`node-subagents`, `markdown-row`, `message-rail-row`, `msg-block`, etc.) ARE registered because each carries its own root class + an independent state set worth cataloguing on its own card.

- **Primitives:** `button` · `ghost-icon-button` · `icon-button` · `text-button` · `split-button` · `comment-split` · `export-control` · `response-format-control` · `segmented-control` · `scope-switcher` · `tab-bar` · `nav-tab` · `timeline-mode-switcher` · `model-selector` · `version-popout` · `source-dropdown` · `color-picker` · `icon-picker` · `multi-select` · `toggle-button` · `mini-toggle` · `switch` · `agree-toggle` · `stepper` · `progress-bar` · `run-strip` · `marquee` · `accordion` · `chip` · `node-chip` · `token-pill` · `attachment-chip` · `jump-to-end-pill` · `timeline` · `inline-confirm` · `toast` · `summarize-slideover` · `hover-card`.
- **Form fields:** `text-input` · `search-input` · `template-select` · `scope-select` · `role-combobox` · `edit-lock`.

### Status markers (`data-status`)

Dormant-but-shipped UI and unresolved design questions are marked on the element itself with a **`data-status`** attribute, so they show up both as a tag on the thing and (via the register below) as a scannable list. Live, wired components carry no marker.

- **`data-status="planned"`** — shipped code/markup that doesn't do its real job yet (a `toast()` or counter stands in). The marker is the attribute alone (optionally a short plain-language code comment); it does **not** cross-reference any external backlog. Current `planned` items: the **link edges** (`drawEdges` / `LINKS` ship but draw nothing — the edge-layer host was removed), the **Link Agents** drawer **Save / Delete** (toast + counter only), the **MCP / plugin enable switches** (scripted demo), and **review / citation / attachment routing** (`sendReview`, `gotoCitation`, `composeAttach`, `openAttachment` — toast or navigation only). *(OD-20 flipped the **Console run bar** planned→built: the Console surface — feed · catalog · run bar — is design-final, so its `data-status="planned"` was removed; `runConsoleCmd` remains a static-mockup demo and the live raw-terminal feed + slash-command route are **backend** wiring tracked in the backend stream, not a design status.)*
- **`data-status="undecided"`** — an element tied to an open design question (see the register). Resolving it means deleting the register line **and** removing the marker; nothing forces a resolution to ship.

### Behavior: wired vs planned

The mockup is heavily wired, not a static prototype — `boot()` renders every list from data arrays and wires the real interaction model: **tab switching**, the **column-tie selection** (clicking a Team Graph card repaints the Agent panel), **select-to-act**, **dropdown/popover** open-close, **accordions**, **empty-state gating**, **cross-panel hand-off**, **live form state**, **data-driven rendering**, **resize/layout**, and transient **`toast`** feedback. The surface that is **not** wired carries `data-status="planned"` (the list above) so dormant code isn't read as live behaviour; `toast()` is the line between "wired" and "acknowledged but mock."

### Open Questions register

Design calls deliberately left open. Each has a stable id, a one-line question, and `status: undecided`; where it attaches to a concrete element, that element (and its gallery specimen) carries `data-status="undecided"`. Resolving an item = delete its line here and remove the marker.

- **OQ-2 — Inbox section: fold vs. catalog.** Whether the Inbox's typed sections (the `inbox-section` wrapper) should be collapsible **folds** or stay a flat **catalog** is unresolved; the wrapper carries `data-status="undecided"`. *(status: undecided)*

*(OQ-1 — the subagent-badge click behaviour — was **resolved by OD-13** and removed from this register: clicking a subagent badge now focuses the parent agent, opens its Details → Subagents accordion to that row, and scopes the Team Feed to that subagent. The subagent badge no longer carries `data-status="undecided"`.)*

**Standing condition (not an open question).** The React / shadcn migration flips when design churn approaches zero — a condition on *timing*, not an undecided design call. Until then the mockup stays static HTML; the token names already mirror neobrutalism's Tailwind theme so the eventual port stays ~1:1.
