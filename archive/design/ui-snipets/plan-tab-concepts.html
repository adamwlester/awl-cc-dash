Documentation panel · Plan tab

concepts

A concept set for a Plan tab living inside a new Documentation panel — the surface that turns Claude Code's native planning into something you read, review, approve and ship from the dashboard, never the raw CLI. It covers both flavours of the real feature: local Plan Mode (read-only research that writes a markdown plan into the plansDirectory) and Ultraplan (planning that runs in a cloud container while your terminal stays free, returning a richer artifact with Simple / Visual / Deep variants). Every mock below is live — click it. Tokens are pulled 1:1 from ui-concept-v9p6.html so any direction drops straight in. Backlog ref: C4 "Plans Tab" + the Documentation-panel loose note.

The short version
The Documentation panel is the new third-from-left column (Agent · Team Graph · Documentation · Feed · Prompts), and Plan is its first, most live tab — sitting beside Specs / Notes / Refs. Recommended build order:
  • SpineConcept 1 — Plan Library. A master–detail list ↔ markdown reader over the real plan files. Everything else hangs off this; build it first.
  • ReviewConcept 2 — Section review. Inline comments + emoji reactions + an outline rail — the dashboard's own copy of Ultraplan's browser review surface.
  • Ship itConcept 3 — Approve → Execute. The gate that routes approval to Requests and fans out to teleport-to-local or cloud → PR, with the live todo checklist.
  • SourceConcept 4 — Ultraplan launcher. Compose a plan, pick a variant, watch the cloud run while the terminal stays free, escalate a local draft up.
  • TrustConcept 5 — Versions & diff. Plans iterate; keep every draft and diff them so an edit is never a black box.

What a plan is

A markdown file Claude writes in plan mode — Goal · Context · phased Approach · Files · Risks · Out-of-scope. The tab renders it; it is not a chat log.

Where it comes from

Local Plan Mode (Shift+Tab, read-only) or Ultraplan in the cloud. Each plan card is tagged with its origin so you know how it was made.

What you do to it

Read → comment / react → edit → approve (or request changes). Approval is a first-class act that surfaces in the Requests inbox.

Where it goes

On accept: teleport to a local agent for execution, or run in the cloud and open a PR. A live todo list tracks the execution it spawns.

1 Plan Library — master & detail

A list of every plan, a reader for the one in focus

The spine of the tab and the closest mirror of how plans actually exist on disk (many markdown files in plansDirectory). The left rail lists every plan as an identity-tagged card — title, status along the draft → review → approved → executing → done ramp, origin (Plan-mode vs Ultraplan), and the owning agent. Click one and it renders, full markdown, in a reader with a sticky action bar. It scales to dozens of plans and reads at a glance, exactly like the Team Feed's other lists.

Buys you
  • Maps 1:1 to the real files — no invented model to maintain
  • Status + origin chips make a fleet of plans scannable
  • Reuses the node-card + list patterns already in the mockup
Costs you
  • Two-pane split needs width — wants the wide Documentation column
  • A very long single plan still needs an inner scroll
  • List + reader is more to wire than a single document view

2 Section review — comments & reactions

Mark up the plan section by section, like the cloud review surface

This is the dashboard's own version of Ultraplan's browser review: the plan splits into collapsible sections, each carrying its own inline comment thread and emoji reactions (a fast 👍 / ⚠️ / 🎯 to signal approval or concern without writing prose). An outline rail navigates long plans and shows which sections have open comments. The loop is leave a note → "ask Claude to address" → review the redraft → repeat, all without leaving the panel.

Buys you
  • Matches the native Ultraplan review UX users already meet on the web
  • Reactions make "I'm fine with this section" a one-click act
  • Outline + comment badges turn a wall of markdown into a worklist
Costs you
  • Section-level threading is real state to model and persist
  • Only worth it once plans are genuinely long / iterated on
  • Risks overlapping with the Feed if comments aren't kept in-panel

3 Approve → Execute

The gate: approve the plan, then choose where it runs

Approval is the moment the plan stops being a document and becomes work. The left card is the gate — Request changes · Edit · Approve — and approving posts the plan to the team-wide Requests inbox (consistent with how every other human-control act routes). The right card is the handoff: pick a target — teleport to a local agent (full environment access) or execute in the cloud → open a PR — with the native clear-context-on-accept toggle. Once it's running, the same card becomes the live todo checklist Claude drives.

Buys you
  • Both real execution paths (local teleport, cloud PR) in one place
  • Approval reuses the Requests inbox — no parallel approval UI
  • The todo list gives execution a progress readout, not a black box
Costs you
  • Cloud execution needs a connected GitHub repo + web account
  • Clear-context-on-accept is destructive — needs a clear warning
  • Two states in one card (gate vs running) adds conditional logic

4 Ultraplan launcher

Compose a plan, send it to the cloud, keep working

Where plans are born. Describe the task, pick a variant — Simple (baseline), Visual (Mermaid / ASCII diagrams), or Deep (subagents run risk & architecture reviews) — and launch. Planning runs in a cloud container (Opus, up to ~30 min) while your terminal stays free, so the right card is a live status readout, not a frozen wait. A local Plan-Mode draft can also be escalated here for a heavier cloud pass. When it lands, the artifact flows into the Library (Concept 1).

Buys you
  • Surfaces the variant choice the CLI hides behind A/B assignment
  • "Terminal stays free" becomes visible — you see it running, not blocking
  • One escalate button bridges local drafts → cloud iteration
Costs you
  • Requires Claude Code web (Pro/Max/Team) + v2.1.91+
  • Plans off a point-in-time repo snapshot — needs a "may be stale" note
  • Cloud status needs real polling/streaming to feel honest

5 Versions & diff

Every redraft kept, every edit diffable

Plans rarely land in one shot — you comment, Claude redrafts, you escalate to the cloud. This concept keeps each version as a pill and shows a line diff between any two, so "what changed when I asked for X" is a glance, not a re-read. It's the trust layer under Concepts 2–4: the natural home for an audit trail of who/what changed a plan and why.

Buys you
  • Makes Claude's redrafts legible — edits stop being a black box
  • Pairs naturally with the comment loop and cloud escalation
  • Cheap if plans are already files — it's git-for-plans, lightly
Costs you
  • Only earns its keep once plans actually iterate
  • A full diff view is more chrome than a small panel wants
  • Needs a snapshot per version kept somewhere

How it ties into the rest of the dashboard

Layout: Documentation becomes the new third column in the reconfigured frame — Agent · Team Graph · Documentation · Feed · Prompts — and Plan is the live tab among quieter doc tabs (Specs / Notes / Refs). Identity: every plan card and cloud run carries the owning agent's color + icon, same as everywhere else. Requests: "approve plan" is an Approval-type request, so it shows up in the Team Feed's Requests inbox rather than inventing a second approval surface; Reply still routes to Prompts → Compose. Triggers / lifecycle: teleport-to-local reuses the same Now / Next / Queue delivery vocabulary as links and sends.
